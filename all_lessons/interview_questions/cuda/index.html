CUDA Kernels — interview prep

A compact, linearized CUDA track for interviews: first learn the vocabulary and call flow, then optimize from the memory hierarchy, then map classic algorithms and data structures onto thread ownership.

Linearized thinking
CUDA questions become tractable when you keep one invariant: every line of code should answer "which thread owns which output, which bytes does it touch, and what must be synchronized before the next step?" The track repeats that invariant until it becomes automatic.

The three-pass structure

PassQuestionOutcome
01 What is a CUDA program mechanically? You can write the declarations, launch a kernel, move bytes, check errors, use streams, and explain the host/device call flow.
02 Why is this kernel slow? You identify the wall first: launch overhead, HBM bandwidth, uncoalesced loads, sync, register pressure, atomics, or math throughput.
03 How do algorithms change on a GPU? You derive reductions, scans, histograms, sparse traversal, queues, and tiled graph/array thinking from ownership and memory traffic.

The lessons

01
Common operations, declarations, call flows
Kernel qualifiers, built-in indices, launch syntax, memory allocation/copy, streams/events, error checking, shared memory, atomics, and the minimal mental model for every CUDA snippet.
02
Highly optimized snippets
Grid-stride loops, vectorized loads, coalescing, shared-memory tiling, warp reductions, block reductions, async copy shape, occupancy knobs, and when each optimization is worth touching.
03
Algorithms, data structures, and the line of thinking
Reduction, prefix scan, histogram, CSR sparse matrix-vector multiply, top-k selection, work queues, and graph frontier expansion as GPU ownership problems.

What an interviewer is testing

  1. Can you linearize? State data layout, choose output ownership, derive index math, name synchronization, then optimize the binding resource.
  2. Can you reason in bytes? A vector add is not "one add"; it is two 4-byte loads plus one 4-byte store. That is a bandwidth test.
  3. Can you keep correctness and speed separate? First make every thread write the right address. Then make warps touch contiguous addresses. Then remove unnecessary traffic.
  4. Can you map CPU algorithms to GPU structure? A serial loop becomes blocks of independent ownership, plus a reduction/scan/atomic/final merge when ownership collides.
