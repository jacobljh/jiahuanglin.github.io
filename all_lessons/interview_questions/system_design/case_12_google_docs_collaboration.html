Design Google Docs-style collaboration

Collaborative editing is a consistency lesson with humans in the loop. The system must converge while preserving user intention closely enough that the editor feels trustworthy.

0. Interview hinge

The hinge is the moment where the prompt stops being generic and the architecture becomes forced. For this case, that hinge is: Operational transform/CRDT choices, presence, conflict handling, document storage.

Invariant to protect
The design is only correct if the user-visible contract survives retries, partial failures, stale caches, and reordering. A concrete invariant to defend here: Session owner dies must be handled by Elect or assign a new owner and replay from the operation log..
Tempting but wrong
A common shortcut is to choose Central session owner vs peer merge just because it buys simple ordering. That shortcut fails when its cost matters: session owner bottleneck. Use it only when the workload matches the condition: online collaborative docs.

1. Clarify the contract

The prompt is best treated as a product contract before it is treated as a component diagram. The system needs to do the following:

  • Multiple users edit one document concurrently.
  • Propagate edits with low latency.
  • Persist document history and support recovery.
  • Show cursor/presence as soft realtime state.
  • Handle offline edits and reconnect.

2. Put numbers on the shape

The exact numbers depend on the interviewer, but these are the estimates that should steer the architecture:

  • Edit operations are tiny but latency-sensitive.
  • Presence updates can outnumber document edits and should not hit durable storage.
  • Long documents require snapshotting; replaying all operations forever is too slow.
  • A large meeting can create many subscribers for one document session.

5. Linearized design

Walk the design in the order a request or event actually moves. This keeps the interview grounded and exposes the first bottleneck naturally.

  1. 1. Load the latest snapshot and recent operation log when a user opens a document.
  2. 2. Route all operations for a document to a collaboration session owner.
  3. 3. Transform or merge incoming operations against operations already accepted.
  4. 4. Broadcast accepted operations with monotonically increasing document versions.
  5. 5. Periodically compact operations into durable snapshots.
  6. 6. Treat cursors and presence as ephemeral pub/sub state.

6. Deep dives

OT vs CRDT

Operational transform centralizes ordering and transforms operations against concurrent edits. CRDTs make offline merge more natural but can carry heavier metadata. Pick based on offline-first requirements.

Versioned operation log

The log is both the replication stream and the recovery record. Snapshots bound recovery time; operations preserve history between snapshots.

Presence is not document state

Cursor position can be stale or missing without corrupting the document. Keeping it out of the durable path protects edit latency.

9. What to say in the interview

Strong signal
Lead with the dominant constraint: Operational transform/CRDT choices, presence, conflict handling, document storage. Then defend the design by naming what you made faster, what you made more reliable, and what complexity you accepted.
