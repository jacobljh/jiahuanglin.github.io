Secure API access, SSO, passwords, and HTTPS

Security design starts by separating identity, permission, transport protection, and secret storage. Blurring them creates brittle systems.

0. Interview hinge

The hinge is the moment where the prompt stops being generic and the architecture becomes forced. For this case, that hinge is: Authn/authz, OAuth/OIDC shape, password hashing, TLS termination, API signing.

Invariant to protect
The design is only correct if the user-visible contract survives retries, partial failures, stale caches, and reordering. A concrete invariant to defend here: Credential stuffing must be handled by Rate limit, monitor, and require MFA for risk..
Tempting but wrong
A common shortcut is to choose Session cookie vs JWT just because it buys server-side revocation. That shortcut fails when its cost matters: session store dependency. Use it only when the workload matches the condition: web apps.

1. Clarify the contract

The prompt is best treated as a product contract before it is treated as a component diagram. The system needs to do the following:

  • Authenticate users or services.
  • Authorize actions on resources.
  • Protect credentials in storage and transit.
  • Support SSO or federated login.
  • Audit sensitive access.

2. Put numbers on the shape

The exact numbers depend on the interviewer, but these are the estimates that should steer the architecture:

  • Password hashing should be intentionally slow, so login endpoints need rate limits.
  • JWT size and validation cost matter at high request QPS.
  • TLS termination centralizes certificate management but creates an internal trust boundary.
  • Audit logs for auth events are write-heavy but compact.

5. Linearized design

Walk the design in the order a request or event actually moves. This keeps the interview grounded and exposes the first bottleneck naturally.

  1. 1. Terminate TLS at a controlled edge and preserve secure identity headers only across trusted hops.
  2. 2. Store passwords with salted adaptive hashes, never reversible encryption.
  3. 3. Use SSO/OIDC to delegate authentication while keeping local authorization decisions.
  4. 4. Validate tokens at gateways or services and check resource permissions close to the data.
  5. 5. Rate limit and monitor login, token refresh, and sensitive mutations.
  6. 6. Write immutable audit events for privileged access.

6. Deep dives

Authentication vs authorization

SSO tells you who the user is. It does not automatically say what that user can do to your resource.

Password storage

A password database will eventually be attacked. Salted adaptive hashing makes offline guessing expensive and isolates users from identical-password leaks.

TLS boundary

HTTPS protects the client-to-edge path. Internal service calls still need an explicit trust model, often mTLS or signed identity headers.

9. What to say in the interview

Strong signal
Lead with the dominant constraint: Authn/authz, OAuth/OIDC shape, password hashing, TLS termination, API signing. Then defend the design by naming what you made faster, what you made more reliable, and what complexity you accepted.
