Design a key-value store

A key-value store is the smallest distributed database that still forces you to choose your consistency, durability, partitioning, and repair story.

Source: Vol. 1 Ch. 6 Case drill Trade-off first
How to read this case
This page is an original educational walkthrough based on the case topic, not a transcription. Read it linearly: contract, estimates, data, flows, bottlenecks, trade-offs, then failure modes.

0. Interview hinge

The hinge is the moment where the prompt stops being generic and the architecture becomes forced. For this case, that hinge is: Partitioning, replication, quorum reads/writes, compaction, failure recovery.

Invariant to protect
The design is only correct if the user-visible contract survives retries, partial failures, stale caches, and reordering. A concrete invariant to defend here: Replica divergence must be handled by Run read repair and scheduled anti-entropy with checksums..
Tempting but wrong
A common shortcut is to choose Quorum vs primary-leader just because it buys availability during single-node failure. That shortcut fails when its cost matters: conflict resolution complexity. Use it only when the workload matches the condition: multi-region or high-write systems.

1. Clarify the contract

The prompt is best treated as a product contract before it is treated as a component diagram. The system needs to do the following:

  • Put, get, and delete by key.
  • Scale beyond one machine for storage and write throughput.
  • Survive node failure without losing acknowledged writes.
  • Offer a clear consistency contract.
  • Repair divergent replicas in the background.

2. Put numbers on the shape

The exact numbers depend on the interviewer, but these are the estimates that should steer the architecture:

  • If value size averages 1 KB, 1B keys is about 1 TB before indexes, replication, and compaction overhead.
  • A write with replication factor 3 creates three storage writes plus coordination metadata.
  • Quorum reads and writes add network fanout to the tail latency path.
  • Compaction can double temporary disk pressure during merges.

5. Linearized design

Walk the design in the order a request or event actually moves. This keeps the interview grounded and exposes the first bottleneck naturally.

  1. 1. Route the key through consistent hashing to a primary replica set.
  2. 2. Append the mutation to a write-ahead log before acknowledging durability.
  3. 3. Serve recent writes from memory and immutable sorted files on disk.
  4. 4. Replicate to multiple nodes and decide whether success means one, quorum, or all replicas.
  5. 5. Use read repair, hinted handoff, and anti-entropy scans to close divergence.
  6. 6. Compact immutable files to bound read amplification and reclaim deleted data.

6. Deep dives

Consistency knob

N, R, and W are not magic variables; they are product promises. R+W>N gives overlapping quorums, but stale reads can still happen with sloppy quorum, failed clocks, or concurrent writes.

Storage engine

An LSM-like engine makes writes sequential and fast, but reads may touch multiple files until compaction catches up. A B-tree would improve point-read predictability but amplify random writes.

Deletes are writes

A delete usually writes a tombstone. If compaction drops it too early, an old replica can resurrect the value. The system needs a retention window tied to repair guarantees.

9. What to say in the interview

Strong signal
Lead with the dominant constraint: Partitioning, replication, quorum reads/writes, compaction, failure recovery. Then defend the design by naming what you made faster, what you made more reliable, and what complexity you accepted.
