Design S3-like object storage

Object storage is simple at the API and ruthless underneath: write bytes once, place them safely, repair them forever, and never lose an acknowledged object.

Source: Vol. 2 Ch. 9, archive Case drill Trade-off first

0. Interview hinge

The hinge is the moment where the prompt stops being generic and the architecture becomes forced. For this case, that hinge is: Object metadata, placement, replication, erasure coding, durability math.

Invariant to protect
The design is only correct if the user-visible contract survives retries, partial failures, stale caches, and reordering. A concrete invariant to defend here: Silent data corruption must be handled by Store and verify checksums end to end..
Tempting but wrong
A common shortcut is to choose Replication vs erasure coding just because it buys simple reads and repair. That shortcut fails when its cost matters: higher storage cost. Use it only when the workload matches the condition: hot or small objects.

1. Clarify the contract

The prompt is best treated as a product contract before it is treated as a component diagram. The system needs to do the following:

  • Put, get, list, and delete objects by bucket and key.
  • Scale to enormous object counts and sizes.
  • Provide high durability and availability.
  • Support multipart upload and metadata.
  • Repair failed disks or nodes automatically.

2. Put numbers on the shape

The exact numbers depend on the interviewer, but these are the estimates that should steer the architecture:

  • Durability target drives replication or erasure coding overhead.
  • Small objects stress metadata; large objects stress bandwidth and placement.
  • Listing by prefix requires ordered metadata or a secondary index.
  • Repair bandwidth can compete with customer traffic after failures.

5. Linearized design

Walk the design in the order a request or event actually moves. This keeps the interview grounded and exposes the first bottleneck naturally.

  1. 1. Authenticate and authorize bucket operations at the front end.
  2. 2. Write object parts to storage nodes chosen by placement policy.
  3. 3. Record object metadata and checksums after enough durable writes succeed.
  4. 4. Serve reads by consulting metadata and streaming from one or more storage nodes.
  5. 5. Run background scrubbers to verify checksums and repair missing fragments.
  6. 6. Use lifecycle and version policies to manage deletion and retention.

6. Deep dives

Metadata scaling

The object data plane can scale by adding disks, but metadata must answer existence, version, and listing queries. This is often the hidden bottleneck.

Replication vs erasure coding

Replication is simple and fast to repair but expensive in storage. Erasure coding saves space for cold data but reads and repairs are more complex.

Read-after-write contract

The API must state freshness. Strong read-after-write simplifies clients but forces metadata coordination.

9. What to say in the interview

Strong signal
