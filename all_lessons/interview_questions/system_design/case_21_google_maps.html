Design Google Maps

Maps is not one system. It is tiles for display, graph search for routes, live signals for traffic, and mobile caching for reliability.

0. Interview hinge

The hinge is the moment where the prompt stops being generic and the architecture becomes forced. For this case, that hinge is: Map tiles, routing, traffic updates, storage, rendering, offline/cache behavior.

Invariant to protect
The design is only correct if the user-visible contract survives retries, partial failures, stale caches, and reordering. A concrete invariant to defend here: Tile hot spot must be handled by Use CDN and origin shielding..
Tempting but wrong
A common shortcut is to choose Raster vs vector tiles just because it buys simple CDN serving. That shortcut fails when its cost matters: less dynamic styling. Use it only when the workload matches the condition: basic maps.

1. Clarify the contract

The prompt is best treated as a product contract before it is treated as a component diagram. The system needs to do the following:

  • Render map tiles at multiple zoom levels.
  • Search and display places.
  • Compute routes between points.
  • Incorporate traffic updates.
  • Support mobile clients with intermittent connectivity.

2. Put numbers on the shape

The exact numbers depend on the interviewer, but these are the estimates that should steer the architecture:

  • Pre-rendered tiles create enormous but cacheable static content.
  • Routing graph size is large but changes slower than live traffic.
  • Traffic updates are streaming data and should not rewrite the base map graph constantly.
  • Popular tiles have excellent CDN cache locality.

5. Linearized design

Walk the design in the order a request or event actually moves. This keeps the interview grounded and exposes the first bottleneck naturally.

  1. 1. Store base map data and generate tiles by zoom, x, and y.
  2. 2. Serve tiles through CDN because they are static and geographically cacheable.
  3. 3. Represent roads as a graph with hierarchy for faster route search.
  4. 4. Overlay live traffic as edge weights or segment penalties.
  5. 5. Compute routes with a routing service that uses base graph plus traffic snapshot.
  6. 6. Let mobile clients cache tiles and recent routes for offline resilience.

6. Deep dives

Tiles vs vectors

Raster tiles are simple to serve and cache. Vector tiles push rendering to clients and allow richer styling, but increase client complexity.

Routing hierarchy

A naive shortest path over a continent-scale graph is too slow. Road hierarchy and precomputed shortcuts shrink the search space.

Traffic as overlay

Live traffic should adjust weights, not mutate the canonical road graph. That separation keeps updates fast and reversible.

9. What to say in the interview

Strong signal
Lead with the dominant constraint: Map tiles, routing, traffic updates, storage, rendering, offline/cache behavior. Then defend the design by naming what you made faster, what you made more reliable, and what complexity you accepted.
