Match buy and sell orders

The matching algorithm is the exchange's kernel. Before scaling it, you must be able to explain exactly which order trades first and why.

0. Interview hinge

The hinge is the moment where the prompt stops being generic and the architecture becomes forced. For this case, that hinge is: Price-time priority, order types, deterministic replay, correctness before scale.

Invariant to protect
The design is only correct if the user-visible contract survives retries, partial failures, stale caches, and reordering. A concrete invariant to defend here: Non-deterministic replay must be handled by Make matching depend only on sequenced inputs and deterministic state..
Tempting but wrong
A common shortcut is to choose Array price levels vs tree map just because it buys fast known tick range. That shortcut fails when its cost matters: memory for sparse prices. Use it only when the workload matches the condition: bounded price instruments.

1. Clarify the contract

The prompt is best treated as a product contract before it is treated as a component diagram. The system needs to do the following:

  • Maintain buy and sell books by price level.
  • Match marketable orders by price-time priority.
  • Support limit, market, cancel, and partial fill.
  • Produce trades and remaining resting orders.
  • Replay deterministically from input events.

2. Put numbers on the shape

The exact numbers depend on the interviewer, but these are the estimates that should steer the architecture:

  • Most operations touch the best price level and one order queue.
  • Cancel performance depends on direct order ID lookup.
  • Partial fills can generate multiple trade events per input order.
  • The order book for one symbol fits in memory in many designs.

5. Linearized design

Walk the design in the order a request or event actually moves. This keeps the interview grounded and exposes the first bottleneck naturally.

  1. 1. Sequence every accepted order command.
  2. 2. Keep bids sorted descending and asks sorted ascending.
  3. 3. Within each price level, keep FIFO order queues.
  4. 4. When a buy crosses the best ask or a sell crosses the best bid, generate trades until quantity is filled or no longer marketable.
  5. 5. Insert any unfilled limit quantity into the appropriate price level.
  6. 6. Log all commands and resulting trades for replay and audit.

6. Deep dives

Price-time priority

Better price wins first; within equal price, earlier order wins. This simple rule is why sequence numbers and FIFO queues matter.

Cancel path

A cancel should not scan the whole book. Maintain an order ID map to the price level and queue node or tombstone efficiently.

Market order risk

Market orders can walk the book and execute at surprising prices. Systems often include collars or risk checks.
