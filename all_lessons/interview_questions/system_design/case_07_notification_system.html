Design a notification system

Notifications are product policy plus distributed delivery. The system must decide not only how to send, but whether sending is still valuable.

Source: Vol. 1 Ch. 10, archive Case drill Trade-off first
How to read this case
This page is an original educational walkthrough based on the case topic, not a transcription. Read it linearly: contract, estimates, data, flows, bottlenecks, trade-offs, then failure modes.

0. Interview hinge

The hinge is the moment where the prompt stops being generic and the architecture becomes forced. For this case, that hinge is: Fanout, device tokens, templates, preferences, retry, dedupe, channel priority.

Invariant to protect
The design is only correct if the user-visible contract survives retries, partial failures, stale caches, and reordering. A concrete invariant to defend here: Provider outage must be handled by Circuit-break that provider and retry with backoff or fallback channel..
Tempting but wrong
A common shortcut is to choose Synchronous send vs queued send just because it buys immediate feedback. That shortcut fails when its cost matters: slow producer and provider coupling. Use it only when the workload matches the condition: tiny internal notifications.

1. Clarify the contract

The prompt is best treated as a product contract before it is treated as a component diagram. The system needs to do the following:

  • Send push, email, SMS, or in-app notifications.
  • Respect user preferences, quiet hours, and legal consent.
  • Deduplicate repeated events.
  • Retry transient provider failures without spamming users.
  • Track delivery, open, and failure events.

2. Put numbers on the shape

The exact numbers depend on the interviewer, but these are the estimates that should steer the architecture:

  • Fanout can turn one celebrity event into millions of notifications.
  • Provider APIs have quotas and variable latency; synchronous sends will not scale.
  • Template rendering and personalization can dominate CPU at high volume.
  • SMS has real cost per message, so policy bugs become money leaks.

3. Define the surface area

Keep the API small. The API should express the product operation, not leak the internal queue, cache, or shard layout.

5. Linearized design

Walk the design in the order a request or event actually moves. This keeps the interview grounded and exposes the first bottleneck naturally.

  1. 1. Ingest notification intents from product services, not raw provider calls.
  2. 2. Evaluate preference, consent, dedupe, and priority before queueing channel work.
  3. 3. Render templates with stable snapshot data so retries do not drift semantically.
  4. 4. Fan out to channel-specific queues with independent rate limits.
  5. 5. Send through provider adapters that normalize success, retryable failure, and permanent failure.
  6. 6. Record attempts and feedback events for analytics and future suppression.

6. Deep dives

Policy before delivery

A notification service that only sends messages becomes a spam cannon. Preferences, quiet hours, and priority are first-class data models.

Retry semantics

At-least-once queues mean duplicates are normal. Every send needs an idempotency key at the provider adapter and a dedupe key at the user-event level.

Fanout control

For huge fanouts, create batches or segments rather than one queue message per user immediately. This lets you pause, rate limit, and cancel campaigns.

9. What to say in the interview

Strong signal
Lead with the dominant constraint: Fanout, device tokens, templates, preferences, retry, dedupe, channel priority. Then defend the design by naming what you made faster, what you made more reliable, and what complexity you accepted.
