Design nearby friends

Nearby friends differs from nearby places because the indexed objects move constantly. The system is about streaming updates, not static search.

Source: Vol. 2 Ch. 2 Case drill Trade-off first

0. Interview hinge

The hinge is the moment where the prompt stops being generic and the architecture becomes forced. For this case, that hinge is: Location streams, pub/sub, friend graph filtering, online state, battery trade-offs.

Invariant to protect
The design is only correct if the user-visible contract survives retries, partial failures, stale caches, and reordering. A concrete invariant to defend here: User appears after opt-out must be handled by Make privacy state authoritative and checked at delivery..
Tempting but wrong
A common shortcut is to choose Push vs poll just because it buys freshness. That shortcut fails when its cost matters: connection and fanout cost. Use it only when the workload matches the condition: small active friend sets.

1. Clarify the contract

The prompt is best treated as a product contract before it is treated as a component diagram. The system needs to do the following:

  • Show nearby opted-in friends.
  • Update locations every few seconds or minutes.
  • Respect privacy and visibility settings.
  • Avoid draining mobile battery.
  • Scale to many concurrent mobile users.

2. Put numbers on the shape

The exact numbers depend on the interviewer, but these are the estimates that should steer the architecture:

  • If 10M users update every 10 seconds, that is 1M location updates/s.
  • Most friends are not nearby; filtering by graph before broadcast saves work.
  • Battery and network constraints may matter more than server CPU.
  • Location precision can be coarser than GPS for privacy and cost.

5. Linearized design

Walk the design in the order a request or event actually moves. This keeps the interview grounded and exposes the first bottleneck naturally.

  1. 1. Require explicit opt-in and store visibility policy.
  2. 2. Mobile clients send throttled, accuracy-aware location updates.
  3. 3. Store latest active location in a low-latency location cache.
  4. 4. Map active users into geospatial cells for coarse candidate lookup.
  5. 5. Intersect nearby candidates with the friend graph and visibility policy.
  6. 6. Push changes over pub/sub or let clients poll depending on freshness needs.

6. Deep dives

Dynamic index

Unlike businesses, users move. Updating a durable geospatial index for every ping is expensive. Keep latest active locations in memory and persist history only if the product needs it.

Graph filter placement

Broadcasting every location update to everyone in a cell is wasteful and risky. Intersect with friendship and privacy before delivery.

Battery trade

The best backend design can be ruined by aggressive GPS use. Adaptive update frequency is part of the system.

9. What to say in the interview

Strong signal
