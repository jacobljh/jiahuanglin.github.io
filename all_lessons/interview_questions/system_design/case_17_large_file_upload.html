Design large file upload and serving

Large upload design is about making failure normal. Networks drop, browsers close, and users still expect not to restart a 20 GB upload.

Source: Archive Case drill Trade-off first

0. Interview hinge

The hinge is the moment where the prompt stops being generic and the architecture becomes forced. For this case, that hinge is: Multipart upload, resumability, checksums, CDN, object lifecycle policy.

Invariant to protect
The design is only correct if the user-visible contract survives retries, partial failures, stale caches, and reordering. A concrete invariant to defend here: Client repeats part must be handled by Make part PUT idempotent by upload ID and part number..
Tempting but wrong
A common shortcut is to choose Proxy upload vs direct upload just because it buys central control. That shortcut fails when its cost matters: application bandwidth bottleneck. Use it only when the workload matches the condition: small files or strict scanning.

1. Clarify the contract

The prompt is best treated as a product contract before it is treated as a component diagram. The system needs to do the following:

  • Upload large files over unreliable networks.
  • Resume after interruption.
  • Verify integrity.
  • Serve downloads efficiently.
  • Clean up abandoned partial uploads.

2. Put numbers on the shape

The exact numbers depend on the interviewer, but these are the estimates that should steer the architecture:

  • A 10 GB upload over 20 Mbps takes more than an hour in ideal conditions.
  • Retrying the whole file after one failure is unacceptable.
  • Small chunk size increases metadata and request overhead; huge chunks waste retry bandwidth.
  • Download traffic should be served from object storage or CDN, not application servers.

5. Linearized design

Walk the design in the order a request or event actually moves. This keeps the interview grounded and exposes the first bottleneck naturally.

  1. 1. Create an upload session with target file metadata and expiration.
  2. 2. Let clients upload numbered chunks directly to object storage using signed URLs.
  3. 3. Verify each part checksum and record completion.
  4. 4. On complete, validate the manifest and atomically publish the file.
  5. 5. Serve downloads through signed URLs and CDN when public or shareable.
  6. 6. Run cleanup for expired sessions and orphaned parts.

6. Deep dives

Chunk sizing

The chunk size is a cost knob. Smaller chunks improve resume granularity and parallelism; larger chunks reduce metadata and request overhead.

Direct-to-object-store upload

Application servers should issue credentials and metadata, not proxy every byte. That keeps the control plane separate from the data plane.

Integrity

A successful HTTP upload does not prove the right bytes landed. Checksums per part and final manifest checks protect both client and storage bugs.

9. What to say in the interview

Strong signal
Lead with the dominant constraint: Multipart upload, resumability, checksums, CDN, object lifecycle policy. Then defend the design by naming what you made faster, what you made more reliable, and what complexity you accepted.
