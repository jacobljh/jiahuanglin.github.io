Debugging CPU, memory, and IO incidents

Incident debugging is system design under time pressure. The winning move is to locate the saturated resource and the blast radius before chasing theories.

0. Interview hinge

The hinge is the moment where the prompt stops being generic and the architecture becomes forced. For this case, that hinge is: Symptom-first diagnosis, telemetry, saturation, deployment rollback.

Invariant to protect
The design is only correct if the user-visible contract survives retries, partial failures, stale caches, and reordering. A concrete invariant to defend here: Debug query overloads production must be handled by Use sampled logs and replicas; cap expensive queries..
Tempting but wrong
A common shortcut is to choose Rollback vs debug live just because it buys fast recovery. That shortcut fails when its cost matters: less immediate evidence. Use it only when the workload matches the condition: clear recent deploy correlation.

1. Clarify the contract

The prompt is best treated as a product contract before it is treated as a component diagram. The system needs to do the following:

  • Diagnose CPU, memory, disk, network, and dependency saturation.
  • Separate symptom alerts from cause exploration.
  • Roll back or mitigate safely.
  • Preserve evidence for postmortem.
  • Avoid making the incident worse with heavy debugging.

2. Put numbers on the shape

The exact numbers depend on the interviewer, but these are the estimates that should steer the architecture:

  • A service at 99 percent CPU has little room for expensive introspection.
  • Queue depth and latency often reveal saturation before error rate spikes.
  • One bad deployment can look like a database outage if every replica calls the database harder.
  • Debug sampling must be small enough not to become the load.

5. Linearized design

Walk the design in the order a request or event actually moves. This keeps the interview grounded and exposes the first bottleneck naturally.

  1. 1. Start from user-facing SLO burn or symptom metrics.
  2. 2. Identify blast radius: service, region, shard, tenant, or version.
  3. 3. Check saturation signals: CPU, memory, disk IO, network, queue depth, connection pools.
  4. 4. Correlate with recent deploys, config changes, traffic changes, and dependency health.
  5. 5. Mitigate first through rollback, traffic shift, rate limit, cache, or feature flag.
  6. 6. Capture a timeline and follow up with root-cause analysis after recovery.

6. Deep dives

CPU high

High CPU can be real work, retry loops, lock contention, serialization, or GC. Compare request rate, latency, and per-endpoint CPU before optimizing code.

Memory high

Memory pressure may be leak, cache growth, queue backlog, or larger payloads. Look at heap, RSS, allocation rate, and eviction behavior.

IO high

Disk or network saturation often appears as latency. Queueing theory matters: once utilization approaches 1, tiny load increases create huge waits.

9. What to say in the interview

Strong signal
Lead with the dominant constraint: Symptom-first diagnosis, telemetry, saturation, deployment rollback. Then defend the design by naming what you made faster, what you made more reliable, and what complexity you accepted.
