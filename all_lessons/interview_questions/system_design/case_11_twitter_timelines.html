Design Twitter-style timelines

A Twitter-like timeline combines feed fanout, graph scale, ranking, and product freshness in one prompt. The trick is to name which timeline you are designing.

0. Interview hinge

The hinge is the moment where the prompt stops being generic and the architecture becomes forced. For this case, that hinge is: Hybrid fanout, social graph reads, ranking, backfill, cache invalidation.

Invariant to protect
The design is only correct if the user-visible contract survives retries, partial failures, stale caches, and reordering. A concrete invariant to defend here: Celebrity overload must be handled by Use pull path for high-follower accounts..
Tempting but wrong
A common shortcut is to choose Push model vs pull model just because it buys fast home reads. That shortcut fails when its cost matters: fanout explosion. Use it only when the workload matches the condition: normal authors.

1. Clarify the contract

The prompt is best treated as a product contract before it is treated as a component diagram. The system needs to do the following:

  • Post short messages and media references.
  • Serve home timeline, user profile timeline, and mentions.
  • Handle follows, unfollows, blocks, and mutes.
  • Support high-follower accounts.
  • Expose near-real-time updates without making every read expensive.

2. Put numbers on the shape

The exact numbers depend on the interviewer, but these are the estimates that should steer the architecture:

  • Read QPS usually dominates write QPS.
  • A single viral author can create more fanout work than millions of ordinary users.
  • Home timelines stored as post IDs are much smaller than duplicated post bodies.
  • Graph queries on the hot path become costly when users follow thousands of accounts.

5. Linearized design

Walk the design in the order a request or event actually moves. This keeps the interview grounded and exposes the first bottleneck naturally.

  1. 1. Store tweets canonically by author and time.
  2. 2. Keep profile timelines as append-only author streams.
  3. 3. Fan out ordinary tweets to follower home timeline caches.
  4. 4. For celebrities, read their profile stream during timeline assembly.
  5. 5. Apply blocks, mutes, and visibility rules at hydration.
  6. 6. Rank a bounded candidate set and return stable cursors.

6. Deep dives

Three timelines

Profile timeline is easy: append by author. Home timeline is hard: personalize across follows. Mentions add notification-like semantics. Separate them or the design muddies.

Graph mutations

Follow and unfollow do not need to rewrite infinite history. Backfill recent posts on follow and filter future reads for unfollow or block.

Cache invalidation

Do not delete a post ID from every timeline on deletion. Mark canonical tweet visibility and filter during hydration; clean asynchronously.

9. What to say in the interview

Strong signal
Lead with the dominant constraint: Hybrid fanout, social graph reads, ranking, backfill, cache invalidation. Then defend the design by naming what you made faster, what you made more reliable, and what complexity you accepted.
