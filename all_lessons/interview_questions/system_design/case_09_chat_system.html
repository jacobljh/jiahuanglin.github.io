Design a chat system

Chat teaches the difference between sending, storing, delivering, and reading. Users call it one message; the system sees multiple contracts.

0. Interview hinge

The hinge is the moment where the prompt stops being generic and the architecture becomes forced. For this case, that hinge is: WebSockets, message ordering, online presence, offline sync, multi-device state.

Invariant to protect
The design is only correct if the user-visible contract survives retries, partial failures, stale caches, and reordering. A concrete invariant to defend here: Duplicate send from retry must be handled by Use client message IDs for idempotency..
Tempting but wrong
A common shortcut is to choose WebSocket vs polling just because it buys low latency. That shortcut fails when its cost matters: connection state and load balancing complexity. Use it only when the workload matches the condition: active chat.

1. Clarify the contract

The prompt is best treated as a product contract before it is treated as a component diagram. The system needs to do the following:

  • Support one-to-one and group conversations.
  • Deliver messages in near real time when users are online.
  • Store history for offline and multi-device sync.
  • Show delivery/read state when required.
  • Preserve order within a conversation.

2. Put numbers on the shape

The exact numbers depend on the interviewer, but these are the estimates that should steer the architecture:

  • Concurrent connections can be much larger than message QPS.
  • A million idle WebSockets is mostly memory and kernel tuning, not CPU.
  • Group messages multiply delivery events but should store the message body once.
  • Read receipts can exceed message volume in active groups.

5. Linearized design

Walk the design in the order a request or event actually moves. This keeps the interview grounded and exposes the first bottleneck naturally.

  1. 1. Terminate persistent connections at gateway nodes.
  2. 2. Route send requests to a conversation owner or partition.
  3. 3. Assign a monotonically increasing sequence number per conversation.
  4. 4. Persist the message before fanout so offline devices can sync later.
  5. 5. Publish delivery events to online recipient connection gateways.
  6. 6. Let clients reconcile through sync cursors rather than trusting push delivery.

6. Deep dives

Ordering

Global order is unnecessary and expensive. Per-conversation order is the right contract. A single sequencer per conversation partition is simpler than distributed total ordering.

Push is not storage

WebSocket delivery can fail silently when mobile networks switch. The source of truth is the message log plus client cursors.

Presence

Presence is soft state. Heartbeats and TTLs are enough; do not use consensus to decide whether someone is online.

9. What to say in the interview

Strong signal
Lead with the dominant constraint: WebSockets, message ordering, online presence, offline sync, multi-device state. Then defend the design by naming what you made faster, what you made more reliable, and what complexity you accepted.
