Money movement and reconciliation

Money movement finishes only when independent records agree. Reconciliation is the system admitting that external truth exists.

Source: Archive Case drill Trade-off first
How to read this case
This page is an original educational walkthrough based on the case topic, not a transcription. Read it linearly: contract, estimates, data, flows, bottlenecks, trade-offs, then failure modes.

0. Interview hinge

The hinge is the moment where the prompt stops being generic and the architecture becomes forced. For this case, that hinge is: Settlement, posting, mismatch detection, retry, compensating correction.

Invariant to protect
The design is only correct if the user-visible contract survives retries, partial failures, stale caches, and reordering. A concrete invariant to defend here: Provider report missing must be handled by Alert on expected file absence and block close of accounting period..
Tempting but wrong
A common shortcut is to choose Realtime recon vs batch recon just because it buys faster detection. That shortcut fails when its cost matters: more integration complexity. Use it only when the workload matches the condition: high-risk rails.

1. Clarify the contract

The prompt is best treated as a product contract before it is treated as a component diagram. The system needs to do the following:

  • Track internal payment or transfer state.
  • Ingest external provider or bank reports.
  • Detect missing, duplicate, or mismatched records.
  • Create correction workflows.
  • Produce auditable reconciliation output.

2. Put numbers on the shape

The exact numbers depend on the interviewer, but these are the estimates that should steer the architecture:

  • Provider reports may arrive daily while user actions happen instantly.
  • Mismatch volume should be tiny; investigation tooling matters more than throughput.
  • Raw reports must be retained because parsing logic can change.
  • Retries without reconciliation hide silent financial drift.

5. Linearized design

Walk the design in the order a request or event actually moves. This keeps the interview grounded and exposes the first bottleneck naturally.

  1. 1. Store every internal money event with provider reference IDs.
  2. 2. Ingest external reports immutably with file checksums and parse versions.
  3. 3. Match external rows to internal records by stable references and amounts.
  4. 4. Classify mismatches: missing internal, missing external, amount mismatch, duplicate.
  5. 5. Create investigation cases and compensating ledger entries when needed.
  6. 6. Track reconciliation completeness by provider, date, and currency.

6. Deep dives

Internal truth vs external truth

Your ledger knows what you intended. The bank or PSP report knows what settled. Reconciliation brings those views together.

Matching keys

A good payment design carries stable references through every external call. Without them, reconciliation becomes fuzzy matching, which is dangerous for money.

Corrections

Do not patch history. Post correcting entries and link them to the reconciliation case.
