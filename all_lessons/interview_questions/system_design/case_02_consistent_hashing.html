Design consistent hashing

Consistent hashing is the idea you reach for when ownership must change gradually as nodes enter or leave. It is less about hashing and more about reducing operational shock.

0. Interview hinge

The hinge is the moment where the prompt stops being generic and the architecture becomes forced. For this case, that hinge is: Rings, virtual nodes, remapping cost, hot shards, cache/database ownership.

Invariant to protect
The design is only correct if the user-visible contract survives retries, partial failures, stale caches, and reordering. A concrete invariant to defend here: Ring split-brain must be handled by Store membership in consensus-backed config and version every decision..
Tempting but wrong
A common shortcut is to choose Client-side routing vs router tier just because it buys one less network hop. That shortcut fails when its cost matters: clients must refresh ring metadata. Use it only when the workload matches the condition: large internal fleets with controlled clients.

1. Clarify the contract

The prompt is best treated as a product contract before it is treated as a component diagram. The system needs to do the following:

  • Map keys to nodes with near-even distribution.
  • Move only a small fraction of keys when capacity changes.
  • Support heterogeneous node sizes.
  • Let clients or routers find the owner quickly.
  • Handle node failure without moving the whole world.

2. Put numbers on the shape

The exact numbers depend on the interviewer, but these are the estimates that should steer the architecture:

  • Modulo hashing remaps most keys on node count changes.
  • With N equal nodes, adding one node should move roughly 1/(N+1) of keys.
  • Virtual nodes smooth variance; hundreds per physical node is common in teaching examples.
  • Replication multiplies ownership metadata but reduces availability risk.

5. Linearized design

Walk the design in the order a request or event actually moves. This keeps the interview grounded and exposes the first bottleneck naturally.

  1. 1. Hash each key onto a large circular space.
  2. 2. Hash each node into many virtual positions, proportional to weight.
  3. 3. Assign a key to the first virtual node clockwise from the key hash.
  4. 4. For replication, continue clockwise while skipping duplicate physical nodes.
  5. 5. Keep ring metadata in a strongly consistent config store, then cache it in clients.
  6. 6. During membership changes, stream only the ranges whose ownership changed.

6. Deep dives

Why virtual nodes exist

One point per machine creates lumpy ownership. Virtual nodes turn one large random draw into many small draws, so each physical machine receives a smoother share of the keyspace.

Replication on the ring

Replication should choose distinct failure domains, not just adjacent virtual nodes. If adjacent replicas live on the same rack, the ring looks balanced while availability is fake.

Operational movement

The real design is the rebalancer. New owners must copy data, verify checksums, then accept reads or writes. Ownership metadata should not flip before data is ready.

Strong signal
