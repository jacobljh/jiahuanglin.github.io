Design Google Drive

Drive is a metadata system wrapped around object storage. The hardest user-visible feature is sync correctness, not file bytes.

Source: Vol. 1 Ch. 15 Case drill Trade-off first

0. Interview hinge

The hinge is the moment where the prompt stops being generic and the architecture becomes forced. For this case, that hinge is: Large file upload, chunking, sync, metadata, conflict resolution, sharing.

Invariant to protect
The design is only correct if the user-visible contract survives retries, partial failures, stale caches, and reordering. A concrete invariant to defend here: Upload commit races must be handled by Use idempotent commit with file version preconditions..
Tempting but wrong
A common shortcut is to choose Chunk dedupe vs simple blobs just because it buys storage savings. That shortcut fails when its cost matters: more metadata and privacy concerns. Use it only when the workload matches the condition: large repeated files.

1. Clarify the contract

The prompt is best treated as a product contract before it is treated as a component diagram. The system needs to do the following:

  • Upload, download, rename, move, and delete files.
  • Sync changes across devices.
  • Share files and folders with access control.
  • Support large files and resumable uploads.
  • Preserve versions or recover deleted files.

2. Put numbers on the shape

The exact numbers depend on the interviewer, but these are the estimates that should steer the architecture:

  • File bytes dominate storage; metadata operations dominate QPS.
  • Chunking large files enables resume and dedupe but increases metadata.
  • Sync clients may reconnect after days with many missed changes.
  • Sharing checks must be fast enough for every metadata and download request.

5. Linearized design

Walk the design in the order a request or event actually moves. This keeps the interview grounded and exposes the first bottleneck naturally.

  1. 1. Store file bytes as chunks in object storage.
  2. 2. Store metadata, hierarchy, versions, and permissions in a transactional metadata service.
  3. 3. Commit uploads by atomically linking uploaded chunks to a new file version.
  4. 4. Append every metadata mutation to a per-user or per-namespace change log.
  5. 5. Let sync clients poll or subscribe using durable cursors.
  6. 6. Authorize downloads by checking metadata permissions before issuing signed blob URLs.

6. Deep dives

Metadata is the source of truth

The object store may contain uncommitted chunks. A file exists for users only after metadata commit. This prevents half-uploaded files from leaking.

Sync log

A sync client needs changes since cursor, not a full tree scan. The change log is derived from metadata mutations and must survive client absence.

Conflict handling

Two offline edits to the same file cannot both be the same version. Create conflict copies or explicit merge flows rather than silently overwriting.

9. What to say in the interview

Strong signal
Lead with the dominant constraint: Large file upload, chunking, sync, metadata, conflict resolution, sharing. Then defend the design by naming what you made faster, what you made more reliable, and what complexity you accepted.
