Design a rate limiter

A rate limiter is the first serious edge system because the product wants bursts, the infrastructure wants smooth traffic, and attackers exploit any ambiguity between the two.

Source: Vol. 1 Ch. 4 Case drill Trade-off first

0. Interview hinge

The hinge is the moment where the prompt stops being generic and the architecture becomes forced. For this case, that hinge is: Token bucket, sliding windows, shared counters, edge placement, abuse protection.

Invariant to protect
The design is only correct if the user-visible contract survives retries, partial failures, stale caches, and reordering. A concrete invariant to defend here: Clock skew between nodes must be handled by Use server-side time from the counter backend or monotonic bucket refill..
Tempting but wrong
A common shortcut is to choose Token bucket vs sliding window just because it buys bursts and constant memory. That shortcut fails when its cost matters: less intuitive fairness at exact window boundaries. Use it only when the workload matches the condition: public APIs with legitimate bursty clients.

1. Clarify the contract

The prompt is best treated as a product contract before it is treated as a component diagram. The system needs to do the following:

  • Limit per API key, user, IP, route, or tenant.
  • Allow short bursts while enforcing a longer-term budget.
  • Return a deterministic allow/deny decision within a few milliseconds.
  • Work across many stateless API servers.
  • Expose remaining quota and retry-after metadata when useful.

2. Put numbers on the shape

The exact numbers depend on the interviewer, but these are the estimates that should steer the architecture:

  • Decision QPS is at least request QPS; a 100k req/s API needs 100k limiter checks/s.
  • A 16-byte counter key plus metadata for 10M active keys is hundreds of MB before replication.
  • One remote Redis round trip can cost more latency than the application handler for cheap endpoints.
  • Per-request writes create hot keys for large tenants, so batching or local admission can matter.

3. Define the surface area

5. Linearized design

Walk the design in the order a request or event actually moves. This keeps the interview grounded and exposes the first bottleneck naturally.

  1. 1. Normalize identity at the gateway: authenticated user beats IP, route policy beats global default.
  2. 2. Read the policy from a local cache so the hot path only mutates counters.
  3. 3. Use token bucket for burst-friendly APIs; use sliding window when fairness at boundaries matters.
  4. 4. Keep the decision service stateless and store shared counters in Redis or a purpose-built in-memory cluster.
  5. 5. Return headers that tell clients how to back off, but never reveal enough to help attackers tune around the limit.
  6. 6. Export allow, deny, near-limit, and backend-error metrics by policy, not by raw user ID.

6. Deep dives

Algorithm choice

Fixed windows are cheap but unfair at boundaries. Sliding logs are fair but store too much. Sliding-window counters are a useful middle. Token buckets are usually the interview default because they separate average rate from burst capacity.

Distributed counters

A shared counter is simple until one tenant becomes a hot key. Local warm buckets reduce latency but permit overshoot. Centralized Redis keeps decisions tighter but adds a dependency to every request.

Fail-open or fail-closed

For login, payment, and abuse-sensitive endpoints, fail closed or degrade to a tiny local budget. For low-risk read endpoints, fail open with alerting may protect availability better than perfect enforcement.

9. What to say in the interview

Strong signal
Lead with the dominant constraint: Token bucket, sliding windows, shared counters, edge placement, abuse protection. Then defend the design by naming what you made faster, what you made more reliable, and what complexity you accepted.
