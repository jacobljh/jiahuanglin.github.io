Avoid double charges

Double charge prevention is the smallest payment design that proves whether you understand retries, state, and external side effects.

Source: Archive Case drill Trade-off first
How to read this case
This page is an original educational walkthrough based on the case topic, not a transcription. Read it linearly: contract, estimates, data, flows, bottlenecks, trade-offs, then failure modes.

0. Interview hinge

The hinge is the moment where the prompt stops being generic and the architecture becomes forced. For this case, that hinge is: Idempotency keys, outbox, state machine, duplicate webhook handling.

Invariant to protect
The design is only correct if the user-visible contract survives retries, partial failures, stale caches, and reordering. A concrete invariant to defend here: Same key different amount must be handled by Store request hash and reject mismatches..
Tempting but wrong
A common shortcut is to choose Permanent idempotency vs TTL just because it buys strong support history. That shortcut fails when its cost matters: storage growth. Use it only when the workload matches the condition: financial operations.

1. Clarify the contract

The prompt is best treated as a product contract before it is treated as a component diagram. The system needs to do the following:

  • Let clients retry safely after timeout.
  • Ensure one business intent creates at most one successful charge.
  • Handle duplicate provider webhooks.
  • Expose clear pending, succeeded, failed, and refunded states.
  • Support audit and customer support investigation.

2. Put numbers on the shape

The exact numbers depend on the interviewer, but these are the estimates that should steer the architecture:

  • Timeouts are common enough that retry must be a normal path.
  • Idempotency records need TTL or permanent retention depending on dispute window.
  • Webhook duplicates are guaranteed at scale.
  • Support tickets cost far more than storing idempotency metadata.

5. Linearized design

Walk the design in the order a request or event actually moves. This keeps the interview grounded and exposes the first bottleneck naturally.

  1. 1. Create a payment intent before contacting the provider.
  2. 2. Bind each client retry to an idempotency key and request fingerprint.
  3. 3. If the key exists, return the original result or current pending state.
  4. 4. Send provider requests from a durable attempt record.
  5. 5. Process webhooks idempotently by provider event ID.
  6. 6. Allow only valid state-machine transitions.

6. Deep dives

Timeout is unknown

A client timeout does not mean the provider charge failed. The only safe response is to query state or retry with the same idempotency key.

Request fingerprint

The same idempotency key with a different amount or merchant should be rejected. Otherwise a client bug can mask a dangerous mutation.

State machine

Payment states should move forward through allowed transitions. Free-form status updates invite impossible combinations.

9. What to say in the interview

Strong signal
Lead with the dominant constraint: Idempotency keys, outbox, state machine, duplicate webhook handling. Then defend the design by naming what you made faster, what you made more reliable, and what complexity you accepted.
