Block, file, and object storage primer

Choosing storage is easier when you ask what the interface promises: raw blocks, hierarchical files, or immutable-ish objects.

How to read this case

0. Interview hinge

The hinge is the moment where the prompt stops being generic and the architecture becomes forced. For this case, that hinge is: Storage interface trade-offs and how they drive product design choices.

Invariant to protect
The design is only correct if the user-visible contract survives retries, partial failures, stale caches, and reordering. A concrete invariant to defend here: Using object store for append-heavy small writes must be handled by Buffer and batch into larger objects or use a log system..
Tempting but wrong
A common shortcut is to choose Block vs object just because it buys random IO and low latency. That shortcut fails when its cost matters: more management. Use it only when the workload matches the condition: databases and VM disks.

1. Clarify the contract

The prompt is best treated as a product contract before it is treated as a component diagram. The system needs to do the following:

  • Explain block, file, and object storage clearly.
  • Map product workloads to storage interfaces.
  • Reason about latency, throughput, sharing, and metadata.
  • Avoid using one storage type because its brand name is familiar.
  • Know what each interface makes hard.

2. Put numbers on the shape

The exact numbers depend on the interviewer, but these are the estimates that should steer the architecture:

  • Block storage offers low-level control but expects a filesystem or database above it.
  • File storage centralizes hierarchy and POSIX-like semantics but can bottleneck metadata.
  • Object storage scales cheaply but is not a mutable filesystem.
  • Metadata operation QPS matters as much as byte throughput for small files.

5. Linearized design

Walk the design in the order a request or event actually moves. This keeps the interview grounded and exposes the first bottleneck naturally.

  1. 1. Start from the access pattern: random writes, shared files, or whole-object reads.
  2. 2. Use block storage under databases and filesystems.
  3. 3. Use file storage when many clients need shared hierarchical files.
  4. 4. Use object storage for blobs, backups, media, logs, and static assets.
  5. 5. Keep mutable metadata in a database when object storage holds bytes.
  6. 6. Design lifecycle, retention, and replication based on the storage interface.

6. Deep dives

Object storage is not a filesystem

You can emulate folders with prefixes, but rename, append, and fine-grained mutation are metadata illusions. Build product semantics above it deliberately.

Metadata bottleneck

File systems make path operations convenient. At very large scale, metadata servers and hot directories become the real scaling problem.

Database on object storage

Databases usually need low-latency random IO and fsync semantics. Object storage can back backups or lake tables, but not a naive OLTP data directory.

9. What to say in the interview

Strong signal
Lead with the dominant constraint: Storage interface trade-offs and how they drive product design choices. Then defend the design by naming what you made faster, what you made more reliable, and what complexity you accepted.
