Design a realtime gaming leaderboard

Leaderboards look like a sorted list until scale asks which list: global, regional, friend-only, seasonal, or anti-cheat verified.

Source: Vol. 2 Ch. 10 Case drill Trade-off first
How to read this case
This page is an original educational walkthrough based on the case topic, not a transcription. Read it linearly: contract, estimates, data, flows, bottlenecks, trade-offs, then failure modes.

0. Interview hinge

The hinge is the moment where the prompt stops being generic and the architecture becomes forced. For this case, that hinge is: Sorted sets, partitioning, top-k, season resets, cheat/fraud hooks.

Invariant to protect
The design is only correct if the user-visible contract survives retries, partial failures, stale caches, and reordering. A concrete invariant to defend here: Cheater takes top rank must be handled by Quarantine suspicious scores until verified..
Tempting but wrong
A common shortcut is to choose Exact rank vs approximate just because it buys precision. That shortcut fails when its cost matters: cost. Use it only when the workload matches the condition: top competitive boards.

1. Clarify the contract

The prompt is best treated as a product contract before it is treated as a component diagram. The system needs to do the following:

  • Update player scores quickly.
  • Return global and nearby rank.
  • Support seasons or time windows.
  • Handle friend or regional leaderboards.
  • Detect or quarantine suspicious scores.

2. Put numbers on the shape

The exact numbers depend on the interviewer, but these are the estimates that should steer the architecture:

  • Score updates can be frequent but top leaderboard reads are much more cacheable.
  • Global exact rank over millions can be expensive if every update changes ordering.
  • Season resets are metadata operations if scores are namespaced by season.
  • Friend leaderboards can be computed from a small set after fetching friend scores.

5. Linearized design

Walk the design in the order a request or event actually moves. This keeps the interview grounded and exposes the first bottleneck naturally.

  1. 1. Validate score submission and attach game session evidence.
  2. 2. Write the best score per player for the season.
  3. 3. Maintain sorted indexes for hot boards.
  4. 4. Cache top-k results aggressively because many users read the same top page.
  5. 5. Compute friend leaderboards by fetching friend scores and sorting locally.
  6. 6. Route suspicious scores to review or quarantine before public ranking.

6. Deep dives

Exact rank

Exact global rank requires ordered data structures or precomputed ranks. Approximate rank may be acceptable for huge boards except near the top.

Sorted set shape

Redis sorted sets are a good mental model: score -> order, player -> member. At large scale, shard by board or score ranges.

Anti-cheat boundary

The leaderboard should not blindly publish every score. Score validation is part of correctness, not a separate nice-to-have.

Strong signal
