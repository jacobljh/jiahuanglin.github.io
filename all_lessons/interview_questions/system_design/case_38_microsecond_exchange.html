Microsecond-latency exchange architecture

When latency is measured in microseconds, architecture becomes subtraction: remove hops, locks, allocations, syscalls, and ambiguity.

Source: Archive Case drill Trade-off first
How to read this case
This page is an original educational walkthrough based on the case topic, not a transcription. Read it linearly: contract, estimates, data, flows, bottlenecks, trade-offs, then failure modes.

0. Interview hinge

The hinge is the moment where the prompt stops being generic and the architecture becomes forced. For this case, that hinge is: Latency budget, colocated components, single-writer design, recovery logs.

Invariant to protect
The design is only correct if the user-visible contract survives retries, partial failures, stale caches, and reordering. A concrete invariant to defend here: Latency outlier must be handled by Profile p99.99 and remove allocation, locks, and cross-core contention..
Tempting but wrong
A common shortcut is to choose Kernel bypass vs normal sockets just because it buys lower latency. That shortcut fails when its cost matters: operational complexity. Use it only when the workload matches the condition: ultra-low-latency environments.

1. Clarify the contract

The prompt is best treated as a product contract before it is treated as a component diagram. The system needs to do the following:

  • Minimize order-entry latency.
  • Keep matching deterministic and auditable.
  • Avoid unpredictable pauses.
  • Publish market data quickly.
  • Recover without losing accepted orders.

2. Put numbers on the shape

The exact numbers depend on the interviewer, but these are the estimates that should steer the architecture:

  • A single network hop can dominate a microsecond budget.
  • Garbage collection pauses are unacceptable in the hottest path.
  • Batching improves throughput but can add waiting latency.
  • Durable logging can be on the critical path unless carefully engineered.

5. Linearized design

Walk the design in the order a request or event actually moves. This keeps the interview grounded and exposes the first bottleneck naturally.

  1. 1. Place gateways, risk, matching, and market data in the same low-latency environment where possible.
  2. 2. Use compact binary protocols and persistent sessions.
  3. 3. Preallocate hot-path memory and avoid dynamic allocation.
  4. 4. Run a single-writer event loop per book partition.
  5. 5. Append sequenced events to a low-latency journal.
  6. 6. Publish market data from the same ordered event stream.

6. Deep dives

Latency budget

Write the budget per hop. If gateway parsing, risk, sequencing, matching, journaling, and market data each get a few microseconds, ordinary service boundaries are too expensive.

Mechanical sympathy

CPU cache locality, kernel bypass, pinned threads, and preallocated memory matter because variance is the enemy.

Recovery

Low latency does not remove durability. It forces a design where the recovery log is compact, sequential, and replayable.

9. What to say in the interview

Strong signal
Lead with the dominant constraint: Latency budget, colocated components, single-writer design, recovery logs. Then defend the design by naming what you made faster, what you made more reliable, and what complexity you accepted.
