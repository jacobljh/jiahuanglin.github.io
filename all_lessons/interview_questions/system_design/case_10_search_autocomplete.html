Design search autocomplete

Autocomplete is a latency exercise: every keystroke asks a ranked retrieval system to answer before the user notices it exists.

0. Interview hinge

The hinge is the moment where the prompt stops being generic and the architecture becomes forced. For this case, that hinge is: Trie/FST style serving, prefix aggregation, personalization, updates, latency.

Invariant to protect
The design is only correct if the user-visible contract survives retries, partial failures, stale caches, and reordering. A concrete invariant to defend here: Unsafe suggestion appears must be handled by Maintain an emergency blocklist checked at serving time..
Tempting but wrong
A common shortcut is to choose Batch index vs realtime update just because it buys stable fast serving. That shortcut fails when its cost matters: freshness lag. Use it only when the workload matches the condition: search suggestions from query logs.

1. Clarify the contract

The prompt is best treated as a product contract before it is treated as a component diagram. The system needs to do the following:

  • Return top suggestions for a typed prefix.
  • Stay within tens of milliseconds at p99.
  • Update popular suggestions regularly.
  • Support localization and optionally personalization.
  • Filter unsafe or disallowed suggestions.

2. Put numbers on the shape

The exact numbers depend on the interviewer, but these are the estimates that should steer the architecture:

  • Keystroke QPS can be 5-10x final search QPS.
  • Top-k per prefix can be precomputed for common prefixes to avoid runtime ranking.
  • A trie over all queries may be large; compressed automata or prefix tables reduce memory.
  • Personalization multiplies cache keys and can destroy hit rate if done too early.

5. Linearized design

Walk the design in the order a request or event actually moves. This keeps the interview grounded and exposes the first bottleneck naturally.

  1. 1. Collect query logs and engagement signals.
  2. 2. Batch aggregate counts by normalized query and locale.
  3. 3. Build a prefix index that stores top-k suggestions per prefix.
  4. 4. Deploy immutable index shards to serving nodes.
  5. 5. Serve each request from memory, then apply safety and lightweight personalization.
  6. 6. Refresh indexes periodically and use a delta path for urgent blocks.

6. Deep dives

Precompute top-k

The trick is moving ranking out of the keystroke path. Runtime work should be prefix lookup plus small filters, not scanning all matching queries.

Trie vs prefix table

A trie is conceptually clean; a compressed automaton or key-value prefix table may be easier to shard and deploy.

Personalization boundary

Personalization should rerank a small candidate set. If it changes retrieval for every user, cache locality collapses.

9. What to say in the interview

Strong signal
Lead with the dominant constraint: Trie/FST style serving, prefix aggregation, personalization, updates, latency. Then defend the design by naming what you made faster, what you made more reliable, and what complexity you accepted.
