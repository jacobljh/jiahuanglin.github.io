Design a URL shortener

A shortener is a clean interview case because the write path is simple, the read path is huge, and every extra feature is a trade against redirect latency.

Source: Vol. 1 Ch. 8 Case drill Trade-off first
How to read this case
This page is an original educational walkthrough based on the case topic, not a transcription. Read it linearly: contract, estimates, data, flows, bottlenecks, trade-offs, then failure modes.

0. Interview hinge

The hinge is the moment where the prompt stops being generic and the architecture becomes forced. For this case, that hinge is: Read-heavy routing, ID allocation, redirects, analytics, abuse, cache hit rate.

Invariant to protect
The design is only correct if the user-visible contract survives retries, partial failures, stale caches, and reordering. A concrete invariant to defend here: Cache stampede on viral link must be handled by Use request coalescing and pre-warm known campaigns..
Tempting but wrong
A common shortcut is to choose Sequential vs random code just because it buys short codes and easy uniqueness. That shortcut fails when its cost matters: enumeration and volume leakage. Use it only when the workload matches the condition: internal or low-abuse systems.

1. Clarify the contract

The prompt is best treated as a product contract before it is treated as a component diagram. The system needs to do the following:

  • Create a short code for a long URL.
  • Redirect code lookups with low latency.
  • Optionally support custom aliases and expiration.
  • Collect analytics without slowing redirects.
  • Detect malware, spam, and abusive destinations.

2. Put numbers on the shape

The exact numbers depend on the interviewer, but these are the estimates that should steer the architecture:

  • Reads often exceed writes by 100:1 or more.
  • A 7-character base62 code gives roughly 3.5 trillion combinations.
  • 100M new URLs/year with 200 bytes metadata is tens of GB before analytics.
  • Redirect latency budget is dominated by cache or database lookup plus network round trip.

3. Define the surface area

Keep the API small. The API should express the product operation, not leak the internal queue, cache, or shard layout.

5. Linearized design

Walk the design in the order a request or event actually moves. This keeps the interview grounded and exposes the first bottleneck naturally.

  1. 1. Validate and canonicalize long URLs before generating a code.
  2. 2. Generate a unique code through an ID allocator plus base62 encoding or collision-checked random codes.
  3. 3. Store code metadata in a primary database and populate a cache on first redirect.
  4. 4. On redirect, keep the hot path to cache lookup, abuse check, and 301/302 response.
  5. 5. Emit click analytics asynchronously to a queue or log.
  6. 6. Run abuse scanning out of band and let the redirect path read a compact blocklist state.

6. Deep dives

301 vs 302

Permanent redirects let browsers cache aggressively, which lowers load but makes changes and abuse takedowns harder. Temporary redirects keep control at the service but cost more requests.

Code generation

Sequential IDs are compact and cache-friendly, but reveal volume. Random codes are harder to enumerate but require collision handling and may use longer strings.

Analytics boundary

Analytics cannot sit synchronously in the redirect path. The correct shape is fire-and-forget event logging with later aggregation.

9. What to say in the interview

Strong signal
Lead with the dominant constraint: Read-heavy routing, ID allocation, redirects, analytics, abuse, cache hit rate. Then defend the design by naming what you made faster, what you made more reliable, and what complexity you accepted.
