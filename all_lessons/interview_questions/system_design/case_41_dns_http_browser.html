DNS, HTTP, browser request lifecycle, and QUIC

What happens after typing a URL is the web performance stack in miniature: naming, routing, encryption, protocol, server, and rendering.

0. Interview hinge

The hinge is the moment where the prompt stops being generic and the architecture becomes forced. For this case, that hinge is: Name resolution, connection setup, protocol evolution, page-load bottlenecks.

Invariant to protect
The design is only correct if the user-visible contract survives retries, partial failures, stale caches, and reordering. A concrete invariant to defend here: DNS outage must be handled by Use multiple authoritative providers and tested failover..
Tempting but wrong
A common shortcut is to choose Long DNS TTL vs short TTL just because it buys cache hit and speed. That shortcut fails when its cost matters: slower failover. Use it only when the workload matches the condition: stable endpoints.

1. Clarify the contract

The prompt is best treated as a product contract before it is treated as a component diagram. The system needs to do the following:

  • Explain the request path from URL to rendered page.
  • Understand DNS caching and failure modes.
  • Compare HTTP/1.1, HTTP/2, and HTTP/3 at a systems level.
  • Name the latency contributors before backend code runs.
  • Design around browser and CDN behavior.

2. Put numbers on the shape

The exact numbers depend on the interviewer, but these are the estimates that should steer the architecture:

  • DNS, TCP, TLS, and redirects can add multiple round trips before application work.
  • HTTP/2 multiplexing reduces connection count but still suffers if TCP packet loss blocks the stream.
  • HTTP/3 over QUIC reduces some handshake and head-of-line costs.
  • Cache headers can remove entire requests from the backend path.

5. Linearized design

Walk the design in the order a request or event actually moves. This keeps the interview grounded and exposes the first bottleneck naturally.

  1. 1. Parse URL and resolve domain through browser, OS, resolver, and authoritative DNS caches.
  2. 2. Connect to the chosen IP through TCP+TLS or QUIC.
  3. 3. Send HTTP request with cookies, headers, and cache validators.
  4. 4. Route at CDN, load balancer, and application server.
  5. 5. Return HTML and assets with cache policy.
  6. 6. Browser parses, fetches subresources, executes JavaScript, and paints.

6. Deep dives

Round trips

The fastest backend cannot save a page that burns round trips on redirects, DNS misses, and cold handshakes. Count the network steps first.

Protocol evolution

HTTP/1.1 uses multiple connections to get parallelism. HTTP/2 multiplexes streams over one TCP connection. HTTP/3 moves to QUIC to avoid some TCP-level head-of-line blocking.

Caching

DNS cache, CDN cache, browser cache, and application cache are different layers. Each can improve latency and each can serve stale data if policy is wrong.

9. What to say in the interview

Strong signal
Lead with the dominant constraint: Name resolution, connection setup, protocol evolution, page-load bottlenecks. Then defend the design by naming what you made faster, what you made more reliable, and what complexity you accepted.
