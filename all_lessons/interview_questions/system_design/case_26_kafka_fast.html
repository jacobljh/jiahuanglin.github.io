Why Kafka is fast and when to use it

Kafka is fast because it narrows the problem: append sequential bytes, batch them, let consumers track offsets, and avoid per-message random work.

0. Interview hinge

The hinge is the moment where the prompt stops being generic and the architecture becomes forced. For this case, that hinge is: Append logs, batching, zero-copy, page cache, partition parallelism.

Invariant to protect
The design is only correct if the user-visible contract survives retries, partial failures, stale caches, and reordering. A concrete invariant to defend here: Tiny-message low-throughput config must be handled by Batch and compress before blaming hardware..
Tempting but wrong
A common shortcut is to choose Large batches vs small batches just because it buys throughput and compression. That shortcut fails when its cost matters: latency. Use it only when the workload matches the condition: analytics pipelines.

1. Clarify the contract

The prompt is best treated as a product contract before it is treated as a component diagram. The system needs to do the following:

  • Explain Kafka-like performance from first principles.
  • Know which workloads fit an append log.
  • Understand partitions as throughput and ordering units.
  • Reason about batching and sequential IO.
  • Avoid using Kafka where request/response semantics are needed.

2. Put numbers on the shape

The exact numbers depend on the interviewer, but these are the estimates that should steer the architecture:

  • Sequential disk IO can be orders of magnitude faster than random IO.
  • Batching amortizes network, syscall, and replication overhead.
  • More partitions increase parallelism but also metadata and rebalancing cost.
  • Consumer lag is a queue length measured in offsets and time.

5. Linearized design

Walk the design in the order a request or event actually moves. This keeps the interview grounded and exposes the first bottleneck naturally.

  1. 1. Write messages to append-only partition logs.
  2. 2. Group messages into batches before network and disk writes.
  3. 3. Use OS page cache for hot log segments.
  4. 4. Let consumers fetch sequential ranges by offset.
  5. 5. Replicate partition leaders to followers for durability.
  6. 6. Retain logs by time or size so consumers can replay.

6. Deep dives

Append-only shape

Random updates require indexes and locks. Append logs mostly move the disk head forward and let consumers remember positions independently.

Batching

A single message is expensive; a batch of messages shares protocol overhead, compression, disk flushes, and replication acknowledgments.

Zero-copy intuition

Serving log bytes can avoid copying through user space. The broker becomes a coordinator of byte movement rather than a transformer of every message.

8. Failure modes

FailureMitigation
Tiny-message low-throughput configBatch and compress before blaming hardware.
Too many partitionsRight-size partition count and monitor controller pressure.
Slow consumerScale group or isolate it so retention is not exhausted.
Hot keyChoose partition keys carefully or split the key.

9. What to say in the interview

Strong signal
Lead with the dominant constraint: Append logs, batching, zero-copy, page cache, partition parallelism. Then defend the design by naming what you made faster, what you made more reliable, and what complexity you accepted.
