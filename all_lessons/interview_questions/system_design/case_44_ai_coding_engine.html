Design an AI coding engine

An AI coding engine is where classic system design meets ML systems: context retrieval, model calls, sandboxed execution, user trust, and feedback loops.

0. Interview hinge

The hinge is the moment where the prompt stops being generic and the architecture becomes forced. For this case, that hinge is: Bridge to ML system design: retrieval, sandboxing, tool calls, evaluation, serving.

Invariant to protect
The design is only correct if the user-visible contract survives retries, partial failures, stale caches, and reordering. A concrete invariant to defend here: Wrong file edited must be handled by Show plan, use precise retrieval, and validate diff against task..
Tempting but wrong
A common shortcut is to choose Large context vs retrieved context just because it buys less missing information. That shortcut fails when its cost matters: cost and distraction. Use it only when the workload matches the condition: small repos.

1. Clarify the contract

The prompt is best treated as a product contract before it is treated as a component diagram. The system needs to do the following:

  • Accept a coding task and repository context.
  • Retrieve relevant files and symbols.
  • Generate patches or explanations.
  • Run tests or static checks safely.
  • Track quality, latency, cost, and user feedback.

2. Put numbers on the shape

The exact numbers depend on the interviewer, but these are the estimates that should steer the architecture:

  • Repository context can exceed model context windows, so retrieval is mandatory.
  • Model latency and token cost dominate many requests.
  • Sandbox execution must assume untrusted code.
  • Evaluation needs task-level outcomes, not only thumbs-up feedback.

5. Linearized design

Walk the design in the order a request or event actually moves. This keeps the interview grounded and exposes the first bottleneck naturally.

  1. 1. Snapshot repository state and permissions for the task.
  2. 2. Retrieve candidate files using lexical search, embeddings, symbols, and recent edits.
  3. 3. Build a bounded prompt context with task, constraints, and selected code.
  4. 4. Call the model to plan, edit, or explain.
  5. 5. Run tests and commands in a sandbox with resource limits.
  6. 6. Store traces, diffs, outcomes, and feedback for evaluation and improvement.

6. Deep dives

Context selection

The engine fails when it gives the model the wrong code. Retrieval should combine exact search, dependency graph hints, and conversational state.

Sandboxing

Generated code and test commands are untrusted. The system needs file, network, secret, and resource boundaries.

Evaluation

A nice-looking patch is not enough. The system should measure compile/test pass, diff quality, review comments, and eventual user acceptance.

9. What to say in the interview

Strong signal
Lead with the dominant constraint: Bridge to ML system design: retrieval, sandboxing, tool calls, evaluation, serving. Then defend the design by naming what you made faster, what you made more reliable, and what complexity you accepted.
