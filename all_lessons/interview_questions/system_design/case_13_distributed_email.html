Design Gmail or a distributed email service

Email is old, but it is a wonderful systems prompt: open protocol ingress, durable delivery, user-facing search, abuse, and enormous attachment storage.

Source: Vol. 2 Ch. 8, archive Case drill Trade-off first
How to read this case

0. Interview hinge

The hinge is the moment where the prompt stops being generic and the architecture becomes forced. For this case, that hinge is: SMTP flow, inbox indexing, spam pipeline, delivery retry, search and attachments.

Invariant to protect
The design is only correct if the user-visible contract survives retries, partial failures, stale caches, and reordering. A concrete invariant to defend here: Remote SMTP retry storm must be handled by Apply backpressure and greylisting; keep accepted mail durable..
Tempting but wrong
A common shortcut is to choose Store raw MIME vs parsed body just because it buys audit and reprocessing. That shortcut fails when its cost matters: more storage. Use it only when the workload matches the condition: protocol correctness.

1. Clarify the contract

The prompt is best treated as a product contract before it is treated as a component diagram. The system needs to do the following:

  • Receive and send email through standard protocols.
  • Store messages durably by mailbox.
  • Index messages for search.
  • Filter spam and malware.
  • Support attachments without bloating mailbox records.

2. Put numbers on the shape

The exact numbers depend on the interviewer, but these are the estimates that should steer the architecture:

  • Attachments dominate storage; message metadata dominates search/index traffic.
  • Inbound spikes can come from remote systems retrying after an outage.
  • Spam filtering may be CPU-heavy and should be isolated from accepted mail durability.
  • Mailbox search wants inverted indexes, not scans over raw messages.

5. Linearized design

Walk the design in the order a request or event actually moves. This keeps the interview grounded and exposes the first bottleneck naturally.

  1. 1. Accept inbound mail at edge SMTP servers and write an immutable raw message record.
  2. 2. Queue messages through spam, malware, and policy processors.
  3. 3. Store clean message bodies and metadata in mailbox storage.
  4. 4. Put large attachments in object storage and reference them from the message.
  5. 5. Update per-user indexes asynchronously.
  6. 6. For outbound mail, queue delivery attempts and retry according to remote server responses.

6. Deep dives

Accepted means durable

Once an SMTP server accepts a message, losing it is a serious correctness failure. The edge should durably enqueue before returning success.

Search index lag

Users tolerate slight search lag more than lost mail. Keep mailbox reads canonical and let search catch up asynchronously.

Abuse boundary

Spam filtering is adversarial and expensive. Isolate it so malicious mail does not starve normal mailbox operations.

9. What to say in the interview

Strong signal
Lead with the dominant constraint: SMTP flow, inbox indexing, spam pipeline, delivery retry, search and attachments. Then defend the design by naming what you made faster, what you made more reliable, and what complexity you accepted.
