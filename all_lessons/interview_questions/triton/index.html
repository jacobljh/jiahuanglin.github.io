Triton Kernels — interview prep

A linearized interview track for OpenAI Triton: start with the tile programming vocabulary, then write optimized kernels, then map algorithms and data structures onto program IDs, masks, reductions, and memory layouts.

Linearized thinking
Triton hides individual CUDA threads, but it does not hide the GPU. Think in tiles: one program owns a rectangular block of output, builds pointer tensors for the bytes it needs, masks invalid lanes, reduces inside registers, and stores the final tile.

The three-pass structure

PassQuestionOutcome
01 What is a Triton kernel mechanically? You can write @triton.jit, tl.program_id, tl.arange, pointer blocks, masks, launch grids, and wrapper functions.
02 What makes a Triton kernel fast? You use vectorized blocks, fusion, tl.dot, online reductions, autotune configs, and memory-aware tile choices.
03 How do algorithms look in Triton? You express reductions, softmax, RMSNorm, top-k, block-sparse layouts, and segmented work as tile ownership plans.

Triton versus CUDA in one sentence

CUDA makes you assign work to threads; Triton makes you assign work to tiles. In both cases, performance still comes from the same physics: coalesced HBM traffic, enough parallelism, data reuse, low synchronization, and math units fed with the right shapes.
