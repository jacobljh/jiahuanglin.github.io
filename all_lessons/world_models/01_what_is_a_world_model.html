What is a world model?

Build the idea from the agent–environment loop: infer what is hidden, predict how actions change it, expose uncertainty, and answer a decision-relevant question.

Linear route through this lesson
We will not start from a particular neural architecture. We will start from the minimum information an agent needs, derive the mathematical contract, trace one numerical control example, distinguish world modeling from graphics and passive prediction, and finish with an engineering test you can apply to any proposed system.

0 · Dependency recap: prediction serves an agent

This first lesson has no model prerequisite, but it does assume a useful distinction. An environment changes whether or not an agent understands it. An agent receives measurements, chooses interventions, and experiences consequences. The purpose of a world model is not merely to describe the environment. It is to give the agent a compact internal mechanism for answering: “If I do this, what is likely to happen next, and what does that imply for my objective?”

That phrasing already tells us what has to be present. There must be something that persists through time, some way to observe it, some intervention whose consequences differ from those of another intervention, and some query whose answer matters. Remove persistence and there is no world to track. Remove actions and the model may be a forecaster but cannot support counterfactual control. Remove the query and “good model” has no operational meaning.

Course operational definition
In this course, a world model is an internal, query-relative mechanism that (1) infers a belief from observation and action history and (2) predicts a distribution over decision-relevant future state or outcomes under candidate actions. This is a deliberately testable course boundary, not a claim that every community uses the term identically. A particular system earns the name only for the queries, horizons, actions, and deployment regimes on which those two operations are validated.

1 · Begin with the smallest agent–environment loop

At time t, imagine the environment has a physical condition st. The agent does not ordinarily get to read it. It receives an observation ot, chooses an action at, and the world moves to st+1. We can express the two outside processes as distributions:

st+1 ∼ T(· | st, at)     and     ot ∼ O(· | st)
  • State st is the information about the environment that makes the next-state distribution independent of older history. Position alone is not state for a moving ball; position plus velocity may be.
  • Transition T says how a current state and chosen action distribute probability over next states. We write a distribution rather than a single answer because contact, other agents, sensor timing, and unmodeled variables can make the future uncertain.
  • Observation model O says how a state produces a measurement. A camera projects 3D geometry through lighting, material, lens, and sampling. A force sensor produces a different, noisy projection of the same world.
  • Action at is an intervention, not just another token. Its value is that holding current evidence fixed while changing the action should change the predicted future in the appropriate way.

“State” does not mean every atom in the universe. It means sufficient state for the horizon and questions under consideration. A mobile robot planning for three seconds may use poses, velocities, traversability, and uncertainty. A cloth simulator may require local deformation and stress. The state definition is therefore a modeling choice, but it cannot omit a variable that changes the queried future.

Why the world is not its pixels

A pixel array is an observation, not usually a state. A ball can occupy exactly the same image location while moving left in one history and right in another. The next image differs even though the current image is identical. Conversely, rotating the camera can alter almost every pixel while the room and its objects remain stationary. Pixels mix scene properties with viewpoint, lighting, occlusion, and sensor noise. A useful internal state must separate persistent causes from their transient appearance when the task requires that separation.

2 · Derive the internal contract

The true state and true laws are inaccessible. The agent therefore constructs a learned internal state xt from what it has actually received: observations through time o≤t and past actions a<t. This state is a compact implementation of its belief about the world. A world model needs at least an inference operation and a predictive operation:

xt = E(o≤t, a<t)     then     x̂t+1 ∼ F(xt, at)

E, the encoder or state estimator, compresses history into the agent’s present internal state. F, the learned dynamics, advances that internal state under an action. A prediction head G can then answer a query κ—future observation, collision probability, reward, occupancy, termination, or a value estimate:

x̂t = xt,    x̂t+i+1 ∼ F(x̂t+i, at+i),    ŷt+k = G(x̂t+k, κ)

The hat marks imagined rather than newly observed state. The index k is the prediction horizon. The action sequence is supplied by a planner or policy. Notice what is not required: a pixel decoder. If the query is “will the gripper collide?”, decoding a cinematic frame may add cost without improving the answer.

The key property is predictive sufficiency. If two histories produce the same xt, then they should produce the same distribution over all future quantities and action sequences that the system promises to answer. A model may merge red and blue boxes for geometry-only navigation. It must not merge a stationary box with a fast oncoming box if collision avoidance is promised.

3 · Running example: braking for a hidden cart

Consider a warehouse robot driving along a one-dimensional aisle. A cart is currently 12 m ahead and moving toward the crossing point at 3 m/s. The robot moves at 4 m/s. Its controller can either coast or apply a constant braking acceleration of −2 m/s² for one second. To keep the arithmetic transparent, let state be st = (dt, vr,t, vc,t): separation, robot velocity, and cart velocity toward the robot’s path.

Use a one-second step. Under the simplified dynamics, the next separation is current separation minus how far both parties close during the second. If the robot coasts, it travels 4 m and the cart travels 3 m, so:

dt+1 = 12 − 4 − 3 = 5 m

One more coast step would close another 7 m, putting the trajectories through one another in the model: a collision before two seconds. Under braking, the robot’s speed falls from 4 to 2 m/s. Using average speed during constant acceleration, it travels (4 + 2) / 2 × 1 = 3 m. The cart still travels 3 m:

dt+1 = 12 − 3 − 3 = 6 m;    vr,t+1 = 2 m/s

If braking continues for the second step, robot speed reaches 0 and it travels an average of 1 m; the cart travels 3 m, leaving 2 m separation. The same present camera crop could show the cart at 12 m whether it was stationary or approaching at 3 m/s. A single image does not contain the needed velocity. History does. The model must therefore infer a state from successive observations, retain the cart while it is briefly occluded by shelving, and roll two different action-conditioned futures.

Now include uncertainty. Suppose perception assigns probability 0.8 that the cart approaches at 3 m/s and 0.2 that it is stationary. Under coast, the simplified two-second collision probability is 0.8; under immediate braking it is near 0 for both hypotheses. If collision costs 100 units and braking costs 3 units, expected coast cost is roughly 0.8 × 100 = 80, while brake cost is 3. A point estimate that averages cart velocity to 2.4 m/s may look precise, yet it erases the two hypotheses and can distort tail risk. Lesson 02 will make that belief explicit.

This small example exposes the complete contract: observations estimate hidden velocity; the latent persists through occlusion; actions enter the transition; rollouts answer a collision query; uncertainty affects the choice. A system that only paints a plausible next aisle image has not necessarily solved any of these steps.

4 · Place graphics, vision, simulation, and generation correctly

The components are easier to understand by direction of inference. The Computer Graphics track primarily studies the forward observation map: begin with geometry, materials, lights, and a camera, then synthesize pixels. The Computer Vision track primarily studies inverse problems: begin with pixels and estimate labels, geometry, motion, or other causes. The 3D Vision track makes camera and geometric structure especially explicit.

A world model can use all three, but adds temporal, interventional dynamics. Instead of memorizing a taxonomy, walk once around the agent–environment loop and ask where each system enters:

  1. Begin with a known scene and camera. A renderer moves forward from that supplied scene to an observation. This solves the appearance question, “What would this state look like?”, but it need not infer state from measurements or say how the state changes.
  2. Now remove the supplied scene and keep only images. Static 3D reconstruction runs the difficult direction backward, estimating geometry and appearance from those images. It may recover a persistent room, yet still say nothing about how a push, collision, or control input changes that room.
  3. Add time but withhold explicit interventions. A passive video predictor uses past frames to produce plausible future frames. This can capture motion statistics, but plausibility under the observed behavior does not establish reliable counterfactuals under a different action.
  4. Instead, supply exact state and action. A physics simulator advances that explicit state to the next explicit state. This provides dynamics, but it usually assumes someone else has already converted raw camera or force measurements into the simulator’s variables.
  5. Finally, close the missing links. A world model starts from observation and action history, infers the present belief, and predicts action-conditioned state or outcomes. Even then, the name is only a contract: whether its data coverage and predictive quality are sufficient remains an empirical question.

This sequence reveals why combining components is legitimate but renaming one component is not. A simulator can supply the dynamics inside a world model. A renderer can serve as its observation decoder. A video generator can provide an observation-level rollout. The naive leap is to treat success at one arrow as success around the entire loop. Photorealism establishes observation synthesis; it does not by itself establish object permanence, causal action effects, or calibrated collision probability.

Linear diagnostic
Write the candidate system as arrows between history, belief state, future state, and future observation. Mark which arrows receive actions. Any missing arrow identifies the unproven capability more precisely than the system’s name does.

5 · Four tests for whether the name is earned

The definition becomes useful only when it can reject a convincing failure. Apply the following tests in order, because each test depends on the capability before it:

  1. First test persistence. Hide an object behind an occluder or move the camera until the object leaves view. Ask whether the internal prediction retains its existence and relevant attributes. If it vanishes merely because its pixels vanish, the system is using the naive rule “not visible means not present.” There is no stable state to advance yet.
  2. Then test dynamics. Once hidden entities persist, stop supplying observations and roll the state or queried outcomes forward. Measure how error and uncertainty grow with horizon. Failure here exposes a static scene representation or a one-step shortcut: the model remembers something, but cannot evolve it reliably.
  3. Then test intervention. Hold inferred context fixed, substitute different feasible actions, and compare predictions with the outcomes of those actual interventions. If the futures barely change, or change in the wrong way, the predictor has learned ordinary temporal correlation while ignoring control. A controller cannot treat it as causal.
  4. Finally test decision value. Give the model to a planner or policy and measure whether it improves task success, safety, or sample efficiency. This catches the subtle failure in which images and average errors look good while rare optimistic mistakes sit exactly where the planner searches.

The ordering is causal rather than presentational. Without persistence, there is no hidden entity to roll forward. Without dynamics, there are no trajectories to compare. Without intervention, those trajectories do not answer “what if I act differently?” And without decision value, none of the preceding accuracy claims establishes that the representation is useful where errors actually carry cost.

6 · Choose the world relative to a query

No finite model is complete. The engineering problem is to identify a query distribution: which questions, over which states, action sequences, and horizons, must the model answer? A warehouse controller might query one-to-five-second occupancy and collision risk. A manipulation policy might query contact, support, and success. A video editor might query appearance under camera motion and semantic edits. Each requires a different abstraction.

This gives a disciplined state-design procedure:

  1. Write the decisions. List the actions the consumer can actually choose.
  2. Write the outcomes. Define what must be predicted, including rare safety events and uncertainty.
  3. Set the horizon and resolution. Centimeter precision for 200 ms contact is different from lane-level prediction for 10 s driving.
  4. Find distinctions that change answers. If two histories require different outcome distributions under some candidate action, the representation must not merge them.
  5. Discard distinctions that never change answers. This is where compression becomes useful rather than arbitrary.

This is why “the model understands the world” is too vague for design or evaluation. Replace it with a contract: “Given the last two seconds of camera, proprioception, and actions, predict a calibrated distribution over object occupancy and contact for feasible controls over the next three seconds.” That statement makes data coverage, architecture, loss, and metrics discussable.

7 · Explicit, learned, and hybrid state

Explicit
Named variables and known rules
Interpretable, editable, and data-efficient when the assumptions are correct; brittle when omitted physics dominates.
Learned
Distributed state and fitted transitions
Can absorb hard-to-write regularities; may use shortcuts and behave unpredictably outside training support.
Hybrid
Known structure plus learned residuals
Useful when geometry, kinematics, or conservation laws are trusted but contacts, appearance, or agent intent are not.

The choice is not “symbolic versus neural.” Ask which variables are observable, which laws are reliable, which residual effects are data-rich, and where uncertainty must be exposed. An autonomous system may combine an explicit ego pose and map, a learned object detector, a probabilistic motion model, and a neural occupancy predictor. That is one world model assembled across representations.

Other design axes are equally important: deterministic versus stochastic transitions; object-centric versus dense spatial state; metric 3D coordinates versus ungrounded features; open-loop rollout versus frequent observation correction; general predictive state versus task-specific value-equivalent state. There is no universally best point. The contract and failure costs decide.

8 · How a planner consumes the model

A world model becomes operational when embedded in a feedback loop. At the current time, the estimator incorporates the newest observation and produces learned belief state xt. A planner proposes several feasible action sequences. The dynamics model rolls each sequence forward, and prediction heads estimate outcomes and uncertainty. An objective scores those outcomes. The controller executes only a short prefix, observes the real world again, corrects its belief, and repeats.

choose at:t+H−1 to maximize E[∑k=0H−1 reward(ẑt+k, at+k)] subject to risk constraints

H is the planning horizon. The sum accumulates predicted reward over that horizon, and the expectation averages uncertain futures. A risk constraint might require predicted collision probability below a threshold. The exact optimizer may sample candidate sequences, follow gradients, expand a search tree, or use a learned policy. Those details change later; the information contract does not.

Why execute only a prefix rather than the full imagined plan? New observations correct errors. In the braking example, the model may slightly misestimate cart speed. Replanning at 10 Hz prevents a two-second open-loop error from accumulating unchecked. This is receding-horizon control: imagine far enough to anticipate consequences, act briefly, then ground the model again in evidence.

The loop also reveals two distinct performance limits. Model error means predicted outcomes differ from the environment. Planning error means the optimizer fails to find a good action even under an accurate model. Compare against an oracle planner using recorded or simulated true dynamics to separate them. A stronger optimizer can actually make a weak model perform worse because it searches more effectively for unrealistic, high-scoring predictions.

Compute creates a concrete tradeoff. Rich pixel generation may be valuable for human inspection but too slow for evaluating thousands of action sequences. A practical system can use a compact latent transition for broad search, a grounded outcome head for scoring, and a renderer only for a few finalist trajectories. Architecture should follow how frequently each prediction will be queried.

9 · Data support defines the trustworthy action region

A learned transition is reliable only near combinations of state and action represented or structurally constrained during training. Passive driving data, for example, contain mostly competent actions. A planner may consider sharp steering or unusual braking that the data rarely show. The model’s confident extrapolation there is not evidence.

Track this boundary explicitly. Measure ensemble disagreement or density as rough warning signals; restrict actions to trusted envelopes; gather targeted intervention data; incorporate known kinematics and constraints; and fall back to conservative control when uncertainty is high. Dataset coverage should be reported jointly over state, action, and horizon. Broad visual diversity without action diversity does not identify action effects.

10 · Common misconceptions and failure cases

  • “Next-frame prediction is sufficient.” It may exploit local optical flow, ignore actions, average multiple futures, and fail beyond one step. Test intervention and long-horizon outcomes.
  • “Photorealism proves physical understanding.” Textures can be excellent while contact timing, count, permanence, and causal response are wrong. Evaluate state and outcome variables directly.
  • “A larger latent is a more complete world.” Capacity may store nuisance appearance rather than velocity or uncertainty. Completeness is relative to queries, not vector width.
  • “One predicted future is enough.” Other agents and unobserved variables create branching futures. A point rollout hides probability mass that a risk-aware planner needs.
  • “Low test loss guarantees safe planning.” A planner actively searches for model errors that yield high predicted return. Average accuracy does not protect against exploitation in unusual action sequences.
  • “More history automatically fixes partial observation.” History helps only if the estimator retains relevant evidence and the data identify the hidden variable. No camera history reveals the mass of a motionless sealed box without an informative interaction.

11 · Practical system and interview reasoning

When asked to design a world model, resist naming a fashionable architecture first. Walk linearly from requirements to representation:

  1. Specify interface: sensors, action space, prediction targets, horizon, latency, and safety constraints.
  2. Identify hidden causes: velocity, occluded objects, friction, intent, camera pose, and any variable that aliases the same current observation.
  3. Choose state structure: dense grid, scene graph, object slots, sequence tokens, metric geometry, or a hybrid.
  4. Choose uncertainty: Gaussian parameters, discrete modes, ensembles, particles, or sampled stochastic latents, matching expected ambiguity.
  5. Choose learning signals: future observations, features, rewards, occupancy, inverse/action prediction, and consistency constraints.
  6. Close the loop: compare planned action sequences, execute a short prefix, observe, update state, and replan.
  7. Evaluate by slices: occlusion, camera motion, rare interaction, novel actions, increasing horizon, calibration, and downstream regret.

A strong answer also names the boundary of the model. If training contains only passive trajectories, action effects can be confounded: the model sees that brakes correlate with stopping but may not separate braking from the hazardous context that caused the driver to brake. Interventional coverage, exploration, simulation, or structural assumptions may be needed. Later lessons return to this causal issue.

12 · Explicit takeaway

Takeaway
A world model is an internal, query-relative state-and-transition mechanism. It summarizes observation/action history into a belief, predicts how candidate actions change that belief or its outcomes, represents relevant uncertainty, and earns its value through better prediction or decisions. Rendering pixels is optional; persistence, action-conditioned dynamics, and empirical decision value are not.

The next dependency is unavoidable: the agent does not know the true current state. One observation often corresponds to several possible worlds. Lesson 02 turns that ambiguity into a belief distribution and derives the predict–correct update.

Interview prompts

  • What is the minimal operational definition of a world model? A mechanism that infers internal state from history and predicts decision-relevant futures under candidate actions.
  • Why is a current image usually not Markov state? It omits hidden causes such as velocity, occluded objects, and physical properties, so older evidence still changes the future distribution.
  • Why is a renderer not automatically a world model? It implements state-to-observation synthesis but need not infer state or model action-conditioned transitions.
  • Must a world model reconstruct pixels? No. It may predict latent state, occupancy, reward, collision, termination, or value if those outputs are sufficient for its query contract.
  • How would you test action conditioning? Hold the same inferred context fixed, vary feasible actions, and compare the predicted differences with interventional outcomes.
  • What is predictive sufficiency? Histories merged into one latent must have the same relevant future distributions under every action sequence the model promises to evaluate.
  • Why can low average rollout error still harm a planner? The planner searches action space and may exploit a rare optimistic model error that average metrics barely weight.
  • When is a hybrid model preferable? When some structure such as camera geometry or kinematics is reliable while residual contacts, appearance, or behavior are better learned from data.
