Evaluation and failure modes

The final test is not whether a frame looks real. It is whether state stays coherent, uncertainty is honest, interventions have the right consequences, and decisions made through the model work safely in the world.

The capstone
We build an evaluation ladder, quantify causal and probabilistic quality, use planning to red-team the model, and complete an end-to-end design for Countertop Assistant. The result is a reusable framework for evaluating or designing any world-model product.

1 · Derive metrics from the decision and loss

From first principles, a world model is useful because an agent will choose differently after consulting it. Evaluation should therefore start with the deployment decision, not a benchmark that happens to be convenient. Define the action space, observation history, horizon, deadline, outcomes, and asymmetric cost of mistakes. Then work backward to the intermediate capabilities required for that decision.

Our running product is Countertop Assistant: place the instructed red ceramic mug on a serving tray within 20 seconds. A successful trial requires the correct object, upright placement, no spill, no human/scene collision, and grip force below the material limit. The product loss might assign 1 to timeout, 3 to wrong object, 10 to spill or breakage, and 100 to human contact. Average pixel error does not contain those asymmetries. Even average task success can hide a rare but unacceptable safety tail.

Write the operational objective explicitly. One simple risk-sensitive form is:

choose action sequence a* = argmaxₐ E[task utility | belief, a] subject to P(human contact | belief, a) < 10⁻⁵ per plan P(breakage | belief, a) < 10⁻⁴ per plan planning latency P99 < 55 ms

These thresholds immediately imply evaluation needs: calibrated rare-event estimates or conservative bounds, correct response to action intervention, sufficient state for humans/contact/material, and tail latency. A photorealistic video metric may remain useful for debugging appearance, but it cannot be the release criterion.

2 · Climb an evaluation ladder instead of compressing to one score

Evaluation should isolate each arrow in the loop before testing the loop as a whole. Lower rungs are cheap and diagnostic; upper rungs are realistic but confounded and expensive. Passing a lower rung is necessary, not sufficient.

Imagine the first failed countertop trial: the robot misses the mug. “The model is bad” is not yet an explanation. We need to move upward only after the layer below is known to work, because each rung turns a vague product failure into one testable question:

  1. Data: did the recorded cause actually precede the effect? Before judging a learned model, verify that observations and actions are physically aligned. Measure clock residual, reprojection error, missingness, and action-effect lag. A failure here localizes dataset corruption or leakage. Continuing upward would only test how well a model fits a broken record.
  2. Observation: did relevant evidence enter the representation? Once timing is credible, ask whether the model can encode and decode what the sensors truly saw. Use perceptual distance, depth and flow, contact/event timing, and ASR grounding. Failure now points to sensor representation, not yet memory or planning.
  3. State: did momentary evidence become a persistent belief? Test whether hidden variables remain recoverable with linear and nonlinear probes, tracking, occupancy, pose, contact mode, and conservation checks. If this rung fails, the belief is insufficient: later dynamics cannot predict a variable the state discarded.
  4. One-step dynamics: did one executed action cause the right local change? Measure state-transition error, mode likelihood, and constraint violation. Failure after the state rung passes usually points to local dynamics or action alignment. This is the smallest action-conditioned unit from which a rollout is built.
  5. Free rollout: does that local rule remain coherent when the model consumes its own predictions? Plot an error-versus-time curve, test survival of identity and constraints, and measure event-time error. A gap between one-step and free-rollout quality localizes compounding drift rather than perception.
  6. Uncertainty: does the model know when several futures remain possible? Ask whether probabilities match empirical frequencies and widen under ambiguity. Inspect NLL, Brier score, ECE, interval coverage, sharpness, and risk–coverage. Failure means either dangerous overconfidence or useless diffusion so broad that it cannot guide a decision.
  7. Intervention: does a changed action change the correct outcome? Use matched counterfactual error, treatment-effect error, and controllability. A model that predicts recorded futures but fails here has learned correlation instead of causal dynamics.
  8. Planning: does optimization through the model rank actions as reality would? Measure regret, top-k action recall, value error after selection, and exploit rate. If intervention tests pass but planning fails, the likely defect is planner–model mismatch or error amplified by selection.
  9. Closed loop: does the assembled agent improve physical outcomes? Measure success, expected cost, collision, recovery, and sample efficiency with sensors, action budget, and safety controller held comparable. This rung reveals integrated behavior, where individually adequate components may still interact badly.
  10. System: does the same loop survive production load and faults? Finally measure P50/P95/P99 latency, throughput, state corruption, fallback, and availability. Failure here defines an operational-envelope problem: a correct plan that arrives late or uses stale state is physically incorrect.

This order is deliberately asymmetric. A high rung can fail because of anything below it, but a low rung does not become valid merely because the final task happened to succeed. For example, one lucky grasp cannot excuse a reversed action timestamp; the corrupted record will fail when speed or delay changes. The implementation consequence is to attach every release gate to the earliest rung that can explain it, and the diagnostic consequence is to stop climbing as soon as a prerequisite breaks.

A release dashboard should show this vector by horizon and scenario slice. One scalar can be useful for ordering experiments only after safety gates are enforced. Aggregate scores otherwise permit compensation: excellent texture should never cancel incorrect braking, and common success should never cancel a catastrophic rare event.

3 · State probes test sufficiency, not visual beauty

A belief state should make task-relevant future variables predictable without needing the full past. Freeze the world model and train small probes for object identity/pose, occluded occupancy, joint state, contact mode, mass/friction bin, human motion, and goal predicates. A linear probe tests whether information is easily accessible; a powerful nonlinear probe tests whether it exists at all. Compare against probes on the latest frame and a simple history baseline.

Probe under controlled ambiguity. Hide the mug for two seconds and check identity and pose uncertainty. Show visually identical dry and wet grasps, then add tactile evidence. Swap instruction wording while preserving the goal. Move the camera while the world remains static. A model may score well on randomly sampled states because labels correlate with backgrounds; counterfactual scene edits and held-out kitchens expose that shortcut.

Measure invariants and equivariances. Object identity should persist across viewpoint. Geometry should transform predictably when the coordinate frame changes. Total rigid-object size should not drift. Free space and occupancy should not overlap. Contact forces should satisfy approximate direction and magnitude constraints. These tests are often more informative than a perceptual average because they specify what a coherent world means.

For Countertop Assistant, do not only probe “is mug present?” Probe the variables the planner queries: mug pose covariance, tray support polygon, probability of stable contact, predicted time-to-contact, human occupancy over the next second, and uncertainty when an observation is absent. If those are not recoverable, no planner hyperparameter can repair the representation.

4 · Rollout evaluation is a curve over horizon and regime

One-step evaluation conditions on the true current state at every step; deployment planning conditions on the model’s own predicted states. Small bias compounds. Evaluate open-loop or free rollout: initialize from real history once, apply the recorded executed action sequence, and compare model distributions to the subsequent physical trajectory at every horizon.

Report event-aware curves, not only average state MSE. Position error may remain small while contact begins 150 ms too early, switching the subsequent dynamics. Use object pose/velocity error, identity survival, contact-mode accuracy, collision false negatives, event-time error, reward/cost error, and physical-constraint violations. Bucket by action speed, contact, occlusion, material, lighting, robot revision, and whether the behavior policy is in support.

Suppose lateral pose error has 4 mm one-step root mean square error. An independent-noise story predicts roughly 4√20 ≈ 17.9 mm after 20 steps, while a consistent 1 mm directional bias per step can create 20 mm plus noise. If measured 20-step error is 58 mm, the model is entering off-manifold states or switching contact modes incorrectly; “one-step error × horizon” underestimates the problem. Plot posterior rollouts, which receive new observations, separately from prior imagination, which does not.

Multimodal futures require distributional metrics. A human near the robot may turn left or right; averaging produces a person who walks through the middle. Evaluate likelihood or proper scores of the realized future, coverage of prediction sets, diversity without duplicates, and whether distinct samples correspond to coherent modes. Best-of-N error alone rewards spraying many samples and hides probability assignment.

5 · Test causal action effects with matched interventions

Passive prediction can succeed by learning what usually follows, while ignoring action. In demonstrations, a gripper closes near a mug and the mug then rises; background and operator intent predict much of the sequence. A causal world model must answer what changes when the action changes while prior state remains comparable.

For action a and outcome y, define an effect between two interventions:

Δtrue(s; a₁,a₀) = E[y | do(a₁), s] − E[y | do(a₀), s] Δmodel(z; a₁,a₀) = Ê[y | z,a₁] − Ê[y | z,a₀] causal effect error = |Δmodel − Δtrue|

The do notation means action is set, not merely observed. Exact counterfactual pairs are impossible in one physical episode, so use resettable simulation, controlled fixtures, randomized safe interventions, or carefully matched initial states. Evaluate direction, magnitude, threshold, and interaction: more grip force should reduce slip up to a point, but may increase breakage.

Worked intervention example

Across 200 matched low-friction trials, a 6 N-per-finger grasp succeeds in 78/100 and a 10 N grasp succeeds in 92/100. The observed treatment effect is 0.92 − 0.78 = 0.14, a 14-point increase. The model predicts success probabilities 0.70 and 0.96, hence an effect of 0.26. Its effect error is 0.12: it understands the direction but exaggerates benefit, inviting the planner to over-squeeze.

Now evaluate a 14 N intervention. Physical success drops to 0.88 because brittle mugs crack, but the model predicts 0.99. The model learned a monotonic shortcut “more force is safer” from data that rarely exceeded 10 N. This failure may be invisible in held-out passive trajectories and obvious under an intervention curve. The release suite should include dose–response plots for speed, force, clearance, and information-gathering actions.

6 · Calibration asks whether confidence deserves trust

A probability forecast is calibrated if events predicted at probability p occur about fraction p among comparable cases. Proper scoring rules reward both calibration and useful sharpness. For a binary event y∈{0,1} with predicted probability p, Brier score is (p−y)²; negative log-likelihood is −[y log p +(1−y)log(1−p)]. NLL heavily penalizes confident mistakes, appropriate for safety but sensitive to label noise.

Expected calibration error bins predictions and averages the gap between mean confidence and empirical frequency. It is readable but depends on binning and can hide subgroup error. Always show reliability diagrams and calibrate by horizon, action selection, and safety slice. A model calibrated on random recorded actions can become overconfident after a planner deliberately selects its highest predicted values.

Worked calibration example

Suppose 100 planned grasps each receive predicted slip probability 0.10, but 24 actually slip. Mean Brier score for this group is [24(0.10−1)² + 76(0.10−0)²]/100 = (19.44+0.76)/100 = 0.202. The calibration gap is |0.10−0.24|=0.14. If the same cases were predicted at 0.24, Brier would be [24(0.76²)+76(0.24²)]/100 ≈ 0.1824. Calibration improves the score, but the forecast remains uncertain; calibration is not accuracy.

For continuous or multimodal futures, evaluate interval/set coverage and sharpness together. A nominal 90% pose region should contain truth near 90% of the time, while remaining as small as possible. A region covering the entire workspace is perfectly covering and useless. Plot coverage versus horizon and OOD distance. Decompose aleatoric uncertainty—irreducible futures—from epistemic uncertainty due to limited knowledge when possible; the latter should often trigger information gathering or abstention.

7 · Risk–coverage turns uncertainty into behavior

An OOD or uncertainty number matters only if it changes policy. Sort cases by uncertainty, abstain or fall back on the highest fraction, and plot error or safety cost on the retained cases. A useful uncertainty measure produces declining risk as coverage falls. If removing the “most uncertain” 20% barely changes error, the score is not ranking danger.

Countertop Assistant can respond by slowing, shortening rollout horizon, choosing a larger-clearance path, making a test contact, requesting a new viewpoint, switching to a verified controller, asking the user, or stopping. Evaluate the entire selective policy, including the cost and safety of fallback. A fallback that is invoked on 60% of normal cases may be safe but defeats the product; a fallback invoked only after collision is not protective.

Use threshold policies chosen on validation and frozen for test. Report coverage, conditional success, expected task cost, catastrophic-event upper confidence bound, fallback frequency, and time-to-recovery. Bucket false confidence: cases where uncertainty is low but error is high are more urgent than merely uncertain cases.

8 · Closed-loop evaluation measures decision quality

Ultimately deploy the model inside the intended policy or planner. Compare against a reactive policy, a no-model controller, an oracle-state simulator bound, and ablations. Keep action budget, sensors, latency, and safety controller comparable. Run enough randomized seeds and scenario repetitions to estimate tails; report confidence intervals rather than a single success percentage.

Metrics for Countertop Assistant include task success, time, wrong-object rate, spill/breakage, minimum human clearance, peak force, recovery success, number of replans, information-gathering actions, energy, deadline misses, and fallback. Separate model failure (wrong consequence), planner failure (correct model but poor search/objective), controller failure (plan not executed), perception/alignment failure, and requirement failure (metric omitted what users care about).

Run paired trials from the same initial conditions where possible. If world-model MPC improves success from 82% to 90% but raises P99 latency so 4% of actions are stale, the integrated system may be worse. Measure learning curves versus real interaction count to quantify sample efficiency. For adaptive systems, evaluate after data collection without leaking evaluation scenarios into training.

Closed-loop success alone is still insufficient: a policy can exploit simulator defects or avoid difficult cases. Combine task utility with the earlier mechanistic tests, real-world canaries, and scenario coverage. The evaluation ladder prevents a top-line number from erasing a known causal flaw.

9 · Planning is an automated red team for model exploitation

A planner optimizes whatever the model predicts, including its errors. As optimization strength increases, selected actions move away from typical data and toward regions where small optimistic bias is largest. This is the model-based analogue of adversarial examples and Goodhart’s law.

Use the planner as a test generator. Search for trajectories with high predicted value and one or more warning signs: ensemble disagreement, low density under the behavior data, physical-constraint violation, extreme action, reward–observation inconsistency, uncertainty that paradoxically shrinks, or disagreement between structured and pixel heads. Validate these candidates in a trusted simulator, hardware fixture, or bounded real experiment.

In our product, a planner may discover that moving the wrist through the edge of an occlusion makes the model “forget” the mug while retaining task reward, or that a 16 N grasp is predicted perfectly safe because training support ended at 10 N. Record post-selection error: prediction error on planner-chosen trajectories, not only random test actions. Increase planner budget during evaluation; a model that appears safe with 16 candidates may be exploited with 1,024.

Mitigations include pessimistic ensemble values, uncertainty penalties, support constraints, conservative terminal values, short model-predictive horizons with frequent real correction, physics/safety constraints outside the learned model, adversarial action collection, and planner-aware training. None replaces external validation. If safety cost and reward share the same flawed representation, penalizing predicted safety cannot protect against an unrepresented hazard.

Goodhart’s world
FVD can reward appearance, state MSE can average modes, likelihood can favor irrelevant detail, return can exploit simulator bugs, and uncertainty can be widened to pass coverage. Every metric needs an orthogonal check and an intervention that tries to break its shortcut.

10 · OOD is not one axis

Distribution shift can enter through appearance (new lighting), geometry (new mug shape), dynamics (wet surface), semantics (unusual instruction), embodiment (worn gripper), sensor process (camera latency), action (planner extrapolation), policy (new collector), or combinations. “Held-out test set” should be a matrix of these shifts with increasing severity, not one mystery bucket.

The intuitive mistake is to treat every unfamiliar input as the same problem and attach one “OOD score.” But different shifts break different arrows in the loop, produce different evidence, and demand different behavior. Walk through six concrete failure stories:

  1. Covariate shift changes appearance while leaving physics intact. In a dark kitchen, visual reconstruction and state probes should degrade even though mug dynamics are unchanged. The evidence says “the camera is less informative,” so the safe response is to rely more on depth and touch, improve the view, and slow down—not to invent a new friction law.
  2. Dynamics shift changes what the same-looking action does. A wet mug or worn finger pad may look familiar while residual and contact disagreement rise after motion begins. Because the mismatch is physical, the robot should perform a test lift, lower acceleration, or regrasp. Merely changing image augmentation would miss the failed causal relation.
  3. Semantic shift changes which world state the user intends. With the instruction “that cup,” the referent posterior should remain multimodal if several cups fit. The evidence is unresolved language grounding, so the safe action is to ask for clarification rather than move confidently toward the most common training referent.
  4. Embodiment shift changes how commands become motion. New arm firmware or actuator delay shifts action-effect timing even in a familiar kitchen. The system should detect that lag, recalibrate, and use only a verified controller until the new mapping is established. This is why an action model cannot be versioned independently of the body that executes it.
  5. Planner-induced shift moves action outside data support. An extreme grip or unusual action sequence may have high predicted value precisely because support is low. Low density and ensemble disagreement are warning evidence. Constrain the action or abstain rather than trusting optimization to turn extrapolation into truth.
  6. A novel hazard may not look uncertain at all. A transparent fragile vessel can be confidently misrecognized because the model maps it to a familiar but wrong class. Internal uncertainty alone may not warn us. External rules, human review, and expanded data are necessary defenses when the hazard itself is absent from the learned state.

These stories derive a stronger requirement than “detect OOD”: identify which conditional relationship has changed—sensor-to-state, state-and-action-to-future, language-to-goal, command-to-execution, or state-to-hazard—and select a response that reduces the corresponding risk. The diagnostic is therefore behavioral. Under darkness the model should seek better evidence; under wet contact it should test dynamics; under ambiguous language it should ask. One generic confidence threshold cannot prove those distinctions.

OOD detectors themselves require evaluation: AUROC or precision–recall for known shift labels, but more importantly risk–coverage and closed-loop cost. Near-OOD cases are often harder than obviously alien inputs because the model remains confident. Test compound shifts, delayed detection, and recovery. Evaluate whether state is reset or contaminated after a sensor glitch.

11 · Safety is an envelope, not a confidence threshold

Define the operational design domain: supported objects/materials, workspace, human proximity, speeds, sensors, robot revisions, latency hardware, and environmental conditions. Outside it, do not imply capability. Inside it, use layered defenses: learned risk prediction, calibrated uncertainty, action and workspace constraints, independent collision monitoring, force/torque limits, emergency stop, safe fallback, and audit logs.

For rare catastrophes, zero events in a small test does not prove safety. If zero failures occur in n independent trials, a rough 95% upper bound on event probability is 3/n (“rule of three”). Zero human contacts in 10,000 trials only supports an upper bound near 3×10⁻⁴, far above a 10⁻⁵ requirement. Use accelerated simulation and adversarial scenarios for coverage, formal or verified constraints where possible, conservative real-world canaries, and continuous monitoring. Be honest about dependence: repeated nearly identical trials contain less evidence than diverse independent ones.

A safety case links hazard → detection/constraint → evidence → residual risk → fallback owner. Test sensor loss, stale belief, packet reorder, model timeout, actuator saturation, human intrusion, planner exploitation, and recovery after stop. A system that fails safely but cannot resume still needs operational recovery design.

12 · System-design capstone: Countertop Assistant

In an interview or design review, reason in a fixed linear order. Do not begin with “use a Transformer.” Each answer should constrain the next.

  1. Decision and deadline. Choose 100 ms action chunks at 10 Hz; compare grasp/transport candidates over 1–2 seconds; maintain a faster independent torque loop. Define success and asymmetric safety costs.
  2. Hidden state and observability. Track persistent object identity/pose, occlusion, free space, human trajectory, mug mass/friction/contents, contact mode, robot body state, delay, and semantic goal. Mark which are distributions, not point estimates.
  3. Representation. Use object slots plus local 3D geometry, a body/contact latent, semantic predicates, and a recurrent stochastic belief. Preserve RGB features only where identity/material needs them; do not roll every pixel for planning.
  4. Time and hierarchy. Align physical timestamps; update visual scene at 12–20 Hz, body/contact state at higher rate, task plan at 1–2 Hz, and reflex at 1 kHz. Exchange action chunks, risk bounds, and interrupts.
  5. Dynamics and outputs. Predict distributions over object/body transition, discrete contact mode, human occupancy, task progress, safety cost, and observation residual. Use multimodal latent dynamics and an ensemble or posterior for epistemic uncertainty.
  6. Data. Combine demonstrations, safe randomized interventions, varied simulation, failures/recoveries, and targeted real data. Store executed actions, clocks, calibration, invalidity, robot revision, and behavior policy. Split by kitchen, object instance, and time.
  7. Training and compute. Pretrain modalities, learn action-conditioned short transitions, extend free rollout, add task/safety heads, and train on planner-discovered failures. Quote trajectory tokens, corpus tokens/FLOPs, rollout product, and P99 serving budget.
  8. Planner/controller. Let a VLA propose familiar chunks; use risk-sensitive short-horizon MPC to compare; warm start; replan after evidence; enforce workspace, clearance, force, and stop constraints independently.
  9. Evaluation. Gate data alignment, state probes, horizon curves, contact timing, causal force/speed sweeps, calibration, risk–coverage, exploit search, closed-loop outcomes, OOD slices, and tail latency.
  10. Deployment and recovery. Ship only inside a written envelope, shadow first, canary by scenario, log model/version/belief/fallback, monitor calibration drift, and rollback. When uncertain, view-change, test-contact, slow, ask, or stop.

Concrete tradeoff decision

A photorealistic video model might preserve material cues and support human-readable simulation, but 12-step diffusion across many candidates violates the 55 ms planning budget. A pure object/BEV model is fast but may discard transparency, wetness, and fine contact. The chosen hybrid uses a rich encoder for belief correction, structured low-dimensional latent rollout for 128 candidates, and a renderer only for debugging and a few finalist futures. This sacrifices universal video generation for decision throughput and causal inspectability.

Likewise, a five-second horizon captures complete pick-and-place but compounds uncertainty and compute. Short MPC over one to two seconds, a terminal value for longer goals, and frequent real observation correction better match the product. An ensemble improves epistemic detection but multiplies rollout cost, so use two members for routine planning, a larger shadow ensemble for monitoring, and independent hard constraints for catastrophic risk. Every design choice follows from decision value per unit latency and residual risk.

Objects + contact belief + short MPC
Prioritize touch, action delay, interventions, and recovery over full-frame photorealism.

Prioritize calibrated collision tails, causal response, map consistency, and strict latency.

13 · Diagnose failures by tracing the loop backward

When a task fails, begin with the physical outcome and work backward rather than staring at final video. Was the wrong action selected? Did predicted ranking match external outcomes? Was risk underpredicted or omitted from the objective? Did the planner leave action support? Was belief wrong because evidence was missing, misaligned, or discarded? Was the correct plan late or incorrectly executed? Did the fallback trigger and recover?

Each common symptom suggests a tempting fix, but the first fix is often wrong. Use the contrast between what works and what fails to isolate the broken arrow:

  1. Good one-step prediction but bad long rollout means the local rule exists, yet cannot survive its own outputs. Compare teacher-forced and free-rollout curves, then inspect mode-switch timing. If the gap opens only in free rollout, add free-rollout training and overshooting; if it opens at contact switches, improve the state and its discrete modes. More image reconstruction would not address the dependency that broke.
  2. Good random-action prediction but bad planned-action prediction means selection moved the system into an optimistic error pocket. Plot post-selection error against data support and ensemble disagreement. If error rises as support falls, constrain the planner, collect targeted interventions, or use pessimistic values. The model may be accurate on the test distribution and still unsafe on the distribution its own optimizer creates.
  3. Good pixels but bad task state means the representation preserved appearance and discarded decision variables. Probe geometry and contact, then test causal action effects. Failure there calls for a task-state bottleneck and targeted objectives, because another perceptual-fidelity term will reinforce the shortcut rather than recover the missing state.
  4. Calibration overall but an unsafe subgroup means averaging hid a conditional error. Draw reliability curves by material, horizon, and policy. If wet mugs or planner-selected grasps are overconfident, use group- and horizon-specific calibration and expand data coverage. A single global temperature cannot supply evidence absent from that subgroup.
  5. Offline success but online failure means the learned prediction may be correct while the streaming system changes its inputs or misses its deadline. Inspect latency, stale state, executed rather than commanded action, and concurrency. Reproduce the issue with streaming replay, define a deadline fallback, and repair the systems path before retraining the world model.
  6. Rising uncertainty without falling risk means the score is not connected to a protective decision. Run risk–coverage analysis and replay the complete selective policy. If abstaining on high-score cases does not reduce danger, learn a better score or threshold; if abstention itself is dangerous, design a safer fallback. Merely displaying uncertainty changes no physical outcome.

The common pattern is physical symptom → controlled contrast → localized requirement → scoped response → regression test. This prevents a systems deadline miss from being mislabeled a representation problem, or a planner exploit from being “fixed” by averaging in more ordinary clips.

Keep failure examples as named regression scenarios with raw trajectories, expected causal behavior, and model/system versions. A growing suite converts incidents into durable knowledge. Do not let retraining “fix” the average while reintroducing an old rare failure.

14 · The whole series in one loop

observe oₜ ──infer──▶ calibrated belief bₜ ──intervene aₜ──▶ future distribution ▲ │ │ │ ├── memory / uncertainty ├── reward / risk / observation │ │ │ └──── correct ◀──── real next observation imagine / search / constrain │ │ └──────── act ◀────────────────┘

Graphics supplies forward models from scene to sensor. Vision supplies evidence from sensor to state. 3D and 4D representations provide spatial persistence. Generative models represent multiple possible futures. Control and reinforcement learning turn predictions into decisions. Data and simulation systems preserve the trajectory and test it in closed loop; evaluation tests each causal arrow and the full closed loop. The world model is the contract that makes these disciplines describe one persistent, actionable world.

The linear dependency matters. A planner cannot repair an unobservable state. More rollout compute cannot repair misaligned actions. Calibration cannot make an omitted hazard visible. A safety threshold cannot help if uncertainty does not rank failures. Closed-loop success cannot establish causality if the environment is too easy. Start from the physical decision, preserve information and time, predict interventions, expose uncertainty, plan within support, correct from reality, and measure what failure costs.

Takeaway
A trustworthy world model maintains a calibrated belief over hidden state, predicts stable action-conditioned distributions, exposes where it does not know, improves closed-loop decisions, and stays inside a measured compute and safety envelope. Pixels, tokens, objects, Gaussians, Transformers, diffusion, and search are implementation choices; the test is whether the complete causal loop remains correct under intervention and pressure.

Interview prompts

  • What is the minimum credible evaluation suite? Data/alignment checks, task-state probes, one-step and free-rollout horizon curves, uncertainty calibration, matched action interventions, planner post-selection error, closed-loop utility/safety, OOD risk–coverage, and tail latency.
  • Why can a beautiful video model be a poor world model? Perceptual quality may preserve texture while shifting identity, contact timing, intent, or collision tails; decisions require causally correct task state and calibrated risk.
  • How do you test whether the model uses action causally? Hold initial state comparable, deliberately vary executed action in simulation or safe trials, and compare predicted versus observed dose–response or treatment effects.
  • What is the difference between calibration and accuracy? Calibration matches confidence to empirical frequency; it does not guarantee sharp or correct individual predictions. A broad forecast can be calibrated but unhelpful.
  • How do you detect model exploitation? Increase planner optimization, find high-value low-support/disagreement trajectories, measure error after selection, and validate candidates in an external simulator or bounded real experiment.
  • Why must OOD uncertainty trigger behavior? A score reduces no physical risk unless it causes information gathering, conservative control, abstention, or fallback; risk–coverage tests whether it ranks dangerous cases.
  • How would you investigate strong offline metrics but weak deployment? Trace from outcome backward through executed action, planner ranking, belief and sensor alignment, then inspect stale state, deadline misses, concurrency, calibration shift, and fallback behavior.
  • How should a world-model system-design answer begin? State the decision, deadline, horizon, unacceptable outcome, and hidden state; only then choose representation, dynamics, data, planner, compute, evaluation, and safety envelope.
