Part I · Foundations of data systems

Orientation - the design space of data systems

The neighboring tracks on this site assume the substrate: backend system design picks Postgres or Cassandra and moves on; the data-engineering track builds pipelines on top of a log it does not open; the ML-systems track trains on a feature store and serves from a vector index as if those were primitives. This track is the layer underneath all three. It opens the box. By the end you should be able to look at any of those products — a database, a queue, a cache, a search index, a feature store, a vector DB — and see the same handful of moves: which facts are authoritative, which reads are cheap, which writes are cheap, how much machines must agree, and what breaks when one of them dies.

Book source
Original synthesis inspired by Martin Kleppmann, Designing Data-Intensive Applications — the Preface and the three Part introductions (Foundations, Distributed Data, Derived Data). This lesson lays out the same three-part arc the book uses; later lessons cite the specific chapters. We do not reproduce DDIA's prose or figures.
Linear position
Prerequisite: You can read a SQL query, you know a process can crash and a network packet can be lost, and you have a hand-wave model of a hash table and a B-tree. No distributed-systems background is assumed — every distributed-data term (quorum, replication lag, linearizability, fencing token, CDC, watermark, materialized view) gets defined at its home lesson.
New capability: You can place any storage or messaging product into one design space, name the single contract it offers, and predict the bill it sends elsewhere — so the rest of the track reads as derivations, not a parade of products.
The plan
Five moves. (1) Define what makes an application data-intensive, and draw the one picture the whole track elaborates: a source of truth that spawns derived views through a change log. (2) Lay out the design space as four pressures applied one at a time to a single truthful copy — scale, failure, change, derivation — and show that every product is a point in that space. (3) Walk the linearized 17-lesson path and say why the order is forced, not arbitrary. (4) Teach the one reading habit that makes the rest cheap: every mechanism is a contract bought at a cost charged somewhere else — name the constraint before the tool. (5) Hand off to lesson 01, which makes the three target properties (reliability, scalability, maintainability) precise.

1 · What "data-intensive" means, and the one picture

An application is compute-intensive when its hardest problem is doing arithmetic fast — a physics simulation, a renderer, a training loop's matmuls. It is data-intensive when CPU is rarely the bottleneck and the hard problems are instead the amount of data, the complexity of the data, and the speed at which it changes. Almost every product you ship is the second kind. The difficulty is not computing an answer; it is storing facts durably, moving them between machines, interpreting them consistently as schemas drift, and getting many machines to agree on them when some are slow or dead.

Strip a data-intensive system to its skeleton and you always find the same shape. There is a system of record — the authoritative copy of the facts, the one you would rebuild everything else from after a disaster. Then there are derived views: copies shaped for a specific read pattern (a search index for full-text queries, a cache for hot keys, an analytics table for aggregation, an embedding index for similarity, precomputed features for a model). Derived views are never authoritative; they are functions of the system of record, kept in sync by a flow of changes. That flow — the change log — is the spine of the entire track.

WRITE (a new fact) | v +------------------------+ | SYSTEM OF RECORD | the one authoritative copy | (durable, transactional)| — survives crashes, is the truth +------------------------+ | change log (an ordered, durable stream of every change) | +---------------------+---------------------+-----------------+ v v v v +-----------+ +-------------+ +-------------+ +---------------+ | SEARCH | | CACHE | | ANALYTICS | | ML FEATURES / | | INDEX | | (hot reads) | | COLUMNAR | | VECTOR INDEX | +-----------+ +-------------+ +-------------+ +---------------+ DERIVED VIEW DERIVED VIEW DERIVED VIEW DERIVED VIEW invariant: every derived view is a CACHE of the system of record. if you cannot recompute it from the log, you cannot trust it.

Read that diagram as a promise and a hazard. The promise: each derived view gets exactly the layout its reads want, so each read is cheap. The hazard: the moment there is more than one copy, they can disagree — a view can lag, drop a change, or apply one twice — and now "what is true?" has more than one answer. Every later lesson is either (a) making one copy faster, safer, or more durable, or (b) keeping the copies honest with each other. Hold onto the rule on the last line: a derived view you cannot recompute from the log is not a view, it is a second source of truth you forgot you had.

2 · The design space: four pressures on one truthful copy

Start from the simplest correct system imaginable: one machine, one database, one copy of every fact, every operation a local transaction. It is wonderfully easy to reason about. Nothing is stale, nothing disagrees, "what is true?" has exactly one answer. We complicate it only because reality applies four pressures, and each one, applied to that single copy, generates a whole region of the design space — and a block of this track.

Pressure 1 · Scale
The data or the request rate outgrows one machine. You partition (split the data across machines by key) for volume and replicate (keep copies) for read throughput. Now reads are cheap and big, but copies can disagree. → lessons 07–09.
Pressure 2 · Failure
Machines crash, clocks drift, networks drop and reorder packets, a node pauses for a 3 s garbage-collection and wakes thinking it is still the leader. You need durability, consistency models, fencing, and sometimes consensus (machines agreeing on one value). → lessons 10–13.
Pressure 3 · Change
Humans keep editing the software and the data shape. Old and new code must read each other's data during a rolling deploy. You need encodings and schema evolution (backward/forward compatibility) so a write today is still readable next year. → lesson 04.
Pressure 4 · Derivation
One layout cannot serve every read pattern. You spawn the derived views above and keep them in sync with batch recomputation and stream processing (incremental updates from the change log). Now end-to-end correctness across systems is the problem. → lessons 14–16.

Crucially these pressures are not independent products you bolt on; they interact, and the interactions are where the famous trade-offs live. Add replicas for scale (pressure 1) and you immediately inherit a failure question (pressure 2): when a follower lags behind the leader, may a user read their own just-written comment and not see it? Spawn a derived view for derivation (pressure 4) and you inherit a change question (pressure 3): when the source schema evolves, does the view's recompute still parse the old records? Every named product is just a point — or a path — in this space. A single-node Postgres lives in the top-left, paying nothing for distribution and buying clean transactions (lesson 10). Cassandra trades those clean transactions for leaderless quorum replication that survives node loss (lesson 08). Kafka is the change log itself, promoted to a first-class durable object (lessons 05, 15). A vector index for RAG is a derived view (pressure 4) whose freshness is a stream-processing question (lesson 15) and whose recompute is a batch question (lesson 14). You will not memorize these verdicts; you will be able to re-derive them.

3 · The linearized path, and why the order is forced

The track is deliberately linear because the dependencies are real: you cannot reason honestly about replication before you know what a durable log is, and you cannot reason about linearizability before you have felt replication lag. The order below is the order of least new assumptions per step.

IFoundations (00–06). Name the target properties (01), choose how data is shaped and queried (02–03), make writes survive deploys and time via encoding and schema evolution (04), then open a single node's storage engine — the append-only log, hash index, and write-ahead log; the B-tree; the LSM-tree and columnar layouts (05–06). Everything here is still one machine.
IIDistributed data (07–13). Now there is more than one machine. Replicate for read scale and availability, and meet replication lag (07). Give up the single leader: quorum and leaderless writes, and the conflicts they create (08). Partition for volume and meet hot keys and rebalancing (09). Hold invariants under concurrency with transactions and isolation (10). Confront the ugly truth of partial failure — crashes, unreliable clocks, fencing tokens (11). Pin down what "consistent" even means: causality and linearizability (12). Finally, get machines to agree: total-order broadcast and consensus (13).
IIIDerived data (14–16). Turn one dataset into many, correctly. Batch processing recomputes views from scratch (14). Stream processing keeps them fresh incrementally from the change log — CDC, event time vs processing time, windows, exactly-once (15). The capstone ties it together: derived data, the lambda/kappa patterns, and end-to-end correctness across systems (16).

If you only have time for the spine: 05 (the durable ordered log, the object the whole track keeps reusing), 07 (replication lag — the first place "more than one copy" hurts), 12 (linearizability — the strongest single-copy illusion and what it costs), and 16 (how derived views stay correct end to end). The other thirteen lessons earn those four.

4 · The one reading habit: contract bought at a cost

Here is the single mental move that makes the rest of the track cheap. Every mechanism is a contract bought at a cost charged somewhere else. There is no free win; there are only relocations of cost. When you meet a new mechanism, ask three questions in this order — and refuse to name the tool until you have answered the first:

1What constraint forces this? A workload number (writes/sec, bytes/record, fan-out), an invariant (no double-spend), a failure model (must survive one rack dying), or a freshness SLA. Name the constraint before the tool — tools are conclusions, not opening moves.
2What contract does it offer? "Reads see the latest write." "A write is durable before I acknowledge it." "Two machines never disagree on the leader." This is the promise, stated precisely enough to test.
3Where is the bill? The cost never disappears — it moves to read latency, write latency, storage, availability during a network partition, freshness, or human operational complexity. If you cannot point at the bill, you have not understood the mechanism.

The trade-off table below is the same habit applied to the four corners of this lesson's design space. Each row is a contract and its bill; the rest of the track fills in the mechanics behind each.

A worked instance to make the habit concrete. Suppose a feature store serves 50,000 reads/sec to an online ranker with a 10 ms p99 budget, and a single primary database tops out at 20,000 reads/sec. The constraint is read throughput (2.5× over capacity) under a tight latency tail. The tool is read replicas: add four followers and you have 5 × 20,000 = 100,000 reads/sec of headroom, well inside budget. The contract is "more read capacity, and a copy survives a primary crash." The bill arrives as replication lag: if a follower is 200 ms behind the primary, a user who just updated their profile and is re-ranked immediately may be scored on yesterday's feature. That single number — 200 ms of staleness — is the price of the throughput, and lesson 07 is where you decide whether you can pay it or must route that one read to the primary (read-your-writes). You bought a contract; the bill landed on freshness. That is the whole track in one example.

Failure modes of skipping the substrate

  • Product-shopping instead of designing. "Postgres vs Cassandra," "Kafka vs RabbitMQ," "cache vs materialized view" become matters of taste rather than consequences of a named workload, failure model, and invariant.
  • The forgotten second source of truth. Treating a search index, cache, or vector DB as authoritative because nobody can recompute it from the log — so when it drifts, there is no clean answer to "what is true?"
  • Free-lunch thinking. Adopting a mechanism (more replicas, async derivation) without locating the bill, then being blindsided when staleness, conflicts, or operational load show up later.
  • Coordination by default. Reaching for strong consistency or distributed transactions everywhere, paying latency and availability for invariants most reads never needed.

How to read every later lesson

  • Name the constraint (workload number / invariant / failure model / freshness SLA) before naming any tool.
  • State the contract the mechanism offers as a testable promise.
  • Locate the bill: read latency, write latency, storage, availability under partition, freshness, or operational complexity.
  • Ask which copy is the system of record and which are derived views — and whether each view is recomputable from the log.
  • Watch which pressure (scale / failure / change / derivation) the lesson is answering, and which other pressure its answer newly exposes.

Checkpoint exercise

Try it
Take a RAG product: documents are ingested, embedded, and stored in a vector index; user queries retrieve nearby chunks; a model answers. (1) Which store is the system of record and which are derived views? (2) For the vector index, name the constraint that justifies it, the contract it offers, and the bill (which of read/write/storage/freshness/availability does it charge?). (3) When a document is edited, which of the four pressures does keeping the index correct exercise, and could you fully recompute the index from the system of record if it were lost? If the answer to that last question is "no," you have found a hidden second source of truth — name it.

Where this points next

We have a map and a reading habit, but three words on it are still vague: a system must be reliable, scalable, and maintainable. Those cannot stay slogans — you cannot trade against a property you cannot measure. Lesson 01 makes each one precise: reliability as the discipline of stopping faults (one component misbehaving) from becoming failures (the whole system breaking its promise); scalability as describing load in concrete units and performance in percentiles, not averages, with a hard look at why latency explodes near saturation; and maintainability as the operability, simplicity, and evolvability cost that humans pay on a bad Tuesday. Those three properties are the yardstick every later mechanism is graded against.

Takeaway
A data-intensive system's hard problems are storing, moving, interpreting, and agreeing on data — not computing. Strip any such system down and you find one shape: an authoritative system of record spawning derived views through an ordered, durable change log, with the iron rule that any view you cannot recompute from the log is a hidden second source of truth. The whole design space is that one truthful copy under four pressures — scale (partition + replicate), failure (durability, consistency, consensus), change (encoding + schema evolution), and derivation (batch + stream) — and every named product (Postgres, Cassandra, Kafka, a vector index) is just a point in it. Read every later lesson with one habit: name the constraint before the tool, state the contract the mechanism offers, and locate the bill it charges to read latency, write latency, storage, availability, freshness, or operational complexity. Nothing is free; there are only relocations of cost.
