Part I - Core mental model

Runtime, Scheduling, and the Object Store

The invisible machinery: head node metadata, worker processes, local object stores, resources, locality, and what happens when bytes move.

Source map
Book coverage: Chapter 2, 'Understanding Ray System Components' - scheduling and executing work on a node, the head node, and distributed scheduling.

The thesis

Most Ray performance questions reduce to three runtime questions: where is the work placed, where are its input objects, and what resource is actually scarce?

Linear path

  1. Stop thinking of a cluster as a bigger laptop. It is a set of nodes with separate CPU, GPU, memory, disk, and network boundaries.
  2. The scheduler must place tasks near resources and, when possible, near data. Every placement is a compromise between locality, fairness, and utilization.
  3. The object store makes intermediate values shareable across workers, but it also introduces a memory budget. When the object store spills, performance can change suddenly.

The mechanism

Head node
The head coordinates cluster metadata and exposes control-plane services. It is not where all useful work should happen.
Local schedulers
Work is placed across nodes based on resources, dependencies, and availability. Resource annotations are part of the program contract.
Plasma-style object store
Large immutable objects can be shared by processes on a node without every worker holding its own Python copy.
Spilling and locality
If objects exceed memory, they may spill to disk or remote storage. If a task runs far from its inputs, network transfer becomes part of task latency.

Decision surface

Trade-offHow to reason about it
Locality vs utilizationWaiting for data-local resources can reduce network movement. Running anywhere can keep CPUs busy but move bytes.
Resource reservation vs flexibilityDeclaring resources prevents overcommit. Over-declaring resources leaves cluster capacity idle.
Object store vs external storageThe object store is excellent for intermediate results. Durable datasets and checkpoints belong in external storage with explicit lifecycle.

Operational checklist

  • Annotate GPUs and custom resources explicitly; do not rely on accidental placement.
  • Watch object-store memory separately from Python heap memory.
  • Use chunk sizes large enough for efficient transfer but small enough for load balancing and retry.
  • Design for spill behavior before production: where do objects spill, how fast is the disk, and what happens under backpressure?

Failure modes

  • Optimizing function code while the real bottleneck is object transfer.
  • Forgetting that one node's object store is not infinite shared RAM.
  • Letting the head node run application-heavy work by accident.
