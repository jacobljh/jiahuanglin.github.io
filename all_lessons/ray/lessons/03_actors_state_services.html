Part I - Core mental model

Actors, State, and Services

When a function is not enough: actors let Ray keep state near the worker that owns it.

Source map
Book coverage: Chapter 2 Core API plus Chapter 3 distributed application design - stateful components and simulation workers.

The thesis

Tasks are for independent work. Actors are for identity: state, caches, environments, connections, model replicas, and long-lived services that should not be recreated for every call.

Linear path

  1. Ask whether the work has memory. If every call is independent, use tasks. If calls must share state, use an actor.
  2. Treat an actor as a remote Python object with a mailbox. Method calls are scheduled to the actor process and can mutate its local state.
  3. Use actors to colocate expensive state with computation, but remember that a single actor can become a serial bottleneck unless you design around concurrency and sharding.

The mechanism

State ownership
An actor owns Python state inside one worker process. That state can be a model, simulator, cache, database connection, or mutable counter.
Method refs
Actor methods also return object references. You can compose actor calls with task graphs, not just call them synchronously.
Resource pinning
An actor can reserve resources for its lifetime, such as one GPU for a model replica.
Sharding
Many small actors can partition state. One large actor centralizes correctness but often limits throughput.

Decision surface

Trade-offHow to reason about it
Actor vs taskActors avoid repeated setup and keep state local. Tasks are easier to parallelize and retry independently.
One actor vs many actorsOne actor simplifies coordination. Many actors increase throughput but force you to design routing and aggregation.
Mutable state vs reproducibilityMutable actors are convenient for services and simulations. They are harder to replay than deterministic task graphs.

Operational checklist

  • Use actors for model replicas, simulators, online learners, caches, queues, and external clients with connection state.
  • Make actor methods coarse enough to amortize RPC overhead.
  • Shard actors by natural key when throughput matters: user id, environment id, model id, partition id, or tenant.
  • Define what happens if an actor dies: rebuild from checkpoint, restart empty, or fail the job.

Failure modes

  • Putting all coordination in one actor and discovering it is the new global lock.
  • Using actors for pure stateless functions, which adds lifecycle complexity without benefit.
  • Forgetting that actor-local state is not magically durable.

class ModelReplica:
    def __init__(self, weights_path):
        self.model = load_model(weights_path)

    def predict(self, batch):
        return self.model(batch)

Exercise
Pick a service you would normally deploy behind HTTP. Split it into stateless work and stateful ownership. Which parts are tasks? Which parts are actors? Which state must be checkpointed rather than merely kept warm?
