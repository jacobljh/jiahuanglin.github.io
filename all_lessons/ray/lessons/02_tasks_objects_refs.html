Part I - Core mental model

Tasks, Objects, and Futures

The first real Ray program: remote functions return object references, and object references let a dependency graph run asynchronously.

Source map
Book coverage: Chapter 2, 'Getting Started with Ray Core' - first Ray API example and overview of Ray Core primitives.

The thesis

A Ray task is not just a function call on another process. It is a node in a distributed dependency graph whose inputs and outputs are tracked by object references.

Linear path

  1. Replace a function call with a task submission. Calling f.remote(x) returns immediately with an object reference instead of blocking for the value.
  2. Use the reference as a dependency token. A downstream task can accept that reference, so the driver does not need to materialize every intermediate value.
  3. Only block at the boundary where the local process truly needs the value. In Ray, ray.get is a synchronization point, and synchronization points define the shape of parallelism.

The mechanism

Task submission
A remote function call packages code identity, arguments, declared resources, and dependencies. The scheduler can place it when inputs and resources are available.
Object references
The result is represented by an ObjectRef. Passing refs around is cheap compared with passing the underlying bytes.
Implicit dependencies
If task B receives task A's object ref, Ray knows B depends on A. You get a DAG without manually building a scheduler.
Argument dereferencing
A top-level ObjectRef argument is dereferenced before the task runs; a ref nested inside a list or dict remains a ref unless the task explicitly calls ray.get.
Synchronization
ray.get pulls results back and blocks. ray.wait lets you consume whichever results finish first, which is often the right shape for straggler-heavy workloads.

Decision surface

Trade-offHow to reason about it
Large tasks vs small tasksLarge tasks amortize scheduling overhead but reduce load balancing. Small tasks expose more parallelism but can drown in overhead.
Eager get vs lazy graphCalling ray.get inside a loop collapses parallelism. Deferring gets increases concurrency but can increase memory pressure.
Object refs vs concrete valuesRefs let the runtime move data lazily. Concrete values are easier to inspect but pull bytes back to the driver.

Operational checklist

  • Batch tiny records into chunks before submitting remote work.
  • Submit a wave of tasks before calling ray.get, unless the next submission truly depends on the previous value.
  • Use ray.wait when task durations vary and you can stream completed results.
  • Pass refs as top-level task arguments when the callee needs the value; nest refs only when the callee should forward or manually fetch them.
  • Keep driver memory boring: the driver should coordinate, not become the warehouse for every intermediate object.

Failure modes

  • Calling ray.get immediately after every .remote() call, turning a parallel program back into a serial one.
  • Calling ray.get inside a task on a top-level argument that Ray has already dereferenced.
  • Returning huge objects from every task when downstream work could have consumed references directly.
  • Letting millions of tiny refs accumulate in the driver without backpressure.

Implementation sketch

@ray.remote
def parse(path):
    return load_and_parse(path)

@ray.remote
def summarize(rows):
    return compute_stats(rows)

parsed = [parse.remote(path) for path in paths]
summaries = [summarize.remote(ref) for ref in parsed]

ready, remaining = ray.wait(summaries, num_returns=10)
first_ten = ray.get(ready)
