Part I - Core mental model

Orientation - Ray as distributed Python

A linear map of the whole series: tasks, actors, objects, ML libraries, clusters, and the trade-offs that make Ray powerful without making it magic.

Source map
Book map: Preface and Chapters 1-11. Current-docs note: modern Ray documentation frames the ML stack as Ray Data, Train, Tune, Serve, RLlib, and KubeRay rather than the older AIR umbrella as the main teaching surface.

The thesis

Ray is best understood as one idea repeated at several scales: write ordinary Python units, let the runtime turn them into a distributed graph, and then use higher-level libraries when the graph shape is common enough to deserve an API.

Linear path

  1. First separate the programming model from the cluster. A local program has functions, classes, and values. A Ray program has remote functions, remote stateful classes, and object references.
  2. Then follow the data. Values that cross task boundaries become serialized objects; large objects live in a shared object store; references move more cheaply than bytes, until a downstream task actually needs the value.
  3. Finally lift the same runtime into ML workflows. Data, training, tuning, serving, and RL are not five different distributed systems. They are five recurring graph shapes on top of Ray Core.

The mechanism

Driver
The process you start is the driver. It submits tasks, creates actors, waits on object references, and owns the application-level control flow.
Workers
Workers execute task functions and actor methods. They are ordinary Python processes scheduled across a node or cluster.
Object store
Intermediate values are addressable by object reference. This lets tasks compose asynchronously, but it also makes object size, serialization, and locality first-class design concerns.
Libraries
Ray Data, Train, Tune, Serve, and RLlib encode common distributed patterns so you do not rebuild shuffle, checkpointing, autoscaling, or rollout collection by hand every time.

Decision surface

Trade-offHow to reason about it
Ergonomic Python vs invisible costsRay lets local-looking Python scale out, but serialization, network transfer, scheduling delay, and failure semantics still exist.
General runtime vs specialized engineRay is flexible for mixed workloads; Spark, Flink, or a purpose-built serving engine may win when the workload is narrower and highly optimized.
One substrate vs many toolsA unified runtime simplifies handoffs between data, train, tune, and serve. It can also concentrate operational blast radius in one cluster layer.

Operational checklist

  • Name the unit of parallelism before writing code: task, actor, dataset block, training worker, tuning trial, serving replica, or RL rollout worker.
  • Ask where the large bytes live: input files, object store, GPU memory, checkpoints, model weights, or response payloads.
  • Decide which layer owns retries and failure recovery: your code, Ray Core, Tune, Train, Serve, or the cluster manager.
  • Keep the first version small enough to run locally with ray.init(); scale the same mental model to the cluster only after the dependency graph is clear.

Failure modes

  • Treating Ray as a speed button. A serial bottleneck wrapped in @ray.remote is still a serial bottleneck plus scheduling overhead.
  • Passing large Python objects through many small tasks. You pay serialization and object-store pressure even if the function body is trivial.
  • Using every Ray library at once before understanding the Core graph. Libraries help most when you already know the shape they are hiding.

Implementation sketch

import ray

ray.init()

@ray.remote
def score(record):
    return expensive_model(record)

refs = [score.remote(r) for r in records]
results = ray.get(refs)
Exercise
Take one Python loop you have written recently. Mark which values are inputs, which values are independent, which values are shared state, and which values are too large to copy casually. That annotation usually tells you whether the Ray shape is a task graph, actor service, dataset pipeline, or not a Ray problem yet.
