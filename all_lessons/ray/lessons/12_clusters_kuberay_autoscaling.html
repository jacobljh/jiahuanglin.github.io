Part III - Cluster and platform layer

Ray Clusters, KubeRay, and Autoscaling

The moment a local Ray app outgrows one machine, the real system becomes head pods, worker groups, resources, networking, logs, and autoscaling policy.

Source map
Book coverage: Chapter 9, 'Ray Clusters' - manual clusters, Kubernetes/KubeRay, cluster launcher, cloud clusters, and autoscaling.

The thesis

A Ray cluster is a runtime topology. The cluster layer decides where code can run, what resources exist, how workers join, how logs are found, and whether scale-up arrives before the workload times out.

Linear path

  1. Begin with the cluster shape: one head node and many worker nodes. The head owns control-plane coordination; workers provide most compute.
  2. On Kubernetes, express that shape as RayCluster resources managed by KubeRay. Kubernetes owns pods; Ray owns distributed execution inside them.
  3. Autoscaling connects application resource requests to infrastructure. It works only if tasks and actors declare realistic resources.

The mechanism

Head and workers
The head exposes the dashboard, job submission, and cluster coordination. Workers register resources and execute tasks or actors.
KubeRay
KubeRay turns Ray clusters into Kubernetes-native custom resources with worker groups, versions, images, and autoscaling settings.
Cluster launcher
For VM-based deployments, Ray cluster configs can define provider, node types, setup commands, and scaling limits.
Autoscaler
The autoscaler observes pending resource demands and adjusts worker count. It cannot infer resources your program forgot to request.

Decision surface

Trade-offHow to reason about it
Kubernetes vs VM launcherKubeRay fits platform teams already operating Kubernetes. VM launcher can be simpler for dedicated cloud clusters.
Fixed cluster vs autoscalingFixed clusters are predictable and fast to start. Autoscaling improves cost efficiency but adds cold-start and scheduling uncertainty.
Shared cluster vs per-job clusterShared clusters improve utilization. Per-job clusters simplify isolation, dependencies, and cost attribution.

Operational checklist

  • Define node groups by workload: CPU data processing, GPU training, serving replicas, high-memory reducers.
  • Set min/max workers and startup time expectations per group.
  • Centralize logs, metrics, and dashboard access before the first incident.
  • Test node loss and worker restart behavior with a noncritical job.

Failure modes

  • Using autoscaling without resource annotations, leaving the autoscaler blind.
  • Letting every team build a slightly different Ray image and dependency stack.
  • Exposing the dashboard or client endpoints without a clear auth and network boundary.

Implementation sketch

# KubeRay shape, abbreviated:
apiVersion: ray.io/v1
kind: RayCluster
spec:
  headGroupSpec:
    rayStartParams:
      dashboard-host: "0.0.0.0"
  workerGroupSpecs:
  - groupName: gpu-workers
    minReplicas: 0
    maxReplicas: 8
    template:
      spec:
        containers:
        - name: ray-worker
          resources:
            limits:
              nvidia.com/gpu: 1
Exercise
A Tune job launches 200 CPU trials and 8 GPU training trials. Design two worker groups. What are the min/max counts? What happens if GPU nodes take 12 minutes to become ready?
