Part I - Core mental model

Distributed Patterns: Map, Shuffle, Reduce

The simple MapReduce example is not toy material. It is the shape behind many Ray Data, Tune, and training pipelines.

Source map
Book coverage: Chapter 2, 'A Simple MapReduce Example with Ray' - mapping, shuffling document data, and reducing word counts.

The thesis

MapReduce teaches the first unavoidable distributed-systems law: maps are easy, shuffles are expensive, and reduces concentrate pressure.

Linear path

  1. Map expands parallelism. Each input block can be processed independently, so the scheduler has many units of work.
  2. Shuffle changes ownership. Records move from the mapper that saw them to the reducer responsible for their key. This is where network, skew, and memory appear.
  3. Reduce collapses parallelism. The output cardinality and key distribution decide whether the final stage is cheap or a hotspot.

The mechanism

Partitioning
A partition function maps each key to a reducer. Good partitioning spreads both bytes and compute, not merely key count.
Combining
Partial aggregation before the shuffle reduces bytes moved. It is one of the highest-leverage optimizations in distributed data systems.
Skew
One hot key can dominate runtime. Skew is not a bug in Ray; it is a property of the data distribution exposed by the reduce stage.
Streaming results
Consuming completed partitions as they finish can improve memory behavior and user-visible progress.

Decision surface

Trade-offHow to reason about it
Manual Core vs Ray DataCore is excellent for learning and custom graphs. Ray Data gives you optimized block execution, common transforms, and ecosystem integration.
Fine partitions vs coarse partitionsFine partitions improve balancing and retries. Coarse partitions reduce scheduling overhead and metadata volume.
Exact aggregation vs approximate sketchesExact counts are simple but heavy for high-cardinality keys. Sketches reduce cost when approximate answers are acceptable.

Operational checklist

  • Measure input bytes, intermediate bytes, and output bytes separately.
  • Add combiners before the shuffle when the operation is associative and commutative.
  • Inspect key distribution before blaming Ray for a slow reduce.
  • Prefer library-level dataset operations once the pattern is a standard data pipeline rather than a custom graph.

Failure modes

  • Making one reducer responsible for a popular key without salting or hierarchical aggregation.
  • Returning raw per-record data from mappers when a partial aggregate would be enough.
  • Using task count as a proxy for parallelism without checking actual bytes per task.

Implementation sketch

@ray.remote
def count_words(lines):
    counts = {}
    for line in lines:
        for word in line.split():
            counts[word] = counts.get(word, 0) + 1
    return counts

@ray.remote
def merge_counts(*partials):
    merged = {}
    for counts in partials:
        for word, n in counts.items():
            merged[word] = merged.get(word, 0) + n
    return merged

partial_refs = [count_words.remote(chunk) for chunk in document_chunks]
total_ref = merge_counts.remote(*partial_refs)
