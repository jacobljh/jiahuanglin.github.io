Part II - Applications and libraries

Your First Distributed Application

A reinforcement-learning maze is useful because it separates environment simulation, policy evaluation, learning, and coordination.

Source map
Book coverage: Chapter 3, 'Building Your First Distributed Application' - simple maze, simulation, RL model, distributed Ray app, and the first look behind the scenes with Ray tooling.

The thesis

The first serious Ray design move is decomposition: decide which part of the application creates experience, which part learns from it, and which part coordinates progress.

Linear path

  1. Model the problem locally first. In the book's RL example, there is an environment, actions, rewards, trajectories, and a learning rule.
  2. Find the parallel boundary. Many rollouts can be generated independently from the same policy snapshot, so rollout simulation is a natural task or actor pool.
  3. Control synchronization carefully. Updating a policy after every tiny rollout wastes time; waiting for all rollouts invites stragglers. Distributed learning lives between those extremes.

The mechanism

Environment workers
Each worker owns a copy of the environment and produces trajectories. This is stateful enough to be actor-shaped in many real RL systems.
Policy snapshot
Workers need a consistent enough policy version to generate useful data, but not necessarily the latest weights at every step.
Learner
The learner consumes trajectories and updates parameters. It is often the coordination bottleneck and may need batching.
Tooling view
The dashboard, task logs, and object-store metrics are part of the app model: they show whether time is spent sampling, learning, transferring objects, or waiting for resources.
Metrics loop
Reward, episode length, sample throughput, and learner throughput must be observed together; one metric alone lies easily.

Decision surface

Trade-offHow to reason about it
Synchronous vs asynchronous rolloutsSynchronous collection is easier to reason about. Asynchronous collection improves utilization but introduces policy staleness.
Tasks vs actors for simulatorsTasks fit cheap stateless rollouts. Actors fit expensive environment setup, persistent simulators, or external connections.
From scratch vs RLlibBuilding once by hand teaches the graph. RLlib is usually better once the problem is a standard RL experiment.

Operational checklist

  • Write the single-process version first and keep it as a correctness oracle.
  • Make rollout workers coarse enough that simulation dominates scheduling overhead.
  • Separate sample throughput from learning throughput; optimizing only one can starve the other.
  • Define the policy-staleness budget if workers can run ahead of the learner.

Failure modes

  • Parallelizing before the environment and reward are correct.
  • Updating shared state from many workers without an explicit owner.
  • Measuring only total reward and missing the fact that the cluster spends most time waiting.

Implementation sketch

@ray.remote
class RolloutWorker:
    def __init__(self, env_config):
        self.env = MazeEnv(env_config)

    def sample(self, policy_weights, episodes=8):
        policy = Policy.from_weights(policy_weights)
        return [run_episode(self.env, policy) for _ in range(episodes)]

workers = [RolloutWorker.remote(cfg) for _ in range(32)]
refs = [w.sample.remote(weights_ref) for w in workers]
trajectories = sum(ray.get(refs), [])
