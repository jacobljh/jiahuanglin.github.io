Ray Tune: Experiments and Hyperparameter Search

Tune is not just random search on more machines. It is a control plane for trials, resources, schedulers, checkpoints, and early stopping.

Source map
Book coverage: Chapter 5, 'Hyperparameter Optimization with Ray Tune' - random search, why HPO is hard, Tune architecture, configuration, RLlib, and Keras examples.

The thesis

Hyperparameter optimization is distributed systems under uncertainty: many trials compete for resources, partial signals arrive at different times, and the scheduler must decide what deserves more compute.

Linear path

  1. Represent each experiment as a trainable with a config, metric, resource request, and checkpoint behavior.
  2. Define a search space and search algorithm. Random search is a baseline; Bayesian and evolutionary methods use observations to choose future trials.
  3. Add a trial scheduler. ASHA, HyperBand, and PBT change allocation over time so weak trials stop early and promising trials receive more budget.

The mechanism

Trainable
The function or class that performs one trial. It reports metrics and checkpoints state so Tune can monitor and resume it.
Search algorithm
Chooses the next config to try. It is about exploration of parameter space.
Scheduler
Allocates time and resources among active trials. It is about exploitation, early stopping, and promotion.
Storage
Metrics and checkpoints are the experiment record. Without durable storage, distributed tuning becomes expensive guesswork.

Decision surface

Trade-offHow to reason about it
Random search vs adaptive searchRandom search is robust and parallel. Adaptive search can be sample-efficient but may bottleneck on sequential decisions.
Aggressive early stopping vs patienceEarly stopping saves compute. It can also kill slow-starting configurations that would have won later.
Trial parallelism vs per-trial scaleMany small trials explore broadly. Fewer distributed trials evaluate large configurations realistically.

Operational checklist

  • Choose the metric and optimization direction before launching trials.
  • Make checkpointing cheap enough to run often, but complete enough to resume faithfully.
  • Specify per-trial resources so Tune can pack the cluster intentionally.
  • Keep a simple baseline search in the experiment plan; clever search without a baseline is hard to trust.

Failure modes

  • Tuning on a noisy validation metric without repeated seeds or confidence intervals.
  • Letting the search algorithm compare trials that ran for very different budgets without a scheduler-aware metric.
  • Storing checkpoints on local disk and losing them when trials move or nodes die.

Implementation sketch

from ray import tune

def trainable(config):
    model = build_model(config)
    for epoch in range(20):
        loss, acc = train_one_epoch(model, config)
        tune.report({"loss": loss, "accuracy": acc})

tuner = tune.Tuner(
    trainable,
    param_space={"lr": tune.loguniform(1e-5, 1e-1), "width": tune.choice([64, 128, 256])},
)
results = tuner.fit()
Exercise
You have 64 GPUs and two options: 64 one-GPU trials or 8 eight-GPU trials. What changes if your final model will train on 8 GPUs? What changes if the search is over data augmentation rather than model size?
