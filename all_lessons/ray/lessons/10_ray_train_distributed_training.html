Ray Train: Distributed Training

Ray Train wraps the recurring shape of distributed ML: workers, data shards, framework launch, checkpoints, metrics, and scale-out.

Source map
Book coverage: Chapter 7, 'Distributed Training with Ray Train' - training basics, example pipeline, trainers, migration, scaling, preprocessing, Tune integration, and callbacks.

The thesis

Distributed training is not merely running the same script on more GPUs. It is coordinating data shards, model replicas, gradients, checkpoints, metrics, and failure recovery as one job.

Linear path

  1. Start from the scaling reason: model too large, data too large, training too slow, or experiment iteration too expensive.
  2. Choose the training topology. Data parallelism, framework-specific distributed strategies, and external libraries such as DeepSpeed each imply different communication patterns.
  3. Use Ray Train as the job harness: launch workers, provide datasets, report metrics, save checkpoints, and integrate with Tune when the training job becomes one trial among many.

The mechanism

Trainer
The trainer owns the distributed job configuration and launches worker processes with the selected framework integration.
Scaling config
Number of workers, GPU usage, and resources define the physical shape of the job.
Datasets and iterators
Ray Data can feed workers with sharded batches, which reduces glue code between preprocessing and training.
Checkpointing
Checkpoints are the recovery boundary and the handoff point to tuning, batch prediction, or serving.

Decision surface

Trade-offHow to reason about it
Minimal migration vs full integrationWrapping an existing training loop is fast. Deep integration with Ray Data, callbacks, and checkpointing gives better operability.
More workers vs communication taxMore GPUs increase compute, but gradient synchronization and input pipeline pressure can erase gains.
Report cadence vs synchronization costtrain.report is a worker synchronization point. Frequent reports improve visibility and checkpoint freshness, but every worker must reach the same reporting steps.
Train alone vs Train plus TuneA single production training job needs stability. A research sweep needs trial orchestration and early stopping.

Operational checklist

  • Measure data-loader throughput before blaming distributed training.
  • Save checkpoints to durable shared storage, not worker-local scratch.
  • Report metrics at a cadence that is useful for monitoring but not so frequent it becomes overhead.
  • Make sure every worker calls train.report the same number of times.
  • Make the single-worker version deterministic enough to debug before scaling.

Failure modes

  • Scaling workers while the input pipeline still starves the GPUs.
  • Checkpointing only model weights and forgetting optimizer, scheduler, tokenizer, or preprocessing state.
  • Reporting or checkpointing from only one worker while the others skip the barrier.
  • Ignoring cluster placement and putting workers across slow links when communication is frequent.

Implementation sketch

from ray import train
from ray.train.torch import TorchTrainer
from ray.train import ScalingConfig

def train_loop_per_worker(config):
    model = build_model(config)
    train_loader = get_data_iterator()
    for batch in train_loader:
        loss = step(model, batch)
        train.report({"loss": loss})

trainer = TorchTrainer(
    train_loop_per_worker,
    scaling_config=ScalingConfig(num_workers=4, use_gpu=True),
)
result = trainer.fit()
