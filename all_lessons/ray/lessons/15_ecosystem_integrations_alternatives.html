Part III - Cluster and platform layer

Ecosystem, Integrations, and Alternatives

Ray is strongest when you know what it should replace, what it should integrate with, and what it should leave alone.

Source map
Book coverage: Chapter 11, 'Ray's Ecosystem and Beyond' - data processing, model training, model serving, custom integrations, and comparisons with other systems.

The thesis

Choosing Ray is an architecture decision, not a popularity vote. The right question is: which control loop do you want Ray to own?

Linear path

  1. Map the surrounding ecosystem: data lakes, Spark jobs, model frameworks, experiment trackers, feature stores, orchestration systems, Kubernetes, and serving gateways.
  2. Decide where Ray is the compute substrate and where it is just one client. Ray can own a whole workflow or just one stage.
  3. Compare alternatives by workload shape, not by feature checklist: batch SQL, streaming events, task queues, model serving, GPU training, RL, and custom Python simulations are different animals.

The mechanism

Data integrations
Ray can read and write common data sources, but lakehouse governance and SQL semantics may still live outside Ray.
Training integrations
Ray Train works with popular ML frameworks, often as launcher and coordinator rather than replacing the framework internals.
Serving integrations
Ray Serve composes Python logic and models; it can coexist with model registries, gateways, and specialized inference runtimes.
Custom integrations
When integrating a library, the pattern is usually: wrap its unit of work in tasks or actors, make resources explicit, and define artifact handoffs.

Decision surface

Trade-offHow to reason about it
Ray vs SparkRay fits Python-native ML and heterogeneous compute. Spark fits SQL-heavy, JVM-integrated, governance-rich batch analytics.
Ray vs DaskBoth scale Python. Ray's actor model and ML libraries are a major distinction; Dask can be more natural for NumPy/Pandas-style dataframes.
Ray Serve vs dedicated inferenceServe is programmable and compositional. Dedicated engines can be faster or simpler for one model family and one serving path.

Operational checklist

  • List incumbent systems and their ownership before introducing Ray.
  • Choose one initial workflow where Ray removes clear glue or unlocks scale.
  • Keep interfaces boring: files, checkpoints, metrics, HTTP/gRPC, and explicit configs.
  • Document what Ray does not own so platform boundaries stay legible.

Failure modes

  • Trying to replace a data warehouse, orchestrator, model registry, and serving platform in one migration.
  • Building custom integrations that hide resource usage from the Ray scheduler.
  • Forgetting that organizational ownership can dominate technical elegance.

Implementation sketch

external_orchestrator

  -> Ray Train writes checkpoint
  -> registry records artifact

Exercise
Pick one platform you know. Draw which system owns orchestration, storage, compute, training, experiment tracking, registry, serving, monitoring, and governance. Where would Ray make the diagram simpler? Where would it make ownership blurrier?
