Ray Data: Blocks, Streaming Execution, and ML Dataflow

Ray Data gives the rest of the Ray ML stack a common way to load, transform, repartition, stream, and hand data to batch inference or training.

Source map
Book coverage: Chapter 6, 'Data Processing with Ray' - Ray Datasets, transformations, streaming-style execution, external integrations with pandas/Dask/Spark-shaped data, and ML dataflow.

The thesis

Ray Data is the bridge between file-oriented data systems and Python ML workloads: it turns data into distributed blocks that can feed transforms, training, batch inference, and serving-adjacent workflows.

Linear path

  1. Load data into blocks. Blocks are the physical units of distributed execution, locality, and backpressure.
  2. Apply maps, filters, repartitions, group-bys, and streaming execution. Each transformation changes the execution graph and often the shuffle cost.
  3. Hand data to ML consumers. The value is not just processing data; it is connecting preprocessing, training, and batch inference without changing substrate.

The mechanism

Dataset blocks
Datasets are partitioned into blocks. Block size controls memory behavior, scheduling overhead, and parallelism.
Lazy-ish execution
Operations build an execution plan that Ray can optimize and execute across workers.
Streaming execution
Streaming blocks through operators can reduce memory pressure and overlap preprocessing with downstream consumers. Shuffle boundaries still force materialization.
External interop
Ray Data can interoperate with file formats, Python functions, pandas-style tables, Dask/Spark-shaped handoffs, tensor batches, and ML framework iterators.

Decision surface

Trade-offHow to reason about it
Ray Data vs SparkRay Data fits Python-native ML dataflow and mixed CPU/GPU workloads. Spark remains stronger for mature SQL-heavy enterprise ETL.
Materialize vs streaming executionBatch materialization is easier to inspect. Streaming execution can reduce memory and latency but is harder to debug.
Map batches vs row functionsBatch functions are faster and vectorizable. Row functions are simpler but often too slow at scale.

Operational checklist

  • Pick block sizes deliberately; default behavior may not match your data distribution.
  • Use batch transforms for vectorized preprocessing and model inference.
  • Track shuffle stages separately from maps; shuffles usually dominate cost.
  • Treat external integrations as boundary contracts: schema, partitioning, ordering, and ownership must still be explicit.
  • Preserve schema and provenance when data will feed training or evaluation.

Failure modes

  • Using Ray Data as a drop-in SQL warehouse replacement.
  • Mapping one Python call per row over millions of tiny rows.
  • Materializing every intermediate dataset when streaming execution would be enough.

Implementation sketch

import ray

ds = ray.data.read_parquet("s3://bucket/events/")

def featurize(batch):
    batch["text_len"] = [len(x) for x in batch["text"]]
    return batch

features = (
    ds.filter(lambda row: row["label"] is not None)
      .map_batches(featurize)
      .repartition(128)
)
