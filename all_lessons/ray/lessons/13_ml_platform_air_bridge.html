Part III - Cluster and platform layer

The ML Platform Bridge

The book's Ray AIR chapter is best read today as a platform lesson: compose Data, Train, Tune, Serve, checkpoints, and offline batch inference around one runtime.

Source map
Book coverage: Chapter 10, 'Getting Started with the Ray AI Runtime' - AIR concepts, Datasets, Preprocessors, Trainers, Tuners, Checkpoints, offline batch prediction, Deployments, execution, memory, failure, and autoscaling.

The thesis

The durable idea behind AIR is not a brand name. It is the handoff discipline between ML workflow stages: data becomes features, features feed training, training emits checkpoints, checkpoints feed batch or online inference, and every handoff must be reproducible.

Linear path

  1. Name every artifact in the workflow: dataset version, preprocessor state, training config, model checkpoint, evaluation report, deployment config.
  2. Use Ray libraries for the stage where they remove real distributed machinery: Data for preprocessing, Train for workers, Tune for trials, Serve for online inference.
  3. Design the handoffs. A platform fails when stages individually work but their contracts drift: schema drift, checkpoint incompatibility, preprocessing mismatch, or deployment using the wrong artifact.

The mechanism

Preprocessors
Preprocessing state is part of the model contract. Scaling feature transforms is not enough if the serving path cannot reproduce them.
Trainers and tuners
Training creates checkpoints; tuning creates many training jobs and selects among them. The artifact lineage matters as much as the best metric.
Offline batch inference
Batch inference is a Ray Data job with model work inside map_batches, not an online service with a bigger timeout.
Failure model
Every stage needs a recovery boundary. Dataset materialization, checkpoints, and deployment versions are the natural boundaries.

Decision surface

Trade-offHow to reason about it
Integrated runtime vs existing platformRay can simplify a Python-heavy ML platform. If your org already has mature feature, training, and serving systems, integration boundaries may matter more than unification.
Materialize vs streamMaterializing artifacts improves reproducibility and debugging. Streaming stages reduces latency and storage but makes replay harder.
One workflow API vs explicit stagesA unified API reduces glue. Explicit stage boundaries make ownership, audit, and rollback clearer.

Operational checklist

  • Record dataset, code, config, environment, checkpoint, and metric lineage together.
  • Make preprocessing state portable from training to batch and online inference.
  • Use durable storage for checkpoints and intermediate artifacts that define correctness.
  • Treat autoscaling as part of the platform contract: CPU preprocessing and GPU training scale differently.

Failure modes

  • Training with one preprocessing path and serving with another.
  • Optimizing workflow speed while making the winning model impossible to reproduce.
  • Assuming a checkpoint alone is enough to deploy a model.

Implementation sketch

# The conceptual platform flow:
raw_data
  -> Ray Data transforms
  -> fitted preprocessor + training dataset
  -> Ray Train checkpoint
  -> Tune-selected best checkpoint
  -> Ray Data map_batches for offline inference
  -> or Ray Serve deployment for online inference
