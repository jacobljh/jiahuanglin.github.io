Part III - Cluster and platform layer

Capstone: Design a Ray ML Platform

Put the series together: one pipeline that ingests data, tunes and trains models, serves them online, and runs on a cluster without hiding the hard trade-offs.

Source map
Book synthesis: Chapters 1-11 as one end-to-end design exercise.

The thesis

A good Ray platform is not a pile of Ray APIs. It is a set of explicit contracts: workload shape, resource model, artifact lineage, failure boundaries, deployment path, and cost controls.

Linear path

  1. Start with one product workflow. Example: ingest customer events, train a ranking model nightly, tune hyperparameters weekly, serve online recommendations, and run batch backfills.
  2. Assign Ray layers to responsibilities. Core for custom parallel work, Data for transforms, Train for distributed training, Tune for search, Serve for online inference, KubeRay for cluster lifecycle.
  3. Add the operational frame: observability, autoscaling, versioned artifacts, rollback, quotas, security, and a clear answer for what happens when any node, task, actor, or deployment fails.

The mechanism

Workload taxonomy
Separate CPU data transforms, GPU training, GPU/CPU serving, tuning trials, and evaluation jobs. They do not scale the same way.
Artifact contracts
Datasets, preprocessors, checkpoints, metrics, model cards, and deployment configs must be named and versioned.
Resource model
Node groups, placement, GPUs, object-store memory, and autoscaling limits define the platform's physical reality.
Control plane
Jobs, dashboards, logs, metrics, and deployment promotion define how humans operate the platform under pressure.

Decision surface

Trade-offHow to reason about it
Central platform vs team autonomyA central platform standardizes cluster, observability, and governance. Team autonomy accelerates experiments but can fragment patterns.
Shared Ray cluster vs workload-specific clustersShared clusters improve utilization. Workload-specific clusters make dependencies, blast radius, and cost attribution cleaner.
Fast iteration vs reproducibilityNotebook-to-cluster flow is attractive. Production needs pinned dependencies, artifact versions, and replayable jobs.

Operational checklist

  • Write the critical path for one request or one training example end to end.
  • Define SLOs for jobs and serving separately: completion time, latency, throughput, success rate, and recovery time.
  • Set quotas and autoscaling bounds before teams discover infinite parallelism.
  • Run a failure drill: kill a worker, kill an actor, lose a node, corrupt an input file, roll back a deployment.

Failure modes

  • Making Ray the answer before the platform question is precise.
  • Optimizing happy-path speed without designing lineage and rollback.
  • Treating cost as a dashboard afterthought rather than a scheduling and architecture constraint.

Implementation sketch

product_events
  -> Ray Data: clean, join, featurize
  -> Ray Train: distributed training job
  -> Ray Tune: scheduled sweep for selected knobs
  -> artifact registry: checkpoint + preprocessor + metrics
  -> Ray Serve: online deployment
  -> monitoring: latency, accuracy proxy, drift, cost
Exercise
Design the first production version with only three Ray components. Which three do you choose, and which responsibilities do you intentionally leave to existing systems? The restraint is the point.
