Part II - Applications and libraries

RLlib: Scalable Reinforcement Learning

RLlib packages the rollout-worker, learner, policy, evaluation, multi-agent, and offline-data machinery you just built by hand.

Source map
Book coverage: Chapter 4, 'Reinforcement Learning with Ray RLlib' - Gym environments, CLI/Python API, experiment configuration, multi-agent, policy clients, curriculum, and offline data.

The thesis

RLlib is valuable when the hard part should be the environment and learning objective, not rebuilding distributed rollout collection every time.

Linear path

  1. Define the environment contract. Observations, actions, rewards, resets, and termination are the interface between the world and the learner.
  2. Choose the algorithm and resource topology. PPO, DQN, IMPALA-style architectures, multi-agent policies, and offline data each imply different data movement.
  3. Tune the experiment as a distributed system. Rollout workers, evaluation workers, learners, GPUs, batch sizes, and checkpoint cadence are coupled knobs.

The mechanism

Environment
A Gymnasium-style environment provides the simulation boundary. The better this contract, the less custom distributed code you need.
Rollout workers
Workers sample trajectories in parallel. Their count controls sample throughput and cluster pressure.
Policy and learner
Policies choose actions; learners update policy parameters. Scaling them together is the real systems question.
Advanced modes
Multi-agent, policy server/client patterns, curriculum learning, and offline data each change the feedback loop shape.

Decision surface

Trade-offHow to reason about it
Library leverage vs abstraction costRLlib saves enormous infrastructure effort, but debugging may require learning its internal execution plan.
Sample freshness vs throughputMore rollout workers increase samples per second but can make samples stale relative to the learner.
Online vs offline RLOnline rollouts adapt to the current policy. Offline data is cheaper and safer, but distribution shift becomes the central risk.

Operational checklist

  • Validate the environment with random actions before training.
  • Track environment steps per second, learner updates per second, reward, KL/divergence, and checkpoint quality.
  • Use Tune integration for systematic sweeps rather than hand-running one config at a time.
  • Treat multi-agent policy mapping as architecture, not configuration trivia.

Failure modes

  • Blaming the algorithm when the environment reward leaks, saturates, or rewards the wrong behavior.
  • Scaling rollout workers until the learner is overwhelmed.
  • Treating offline data as if it came from the current policy.

Implementation sketch

from ray.rllib.algorithms.ppo import PPOConfig

config = (
    PPOConfig()
    .environment("CartPole-v1")
    .env_runners(num_env_runners=4)
    .training(train_batch_size_per_learner=2000)
)

algo = config.build_algo()
for _ in range(10):
    print(algo.train()["env_runners"]["episode_return_mean"])
Exercise
Suppose reward improves slowly but GPU utilization is 15 percent. Name three possible causes: environment sampling, learner batch size, policy transfer, or evaluation frequency. Which metric would distinguish them?
