Part I - Core mental model

Why Ray Exists

The problem Ray attacks: Python is pleasant locally, but most real ML and simulation workloads eventually need distributed execution.

Source map
Book coverage: Chapter 1, 'An Overview of Ray' - motivation, design principles, Core/libraries/ecosystem, and the data-science workflow.

The thesis

Ray exists because the gap between a useful Python prototype and a scalable distributed application is too wide. Its design tries to preserve the shape of Python while exposing just enough scheduling and resource control to run on a cluster.

Linear path

  1. Start with the uncomfortable fact: much data-science and ML code begins as ordinary Python, but ordinary Python does not naturally use many machines.
  2. Notice the recurring workload shapes: embarrassingly parallel maps, simulations, model training, hyperparameter search, data transforms, and online model serving.
  3. Ray's bet is that a small core runtime plus specialized libraries can cover these shapes without forcing every practitioner to become a distributed-systems engineer first.

The mechanism

Small primitive set
Tasks, actors, and objects are deliberately few. The API surface is small because the composition space is large.
Layered stack
Ray Core handles distributed execution. Libraries like Data, Train, Tune, Serve, and RLlib provide workflow-specific control loops.
Resource-aware scheduling
Remote work can declare CPUs, GPUs, memory-like custom resources, and placement constraints. This is the bridge between Python code and cluster topology.
Ecosystem stance
Ray does not require one ML framework. It tries to sit underneath PyTorch, TensorFlow, XGBoost, scikit-learn, Hugging Face, and custom Python logic.

Decision surface

Trade-offHow to reason about it
Accessibility vs sharp edgesThe API is approachable, but distributed execution still introduces nondeterminism, retries, and hidden data movement.
Python-first vs language-neutralPython-first design fits ML teams well. It is less natural for polyglot backend systems where JVM, Go, or Rust services dominate.
Flexible libraries vs one blessed workflowRay supports many workflow shapes. That flexibility means teams must still choose architecture rather than receive one complete platform by default.

Operational checklist

  • Reach for Ray when you have many independent or semi-independent Python units that need CPUs, GPUs, or both.
  • Prefer a higher-level library when the workload is a standard ML workflow: distributed dataset, training job, tuning run, serving app, or RL experiment.
  • Prefer a specialized system when the problem is already solved by a mature engine with stronger domain guarantees, such as SQL warehouses for BI or stream processors for exactly-once event pipelines.
  • Write down the expected bottleneck before scaling: CPU, GPU, memory, object-store capacity, network, scheduler overhead, or external service limits.

Failure modes

  • Confusing distributed with faster. Small tasks can become slower because scheduler overhead dominates useful work.
  • Assuming the library layer removes the need to understand Core. When performance breaks, the explanation is usually in tasks, actors, objects, and resources.
  • Adopting Ray as a platform decision before proving that the workload is actually Python-heavy and parallelizable.

Implementation sketch

# Local Python shape:
results = [simulate(seed) for seed in seeds]

# Ray shape:
simulate_remote = ray.remote(simulate)
refs = [simulate_remote.remote(seed) for seed in seeds]
results = ray.get(refs)
Exercise
Classify five workloads you know: batch transform, training job, online service, search/tuning loop, and simulation. For each, ask whether the unit of work is stateless, stateful, streaming, GPU-bound, or coordination-heavy. That classification predicts which Ray layer you should start with.
