Part II - Applications and libraries

Ray Serve: Online Inference

Serve turns Ray actors into scalable deployments with HTTP ingress, handles, batching, composition, and independent autoscaling.

Source map
Book coverage: Chapter 8, 'Online Inference with Ray Serve' - online inference characteristics, Serve architecture, endpoints, scaling, batching, multimodel graphs, and an NLP API example.

The thesis

Online inference is a latency contract, not just a prediction function. Ray Serve gives Python-native model composition while making replica count, batching, and routing explicit design knobs.

Linear path

  1. Separate the model from the application. Real APIs include preprocessing, retrieval, business rules, validation, and one or more models.
  2. Deploy each scalable unit as a Serve deployment. A deployment can own model state, request batching, resource needs, and autoscaling policy.
  3. Compose deployments with handles or an ingress app so that the user-facing API remains one service while internal stages scale independently.

The mechanism

Deployment
A deployment is a replicated, stateful callable, often backed by a Ray actor per replica.
Ingress and handles
HTTP ingress receives requests; deployment handles let Python code call other deployments directly.
Batching
Dynamic batching trades a little queueing latency for higher throughput, especially for GPU-bound models.
Autoscaling
Serve can scale deployments independently based on request pressure and configured policies.

Decision surface

Trade-offHow to reason about it
Serve vs framework-specific servingServe is excellent for Python composition and heterogeneous logic. Triton, vLLM, or TensorRT-LLM may be better for highly optimized single-model inference kernels.
Batching vs latencyBatching increases throughput but can hurt p50 or p99 latency if queueing is too aggressive.
One deployment vs graphOne deployment is simple. A graph lets each stage scale independently but makes tracing and failure handling more important.

Operational checklist

  • Define the SLO: p50, p95, p99, throughput, queue time, model time, and error budget.
  • Batch only where the model benefits; do not batch cheap Python glue just because you can.
  • Use separate deployments for stages with different resource profiles.
  • Trace request paths across deployments; user latency is the sum of every internal hop.

Failure modes

  • Treating Serve as a model registry or full ML platform. It focuses on serving primitives and composition.
  • Setting replicas high enough for throughput but forgetting model memory per replica.
  • Hiding slow preprocessing inside the model deployment and then scaling expensive GPUs to fix a CPU bottleneck.

Implementation sketch

from ray import serve
from starlette.requests import Request

@serve.deployment(num_replicas=2, ray_actor_options={"num_gpus": 1})
class Classifier:
    def __init__(self):
        self.model = load_model()

    async def __call__(self, request: Request):
        payload = await request.json()
        label = await self.classify(payload["text"])
        return {"label": label}

    @serve.batch(max_batch_size=16, batch_wait_timeout_s=0.01)
    async def classify(self, texts):
        return self.model(texts)

app = Classifier.bind()
serve.run(app)
