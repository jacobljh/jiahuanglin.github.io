Instance groups & concurrency

Multiple copies of a model on each GPU. The knob is one config block; the question is where the diminishing-returns knee sits. "More instances" buys throughput only until they all start fighting for the same tensor cores.

Why more than one instance per GPU

A single model instance with a single CUDA stream serialises every inference. While the GPU is mid-kernel for batch n, the CPU-side dispatch of batch n+1 waits. Two instances on the same GPU each get their own CUDA stream, so dispatch of one batch overlaps with compute of the other. Concretely:

  • Hide host-side dispatch latency. The ~50–200µs of CPU-side prep per batch is parallelised across instances.
  • Overlap memcpy with compute. One instance copies inputs to the GPU while the other runs kernels.
  • Saturate small models. A tiny model can't fill a GPU at any single batch size; running 4 of them in parallel can.

The config block

instance_group [
  {
    kind: KIND_GPU       # KIND_GPU | KIND_CPU | KIND_MODEL
    count: 2              # 2 instances of this model
    gpus: [ 0, 1 ]        # placed on GPU 0 and GPU 1 (2 instances total, one per device)
  }
]

Some variations:

PatternMeaning
count: 4, gpus: [0]4 instances all on GPU 0.
count: 2, gpus: [0, 1]2 instances on GPU 0 and 2 on GPU 1 (count is per-GPU when gpus lists multiple).
kind: KIND_CPU, count: 88 CPU instances (no GPU); useful for tokenizers, postprocessors.
kind: KIND_MODELLet the model file decide placement (used by TensorRT engines that bind GPUs at build time).
Multiple instance_group blocksHeterogeneous placement, e.g. 1 instance on a fast H100 + 1 on a cheaper L40 with different priority.

The diminishing-returns curve

Two regimes:

  • Steep climb (1 → ~3 instances). You're filling unused host-side overhead and stream concurrency. Each new instance buys a real fraction of the model's peak throughput.
  • Plateau (4+). All instances now compete for tensor cores. Adding another just makes the queue longer per instance; total throughput barely moves, and per-request latency rises.

Where the knee sits depends on the model — fast (small) models plateau later because they're easier to keep busy in parallel; big models plateau at 2.

The interaction with max_batch

An instance running at max_batch=64 uses 4× the GPU per call as one at max_batch=16. So:

1 instance × b=64
≈ same GPU load
4 instances × b=16
≈ same GPU load

They differ in shape: 4×16 hides dispatch latency better and reaches the same throughput at lower queue delay; 1×64 has lower per-batch overhead. The rule of thumb:

Sizing rule
Pick max_batch first (the value TensorRT was built for, or the largest your latency SLO allows). Then add instances until throughput stops growing. Two instances is almost always better than one; four is a common sweet spot; beyond that, profile (lesson 09).

Rate limiting & priorities — keeping models from starving each other

One Triton process serves many models. A bursty model can starve the rest. Two mechanisms:

# Cap the requests in flight for this model on this device
instance_group [{ kind: KIND_GPU count: 2 rate_limiter { resources [{ name: "gpu0" count: 1 }] } }]

# Multi-priority queue
dynamic_batching {
  priority_levels: 3
  default_priority_level: 2
}

Clients then attach priority to each request; the scheduler dispatches the highest-priority queue first.

Multi-GPU placement

With gpus: [0, 1, 2, 3] and count: 2 you get 8 total instances (2 per device). Triton round-robins request dispatch across them. For asymmetric devices (e.g. an H100 + an L40), use multiple instance_group blocks to assign different counts per device.

Note: an instance is tied to a single GPU. For tensor parallelism across GPUs, the underlying backend has to support it (TensorRT-LLM does — see lesson 10); Triton doesn't do TP itself.

The starting point — and where to go from there

  1. Default: count: 2 per GPU. Cheap throughput win over the default count=1.
  2. Run Perf Analyzer with --concurrency-range 1:32:1 (lesson 09). Watch throughput vs concurrency.
  3. If throughput is still climbing at concurrency=32, increase count to 3 and rerun.
  4. If p99 latency is the constraint, the throughput-vs-latency Pareto from Model Analyzer (lesson 09) tells you the right count.
