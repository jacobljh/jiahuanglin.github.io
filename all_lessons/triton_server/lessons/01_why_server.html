Why a model server?

A FastAPI loop in front of model(x) looks fine until you put load on it. This lesson is the case for a dedicated inference server: per-request inference wastes the GPU; the throughput is in cross-request batching. Five jobs a real server has to do, and the structural reason a script won't do them.

The naive approach — and where it falls over

app = FastAPI()
model = load_model("model.pt").cuda().eval()

@app.post("/infer")
async def infer(req: Request):
    x = torch.tensor(req.json()["x"]).cuda()
    with torch.no_grad():
        y = model(x)
    return {"y": y.tolist()}

This serves one request at a time. Each call:

  • copies one item to the GPU,
  • runs the model on a batch of size 1,
  • copies the result back.

The GPU's tensor cores want batches of 32–256 to be busy. At batch size 1, you're typically using 1–5% of the device's flops. The GPU spends most of its time idle waiting for the next request to arrive.

The structural fix — batch across requests

That's the single biggest reason a dedicated server exists: batching across concurrent users. The "wait briefly to grow the batch" decision is invisible to clients but multiplies GPU throughput 2–10×.

The five jobs a real model server has to do

JobWhat it meansWhy "just Flask" can't
1 · Batch across requests Coalesce in-flight requests into one inference call; wait a bounded time for the queue to fill. Flask hands one request to one worker. There's no scheduler to see the next request before this one returns.
2 · Concurrency on the GPU Run multiple batched inferences in flight simultaneously, on the same device. PyTorch with a single CUDA stream serialises kernel launches; you need explicit streams and pinning.
3 · Multiple model formats One process serves a TensorRT engine, a PyTorch checkpoint, and an ONNX model side by side. Each runtime has its own loader, threading model, dtype quirks. Hand-rolling means writing all three.
4 · Versioning & hot-reload Drop a v2 of a model in; the server picks it up, optionally serves both versions, retires v1. "Restart the service to update" loses requests; rolling deploys are a whole infrastructure layer.
5 · Observability Per-model metrics: queue depth, batch size distribution, p50/p99 latency, GPU utilisation. Flask gives you request count. Everything else you instrument from scratch.

Each is non-trivial to build well; together they're a multi-month engineering project. Triton is the project, productionised by NVIDIA.

The throughput cliff, made concrete

Take a ResNet-50 inference at fp16 on an A100 (peak ~624 TFLOP/s). One image at 224×224 is ~8 GFLOP. So in the absence of overhead:

batch=1, peak
~78K img/s
batch=1, real (PyTorch)
~1.5K img/s
batch=64, real
~22K img/s
batch=64 / batch=1
~15× more throughput

Single-request inference is 2% of what the GPU could do. Batching to 64 climbs to ~30%. To get more you need TensorRT (with proper kernel fusion), instance groups (multiple in flight on the same GPU), and CUDA Graphs. Triton handles all three.

Two architectural insights that fall out

  1. The server is a scheduler, not just a request handler. The scheduler is where throughput is made or lost. Lesson 06 is the lesson on this.
  2. The model is data, not code. The server doesn't know what your model does; it knows its inputs, outputs, and how to call its backend. That's the declarative interface of lesson 03.

When you actually shouldn't use Triton

  • You serve one model, it's an LLM, and you want max throughput. Reach for vLLM or SGLang first — they specialise harder on the LLM scheduler (paged KV, prefix sharing, in-flight batching). Triton + TRT-LLM matches or beats them for production deploys, but only after sizing; vLLM is simpler day-one. (Lesson 11.)
  • You have one Python model and you'll always have one. A custom FastAPI server is faster to build and easier to debug. Triton's value scales with the number of models & backends in the same process.
  • You need to serve from a CPU laptop. Triton runs on CPU but loses most of its value (no GPU concurrency, no TensorRT). ONNX Runtime in-process is lighter.

Naive vs batched throughput

Toy model: each inference at batch B takes 5 + 0.5·B ms. Naive serialises; batched gathers up to max_batch within the queue delay.
