Orientation — the map before the server

Triton Inference Server is a process that loads model files from a directory and exposes them over HTTP/gRPC, with dynamic batching and multiple backends. One idea defines it: the model repository is the API. Drop a model in, restart-free, and Triton serves it. This page is the map.

Wait — which Triton?
Two unrelated projects share the name. This series is NVIDIA Triton Inference Server, a model-serving runtime. The other one — OpenAI Triton — is a Python DSL for writing GPU kernels (@triton.jit, tl.dot). That gets its own track at all_lessons/triton_kernels. They share zero code; only the name.

What Triton Inference Server actually is

You have a model — a TensorRT engine, a PyTorch checkpoint, an ONNX file, whatever. You want to expose it as a service. You could write Flask. Don't. Triton does the same job, plus:

  • Dynamic batching — coalesces multiple in-flight requests into one inference call. Throughput multiplier on the GPU.
  • Multiple backends — TensorRT, PyTorch, ONNX Runtime, Python, vLLM, TensorRT-LLM. One server, many model formats.
  • Concurrency per GPU — multiple "instances" of a model on each device so the server can answer many requests at once.
  • Model versioning & hot-reload — drop a new version in the repository and Triton picks it up. No restart.
  • HTTP, gRPC, and a shared-memory C-API — pick your transport. gRPC is the high-throughput default.

The defining idea — declarative, not imperative

The defining sentence
In a custom Flask server, you write Python that loads a model and routes requests. In Triton, you put a model and a config.pbtxt in a directory. The server is the routing, batching, and scheduling — written once by NVIDIA, used by every model. You make changes by editing config files, not code.

That declarative interface is what makes Triton scale across teams: an MLE drops a model and a config; the platform team handles the server. No new Python per model.

The minimal glossary (for the first three lessons)

  • Model repository — a directory laid out as repository/<model_name>/<version>/<model_file> plus repository/<model_name>/config.pbtxt. The server scans it at startup and on demand.
  • Backend — the runtime that actually executes a model (TensorRT, PyTorch LibTorch, ONNX Runtime, Python, vLLM, TRT-LLM). Triton ships several; you can write your own.
  • Instance — a loaded copy of a model on a specific GPU. Multiple instances per model = concurrent inference on the same device.
  • Dynamic batching — the default scheduler. Waits up to max_queue_delay_microseconds to coalesce requests into a batch.
  • config.pbtxt — a Protocol-Buffers text file that declares everything Triton needs about a model: inputs, outputs, dtypes, batch dim, backend, instance groups, batching mode, sequence config, …
  • Ensemble — a virtual "model" that's actually a DAG of other models (preprocess → encoder → postprocess). Lives inside the server.
  • BLS — Business Logic Scripting. A Python-backend script that calls other models programmatically (branches, loops, async).

Everything else is defined as you meet it. Onward.
