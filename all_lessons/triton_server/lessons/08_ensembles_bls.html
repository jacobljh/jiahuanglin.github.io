Ensembles & Business Logic Scripting

Two ways to chain models inside the server. Ensembles: a declarative DAG of models that runs without a client round-trip. BLS: a Python backend that calls other models programmatically, with branches and loops. Pick the right one by the shape of your pipeline.

The problem — multi-step inference

Real serving paths are rarely a single model call:

  • Vision: resize + normalise → ResNet → label lookup.
  • NLP: tokenise → encoder → classifier head → label decoder.
  • RAG: classify intent → retrieve → rerank → generate.

The naive shape — run each step as its own model call from the client — costs you a network round-trip per step. For three steps at 5 ms each, that's 15 ms of network on top of compute, and the tensors leave the server box twice (decode → encode → decode → encode → …).

Triton ships two ways to run all of it inside the server.

Ensembles — declarative DAG

An ensemble is a "model" whose config.pbtxt has platform: "ensemble" and an ensemble_scheduling block describing a static graph of steps. No code; the steps are wired together by name.

name: "vision_pipeline"
platform: "ensemble"
max_batch_size: 16

input  [ { name: "raw"   data_type: TYPE_UINT8 dims: [ -1 ] } ]
output [ { name: "label" data_type: TYPE_STRING dims: [ 1 ]  } ]

ensemble_scheduling {
  step [
    { model_name: "preprocess"   model_version: -1
      input_map  { key: "raw" value: "raw" }
      output_map { key: "x"   value: "prep_x" } },

    { model_name: "resnet50"     model_version: -1
      input_map  { key: "x"   value: "prep_x" }
      output_map { key: "y"   value: "logits" } },

    { model_name: "label_lookup" model_version: -1
      input_map  { key: "y"     value: "logits" }
      output_map { key: "label" value: "label" } }
  ]
}

The wiring rule: each step's input_map says "for the inner model's input named key, pass the tensor that the ensemble calls value". Each step's output_map registers an internal tensor name. Top-level input/output are the ensemble's external interface.

What ensembles are good atWhat they can't do
Linear pipelines (A → B → C)Branching (if intent==X then A else B)
Fan-out / fan-in DAGs (A → {B, C} → D)Loops (decode until EOS)
Tensor passing without copy when models are on the same deviceConditional execution based on data
Per-step scheduling — each step has its own queue/batcherCustom retry / error-handling logic
The defining property
An ensemble is a static DAG. The set of steps and the wiring is decided at config-load time; the graph cannot change at request time. If your pipeline branches on a model's output, you need BLS instead.

Per-step batching is preserved

Each step in an ensemble is a real Triton model — with its own dynamic_batching config, its own instance group, its own GPU placement. So a 32-wide ensemble request is presented to preprocess as a batch of 32, then resnet50 sees a batch of 32 (possibly merged with other ensemble requests at that step thanks to dynamic batching), then label_lookup sees its batch.

This is the killer feature: the steps cross-batch independently. If preprocess is slow at small batches but cheap at large ones, and many ensemble requests are in flight, Triton can fatten the preprocess batch even though no single client asked for that.

BLS — Business Logic Scripting

Some pipelines need code. Branching, looping, async fan-out, retries — none of these fit a static DAG. BLS is the escape hatch: a Python backend model whose execute can call other models in the server through a Python API.

import triton_python_backend_utils as pb_utils
import numpy as np

class TritonPythonModel:
    def execute(self, requests):
        responses = []
        for req in requests:
            query = pb_utils.get_input_tensor_by_name(req, "query").as_numpy()

            # 1. classify intent
            cls_req = pb_utils.InferenceRequest(
                model_name="classifier",
                requested_output_names=["label"],
                inputs=[pb_utils.Tensor("text", query)],
            )
            cls_resp = cls_req.exec()
            intent = cls_resp.output_tensors()[0].as_numpy()[0]

            # 2. branch
            if intent == b"shopping":
                retr = pb_utils.InferenceRequest(
                    model_name="ann_shopping",
                    requested_output_names=["docs"],
                    inputs=[pb_utils.Tensor("q", query)],
                ).exec()
            else:
                retr = pb_utils.InferenceRequest(
                    model_name="ann_general",
                    requested_output_names=["docs"],
                    inputs=[pb_utils.Tensor("q", query)],
                ).exec()
            docs = retr.output_tensors()[0].as_numpy()

            # 3. generate
            gen = pb_utils.InferenceRequest(
                model_name="generator",
                requested_output_names=["answer"],
                inputs=[pb_utils.Tensor("q", query),
                        pb_utils.Tensor("ctx", docs)],
            ).exec()

            responses.append(pb_utils.InferenceResponse(
                output_tensors=[gen.output_tensors()[0]]
            ))
        return responses

What you get: branches, loops, exception handling, async (async_exec() returns a future you can await — fan out to N models in parallel and merge their results). What you pay: the BLS model is a Python-backend process; each InferenceRequest.exec() call goes through Triton's IPC layer to the target model and back.

The cost difference, made concrete

ensemble step → step
~0 µs (in-process)
BLS inner call
~100–500 µs IPC
3-step pipeline ensemble
≈ sum(model times)
3-step BLS pipeline
≈ sum(model times) + 3 × IPC

For a pipeline of three models that each run in 5 ms, ensembles add roughly zero overhead; BLS adds ~1 ms of IPC. That's a 7% latency tax — not catastrophic, but real. If your pipeline is purely linear, take the ensemble.

Async BLS — fan out, then merge

def execute(self, requests):
    responses = []
    for req in requests:
        q = pb_utils.get_input_tensor_by_name(req, "q").as_numpy()
        # fire three retrievers in parallel
        futures = [
            pb_utils.InferenceRequest(model_name=f"retriever_{i}",
                requested_output_names=["docs"],
                inputs=[pb_utils.Tensor("q", q)]).async_exec()
            for i in range(3)
        ]
        # gather
        all_docs = np.concatenate([f.get_response().output_tensors()[0].as_numpy()
                                   for f in futures], axis=0)
        # rerank + answer
        ans = pb_utils.InferenceRequest(model_name="generator",
            requested_output_names=["a"],
            inputs=[pb_utils.Tensor("q", q),
                    pb_utils.Tensor("ctx", all_docs)]).exec()
        responses.append(pb_utils.InferenceResponse(
            output_tensors=[ans.output_tensors()[0]]))
    return responses

That's a pattern that an ensemble's DAG can express statically — but only with all three retrievers always called. If which retrievers you call is conditional, only BLS works.

Decision tree — which to use

Pipeline shapePickWhy
Linear: A → B → CEnsembleZero Python in the hot path, no IPC overhead, per-step batching for free.
Static fan-out: A → {B, C, D} → EEnsembleSame — the DAG handles parallel branches.
Conditional branch on a tensor valueBLSEnsembles can't read tensors at scheduling time.
Loop until a condition (e.g. decode tokens)BLS (or a specialised LLM backend, lesson 10)No loops in ensembles.
Retries / fallbacks on a failed inner callBLSError handling needs code.
Mix of static plumbing + small bit of logicBoth: BLS at the top calling an ensemble modelReuse the ensemble for its zero-overhead linear segment.
The composable trick
Ensembles and BLS aren't either-or. A BLS model can call an ensemble model (and vice versa). The common pattern: linear preprocessing is an ensemble; the branching logic is a BLS model whose first inner call is that ensemble. You pay IPC only once, at the boundary.

What you don't get from either

  • Cross-host pipelines. Both ensembles and BLS run inside one Triton process. To chain across machines, do it from the client.
  • Cross-GPU tensor avoidance. If step A is on GPU 0 and step B on GPU 1, the tensor copies between them. Pin pipeline-mates to the same device when latency matters.
  • Streaming partial outputs from an ensemble. Ensembles return once. For token-streaming or progressive results, use a backend that supports decoupled responses — enabled via model_transaction_policy { decoupled: true } in config.pbtxt, with the Python backend using InferenceResponseSender to emit multiple responses per request. The TRT-LLM and vLLM backends do this natively.
