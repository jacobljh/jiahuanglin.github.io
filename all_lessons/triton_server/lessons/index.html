Triton Inference Server — Multi-Backend Model Serving

A linearized tour of NVIDIA Triton Inference Server, the framework that turns a model file into a production HTTP/gRPC service with dynamic batching, multiple backends, and per-GPU concurrency. (Looking for OpenAI Triton, the kernel DSL? Different track.)

This series of eleven interactive lessons unwraps Triton Inference Server from scratch. Part I (lessons 01–02) covers the why and shape: what problem a model server solves that a Flask app doesn't, and the five components every request flows through. Part II (lessons 03–05) covers the declarative interface: the model repository, the backends (TensorRT, PyTorch, ONNX, vLLM, TRT-LLM, Python), and how instance groups scale one model across a GPU. Part III (lessons 06–07) covers batching: dynamic batching and sequence batching, the two scheduler modes that decide throughput. Part IV (lesson 08) covers composition: ensembles and Business Logic Scripting for multi-step pipelines. Part V (lessons 09–11) is production: Performance Analyzer / Model Analyzer for sizing, the TensorRT-LLM backend for LLM serving, and the Triton-vs-vLLM-vs-SGLang decision tree.

Who this is for
You've trained or loaded a model in PyTorch / TensorFlow / ONNX and now you have to expose it as an HTTP service that handles many concurrent requests, batches them efficiently, runs on a GPU, and survives a real load. You know what a request lifecycle is; you don't yet know why "just put PyTorch behind FastAPI" is the wrong answer at scale. By the end you can pick between Triton, vLLM, SGLang, or your own server with eyes open.

The system you're learning

A Triton instance is one process that loads many models from a directory, exposes them over HTTP/gRPC/C-API, and runs them on multiple GPUs with dynamic batching. The five components are the same regardless of which backend the model is written in.

Part I · The case for a model server (lessons 01–02 · why this exists and what's inside)

01
Why a model server?
What breaks when you put PyTorch behind FastAPI. Per-request inference is GPU-starved; batching across requests is where the throughput lives. The five jobs a real server has to do.
02
The architecture
Frontend → scheduler → backend → GPU, plus the model repository as the source of truth. The trace of one request, end to end, with a live latency-breakdown widget.

Part II · Declarative serving (lessons 03–05 · the repository, the backends, the knobs)

03
The model repository
A directory is the API. config.pbtxt declares inputs, outputs, dtypes, batch dimension, version policy. Hot-reload, A/B versions, model-control-mode. The "models as data" idea.
04
Backends — TRT, PyTorch, ONNX, Python, vLLM, TRT-LLM
One server, six runtimes. Each backend handles its own model format and execution. Which backend to pick: a decision matrix on latency, dtype support, dynamic shapes, and dev velocity.
05
Instance groups & concurrency
Multiple model instances per GPU. Why "more instances" is not "more throughput" past a knee. The right rule: instances × max_batch ≈ what the model can keep busy. Live throughput-vs-instances simulator.

Part III · Batching (lessons 06–07 · the scheduler decides throughput)

Two scheduler modes cover ~95% of production workloads. Dynamic batching is the throughput one. Sequence batching is the streaming/stateful one. Both are config-pbtxt knobs.

06
Dynamic batching
Triton waits a bounded time, coalesces compatible requests, fires one batched inference. The latency/throughput tradeoff in one knob (max_queue_delay_microseconds). Live queue simulator.
07
Sequence batching
Stateful models — recurrent, KV-cached, streaming chat — pin a sequence's requests to the same model instance and keep its state slot warm. Direct vs Oldest scheduling. Why this exists, what breaks without it.

Part IV · Composition (lesson 08 · multi-step pipelines)

08
Ensembles & Business Logic Scripting
Two ways to chain models inside the server. Ensembles: a declarative DAG (preprocess → model → postprocess), no client round-trips. BLS: a Python backend can call other models programmatically — branches, loops, retries. When each beats the other.

Part V · Production (lessons 09–11 · sizing, LLMs, the decision tree)

09
Perf Analyzer & Model Analyzer
Two tools that turn "how should I configure this?" into a number. Perf Analyzer measures latency/throughput at a load level; Model Analyzer sweeps configs (instance count, max_batch, delay) and produces a Pareto frontier. Reading their output.
10
Triton + TensorRT-LLM
The LLM-specific backend. Paged KV cache, in-flight batching, speculative decoding — all the things vLLM does, but as a Triton backend so you get the server stack (concurrency, dynamic batching across users, ensembles for RAG) on top of TRT-LLM's runtime.
11
Triton vs vLLM vs SGLang
Three frameworks, overlapping but not identical sweet spots. Triton: many models, many backends, integration with a wider stack. vLLM: one LLM, max throughput. SGLang: LLM programs, prefix sharing, constrained decoding. The matrix and the decision tree.

How to use this

  1. Linearly. Each lesson assumes the previous. Lesson 06 (dynamic batching) is where it all clicks; lessons 03–05 set up the language you need.
  2. Touch every knob. Each widget has a setting that makes the server slow or wrong. Find it.
  3. Spin up a real Triton. The official container is one docker run away. The lessons include the model-repository scaffolding so you can actually serve the example model.
