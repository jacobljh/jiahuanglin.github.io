Perf Analyzer & Model Analyzer

Two NVIDIA-built tools that turn "how should I configure this?" into a number. Perf Analyzer measures one config under load. Model Analyzer sweeps configs and gives you a Pareto frontier. Together they answer every sizing question lessons 05–07 raised.

Perf Analyzer — measure one config

perf_analyzer is a load generator and measurement tool. It opens many concurrent clients against a running Triton, sends inferences, and reports latency percentiles plus throughput.

# the canonical run
perf_analyzer \
  -m resnet50 \                                # model name
  --concurrency-range 1:64:1 \                  # sweep concurrency from 1 → 64 step 1
  -b 1 \                                        # client-side batch size per request
  --measurement-mode count_windows \            # stable measurement window
  --shared-memory cuda \                        # avoid HTTP serialisation cost
  --input-data random                           # random tensor in the configured dtype

Useful flags:

FlagJob
-m <name>Which model to drive.
--concurrency-range A:B:stepSweep simultaneous client connections.
-b / --batch-sizeClient-side batch per request. Different from the server's max_batch_size.
--shared-memory cuda|systemSkip HTTP/gRPC serialisation. Realistic for in-cluster clients.
--input-data <file|random|zero>Don't use zero for anything that compresses or quantises — it's unrealistically fast.
--latency-thresholdStop the sweep when p99 crosses this SLO.
--service-kind triton|tfserving|torchserveYes, Perf Analyzer can drive other servers too — useful for apples-to-apples comparisons.

Reading the output

Inferences/Second vs. Client Average Batch Latency
Concurrency: 1,  throughput: 215 infer/sec, latency 4623 usec
Concurrency: 2,  throughput: 401 infer/sec, latency 4983 usec
Concurrency: 4,  throughput: 742 infer/sec, latency 5384 usec
Concurrency: 8,  throughput: 1180 infer/sec, latency 6776 usec
Concurrency: 16, throughput: 1420 infer/sec, latency 11261 usec    ← saturation
Concurrency: 32, throughput: 1450 infer/sec, latency 22049 usec    ← past saturation
Concurrency: 64, throughput: 1455 infer/sec, latency 43984 usec

Two patterns to recognise:

  • Throughput climbs ~linearly while latency stays flat: you have GPU headroom — keep ramping concurrency.
  • Throughput plateaus, latency grows linearly with concurrency: you've hit the GPU. Per-request work is the same; you're just queueing. This is the saturation point. Pick the highest concurrency before latency leaves SLO.

Common pitfalls in Perf Analyzer runs

PitfallSymptomFix
Client on same host as serverThroughput suspiciously low; CPU pegged on client sideRun client on a separate host, or pin CPU affinity (taskset).
--input-data zero on a quantised modelThroughput looks 2× realUse random or a representative data file.
No warmupFirst concurrency reading is much slower--measurement-mode count_windows handles this; or set --warmup-request-count.

Model Analyzer — sweep configs

Perf Analyzer measures one config. Model Analyzer wraps it to sweep many — different instance counts, max_batch_size, dynamic_batching delays — runs Perf Analyzer on each, and produces a report.

model-analyzer profile \
  --model-repository /models \
  --profile-models resnet50 \
  --triton-launch-mode docker \
  --run-config-search-max-instance-count 4 \
  --run-config-search-max-concurrency 64 \
  --run-config-search-max-preferred-batch-size 64

It launches Triton (or attaches to a running one), enumerates configurations within the search space, runs each through Perf Analyzer, and writes results to a CSV plus a PDF/HTML report.

The Pareto frontier

Blue dots = configs that are Pareto-optimal (no other config has both higher throughput AND lower latency). Gray = dominated. To pick: draw a vertical line at your SLO and pick the highest-throughput blue point to the left of it.

What sweeps cost

A typical sweep — 4 instance counts × 6 batch sizes × 4 delay values = 96 configs — runs ~30 minutes on a single A100. Model Analyzer caches results so you don't repeat measurements. The output is the answer to "what config should I ship?"

Workflow
Don't tune by hand. Set the SLO, run Model Analyzer for an hour, pick from the Pareto report. The instinct to guess "8 instances × max_batch=64" is almost always wrong — usually by 2× in either direction.

Production rituals

  • Re-run Model Analyzer on every model update. A new model size or a new TensorRT engine can shift the Pareto frontier dramatically.
  • Run weekly in CI against a "golden" config. Catches regressions where a Triton or TRT upgrade drops throughput.
  • Use realistic input data. Save real production samples to a JSON file and feed via --input-data <file>. Zero/random tensors lie.
