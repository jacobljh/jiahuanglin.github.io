The model repository

A directory is the API. config.pbtxt declares everything; the file layout is what Triton scans. There is no Python file you have to write to register a model — the server's behaviour for your model is fully determined by what is in this tree.

The directory layout

Triton scans a top-level model repository directory. Every subdirectory is one model. Inside each model lives a config.pbtxt and one or more numeric version directories containing the actual model file:

model_repo/
├── resnet50/
│   ├── config.pbtxt              ← schema · backend · batching · instances
│   ├── 1/
│   │   └── model.plan            ← TensorRT engine (filename matters)
│   └── 2/
│       └── model.plan            ← v2, hot-dropped in later
│
├── bert_base/
│   ├── config.pbtxt
│   └── 1/
│       └── model.onnx            ← ONNX file
│
└── preprocess/
    ├── config.pbtxt
    └── 1/
        └── model.py              ← Python backend script

Three rules that catch everyone the first time:

  • Version directories are positive integers — 1, 2, 3… not v1, not latest. Triton sorts them numerically.
  • The model filename is fixed per backend. TensorRT wants model.plan; ONNX wants model.onnx; PyTorch wants model.pt; Python wants model.py. You can override with default_model_filename in the config, but the defaults exist for a reason — stick to them.
  • The model name in the URL is the directory name. A request to /v2/models/resnet50/infer looks up the resnet50/ directory. The name: field in config.pbtxt is informational and must match.

Anatomy of a config.pbtxt

This is real Protocol Buffers text format (the schema is ModelConfig in tritonserver's repo). A typical TensorRT vision model:

name: "resnet50"
backend: "tensorrt"
max_batch_size: 64

input [
  {
    name: "input__0"
    data_type: TYPE_FP16
    dims: [ 3, 224, 224 ]
  }
]

output [
  {
    name: "output__0"
    data_type: TYPE_FP16
    dims: [ 1000 ]
  }
]

instance_group [
  { kind: KIND_GPU  count: 2  gpus: [ 0 ] }
]

version_policy: { latest { num_versions: 1 } }

dynamic_batching {
  preferred_batch_size: [ 16, 32, 64 ]
  max_queue_delay_microseconds: 5000
}

Each field has a job:

FieldWhat it tells Triton
nameSanity-check string. Must equal the directory name. Lessons 09+ also use it for metrics labels.
backendWhich runtime loads this model. Modern field. (See next row for the legacy alternative.)
platformLegacy field that picked the backend implicitly (tensorrt_plan, onnxruntime_onnx, pytorch_libtorch). Some configs still use it; new ones should prefer backend.
max_batch_sizeThe implicit leading batch dimension. If > 0, every input gets an extra leading [batch] axis automatically and Triton may batch up to this size. If 0, you declare the full shape yourself and Triton will not batch.
input[] / output[]The contract. Names, dtypes, shapes. Triton enforces this at the frontend.
instance_groupHow many copies on which device. Lesson 05.
version_policyWhich version directories to actually load.
dynamic_batchingThe scheduler config. Lesson 06.

Data types — the dtype enum

Triton uses an explicit dtype enum on every input and output. The common values:

Triton typeMaps toUsed for
TYPE_FP32float32Default training dtype; safe for any backend.
TYPE_FP16float16Vision inference on tensor cores; TRT default.
TYPE_BF16bfloat16LLMs on Hopper / Ampere; TRT-LLM and vLLM workloads.
TYPE_INT8int8Quantised vision models; needs calibration.
TYPE_INT32 / TYPE_INT64intToken ids, indices, masks.
TYPE_STRINGbytesRaw text inputs / outputs. Each element is a length-prefixed UTF-8 string.
TYPE_BOOLbool (1 byte on the wire)Attention masks, padding flags. Distinct from TYPE_UINT8.
What max_batch_size = 0 actually means
With max_batch_size: 0, Triton treats your dims as the full tensor shape. There is no implicit leading batch axis, and the dynamic batcher cannot stack requests for you. Use this for models where the "batch" is already inside the input shape (sequence-batched LLMs that handle their own batching, or models with a non-batch leading axis).

Variable shape dimensions

Use -1 for any dimension whose size you only know at request time. The classic case is image preprocessing with arbitrary HW:

input [
  {
    name: "image"
    data_type: TYPE_UINT8
    dims: [ 3, -1, -1 ]        # channels fixed, height & width variable
  }
]

Two things to know:

  • Not every backend supports it. TensorRT engines were built with a specific set of input shapes (or shape profiles). ONNX and PyTorch usually do. The Python backend always does.
  • Dynamic batching still requires per-axis compatibility. Two requests with different H × W cannot be stacked into one batch — they have to be padded by the client or routed to separate ragged-batching backends (lesson 07).

Version policy

You can have multiple numeric version directories. version_policy decides which Triton actually loads and exposes:

# Default — only the highest-numbered version
version_policy: { latest { num_versions: 1 } }

# Keep the two newest, for canary / shadow comparisons
version_policy: { latest { num_versions: 2 } }

# Load every version directory present
version_policy: { all { } }

# Load specific versions
version_policy: { specific { versions: [ 1, 3 ] } }

A client requesting POST /v2/models/resnet50/versions/2/infer will reach v2 only if v2 is loaded. The default latest { num_versions: 1 } always serves the newest you have dropped in.

Model control modes — how the repository is refreshed

Triton has three modes that determine when it picks up changes to the repository, set at server start with --model-control-mode=...:

The hot-reload story:

  1. POLL. Drop resnet50/2/model.plan into the repo. On the next poll tick (default 15s), Triton sees the new version directory, loads it in a background thread, and — once it's ready — flips the routing. In-flight requests on v1 finish; new requests respect version_policy. No restart.
  2. EXPLICIT. Same file drop, then POST /v2/repository/models/resnet50/load tells Triton to reread the config and load whatever's new. Symmetric /unload exists for memory-constrained nodes.
  3. NONE. Triton ignores changes. You have to restart the process. Simpler, fewer moving parts; fine for ephemeral pods that get re-deployed when the model changes.
POLL gotcha — partial writes
If you copy the model file in place over an existing version (or use cp rather than mv), POLL may scan mid-copy and load a truncated file. Always stage the version dir on the same filesystem and mv it into place atomically, or use a new version number.

Auto-completion — when Triton fills in the blanks

For some backends, Triton can read shape / dtype information directly from the model file. TensorRT engines and ONNX models carry their input/output schema; PyTorch's TorchScript files usually don't. With auto-completion enabled, you can ship a minimal config and let the server fill in the rest:

# Minimal config — Triton reads dims and dtypes from model.plan
name: "resnet50"
backend: "tensorrt"
max_batch_size: 64

This is convenient for prototyping. For production you almost always want the full config checked in: it's the contract, and reading dims out of a binary blob is not a great review experience. Auto-completion is on by default in current Triton; --disable-auto-complete-config turns it off (the legacy flag was --strict-model-config=true).
