Dynamic batching

The scheduler that turns a stream of independent requests into batched inferences. One config block, one knob, an order of magnitude in throughput. This is the lesson where lesson 01's promise — "batching across users is where the throughput lives" — becomes a config file you can actually write.

The mismatch the scheduler fixes

Two facts that can't both be true at the request layer:

  • The GPU wants batches. A ResNet-50 at batch=32 is ~15× the throughput of batch=1 (lesson 01). A TensorRT engine is often compiled for a specific preferred batch size.
  • Clients send one at a time. User A and user B don't know about each other, can't coordinate, and shouldn't have to.

Dynamic batching closes the gap inside the server. When a request arrives, Triton doesn't dispatch it; it enqueues. The scheduler dispatches when either:

  1. It has enough requests to make a preferred batch, or
  2. max_queue_delay_microseconds has elapsed since the head of the queue arrived.

Whatever happens first. Batches always respect max_batch_size (the model's hard ceiling, declared earlier in config.pbtxt).

The config block, minimal

max_batch_size: 64

dynamic_batching {
  preferred_batch_size: [16, 32]
  max_queue_delay_microseconds: 5000
}

That's the whole API for the common case. Three values do most of the work:

KnobMeaningTypical
max_batch_sizeHard cap on a single dispatched batch. Set to what your model was compiled for.32 – 256
preferred_batch_sizeList of sizes Triton tries to dispatch at — often what TensorRT optimised for. If it can't reach one, it falls back to whatever is in queue after the delay.[16, 32]
max_queue_delay_microsecondsHow long the head of the queue is allowed to wait for the batch to grow.1000 – 10000
The defining sentence
Dynamic batching is one decision repeated forever: do I have enough to fire a preferred batch, or has the oldest request waited long enough that I should fire anyway? Everything else — priority, ordering, ragged inputs — is a refinement of that one rule.

The throughput–latency curve

The one knob that matters most is max_queue_delay_microseconds. Sweep it and you trace out a curve that looks the same on every model:

As delay grows from 0 → 10 ms, throughput climbs steeply, then plateaus once batches consistently fill to a preferred size. Latency climbs linearly the whole way. The knee — usually around the GPU's natural single-batch inference time — is the sweet spot. Past it you pay latency for diminishing throughput.

Priority queues

One model often serves multiple tenants — an interactive UI and a batch crawler, say. Without help the crawler can starve the UI. Triton's answer:

dynamic_batching {
  preferred_batch_size: [32]
  max_queue_delay_microseconds: 5000
  priority_levels: 2
  default_priority_level: 2
}

Each request can carry a priority field (1 = highest). Triton maintains one queue per level and drains higher-priority queues first. Lower numbers = higher priority; default_priority_level is what unmarked requests get.

Preserve ordering

By default Triton may reorder requests between arrival and dispatch — completing requests in whatever order their batch happens to finish. For most users that's fine. If your client assumes FIFO (e.g. a streaming protocol that expects in-order responses), set:

dynamic_batching {
  preserve_ordering: true
  ...
}

Cost: throughput drops because Triton can't pack across "incompatible" requests as aggressively. Pay it only if you need it.

Queue policy — what to do when the queue is too long

Under overload, queues grow unboundedly. Triton lets you set a policy:

dynamic_batching {
  preferred_batch_size: [32]
  max_queue_delay_microseconds: 5000
  default_queue_policy {
    timeout_action: REJECT
    default_timeout_microseconds: 100000   # reject after 100 ms in queue
    max_queue_size: 256
    allow_timeout_override: true
  }
}
FieldWhat it does
timeout_actionREJECT = return a 503-style error; DELAY = keep waiting.
default_timeout_microsecondsHow long a request waits before its action fires. Distinct from max_queue_delay: that one bounds the head-of-line wait for batching; this one is a per-request fairness deadline.
max_queue_sizeHard cap on enqueued requests. Beyond this, new requests are rejected immediately. Stops queue blow-up from cascading into memory pressure.
allow_timeout_overrideLets a client send a per-request timeout that overrides the default.

Per-priority queue policies exist too (priority_queue_policy), so the UI tenant gets a generous timeout while the crawler gets a short one.

Batchability — what can actually be batched together

Two requests are batched into one inference only if their inputs have identical shapes, modulo the batch dimension. Two image requests of shape [3, 224, 224] can join a batch of [N, 3, 224, 224]. Two image requests where one is 224×224 and the other 256×256 cannot. They become two separate batches.

For sequence inputs (variable-length text), this is fatal. Triton's answer is ragged batching:

input [
  {
    name: "input_ids"
    data_type: TYPE_INT64
    dims: [-1]
    allow_ragged_batch: true
  }
]
batch_input [
  {
    kind: BATCH_MAX_ELEMENT_COUNT_AS_SHAPE
    target_name: "input_lengths"
    data_type: TYPE_INT32
    source_input: ["input_ids"]
  }
]

With allow_ragged_batch: true Triton flattens variable-length inputs end-to-end into one tensor and gives the model a side-input describing where each item begins (here, input_lengths). The backend needs to know how to read it — TensorRT-LLM and most LLM backends do natively.

What dynamic batching does not do

Limitations
  • Stateful models — sequences with KV cache or RNN hidden state — need sequence batching instead. Two requests in the same session aren't independent. That's lesson 07.
  • Pipelined models — multi-stage where stage outputs feed stage inputs — need ensembles or BLS. That's lesson 08.
  • Across-model batching — batching a ResNet request with a BERT request — is not a thing. Batches are per-model.
