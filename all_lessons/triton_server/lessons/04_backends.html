Backends — TensorRT, PyTorch, ONNX, Python, vLLM, TRT-LLM

One server, six runtimes. Each backend brings its own loader, threading model, dtype support. Picking the right one is most of the latency budget on a fresh deploy.

What a backend actually is

A Triton backend is a shared library (.so on Linux) loaded by the server at startup. It implements a small C API — ModelInitialize, ModelFinalize, ModelInstanceInitialize, ModelInstanceExecute, ModelInstanceFinalize — and Triton calls into it for every model that names it in config.pbtxt.

One process, many backend libraries, many model instances per backend. Each backend manages its own runtime state — TensorRT engines and execution contexts, ONNX Runtime sessions, a libtorch interpreter, a Python subprocess pool. Triton is the conductor; the backend is the band.

The six common backends

TensorRT — backend: "tensorrt"

Loads .plan files: pre-built engines produced by trtexec or the TensorRT Python API. Fastest path on NVIDIA GPUs — kernel fusion, fp16/int8/fp8 quantisation, autotuned kernel selection. The cost is that you build the engine ahead of time for the exact GPU and (mostly) the exact shape.

  • Strengths. Lowest latency on vision, highest tensor-core utilisation, fp16/int8/fp8 supported.
  • Constraints. Engines are GPU-architecture-specific (an A100 engine won't run on H100). Dynamic shapes require explicit optimisation profiles at build time.
  • Use for. Production vision models, CTR / recsys models, any model where shape is mostly fixed and latency matters.

PyTorch — backend: "pytorch"

Loads TorchScript (model.pt produced by torch.jit.trace or torch.jit.script) via the C++ libtorch runtime — no Python interpreter in the request path. Newer Triton releases are adding limited support for the PyTorch 2 export path (torch.export / AOTInductor); TorchScript is still the default and most production-tested format.

  • Strengths. Highest dev velocity. No compilation step, no shape profiles, no dtype calibration. What you trained is what you serve.
  • Constraints. Slower than TensorRT (no aggressive fusion or quantisation by default). Some PyTorch features (custom ops, NumPy fallbacks) don't survive TorchScript.
  • Use for. Iteration, research-to-production handoff, models where the last 20% of latency isn't worth the engineering.

ONNX Runtime — backend: "onnxruntime"

Loads .onnx files via Microsoft's ONNX Runtime. Cross-platform — runs on CPU, NVIDIA, AMD, Intel, with the right execution provider. Broad op coverage; almost every model export tool can produce ONNX.

  • Strengths. Portability across hardware. Good when you need to serve the same artefact on CPU dev and GPU prod.
  • Constraints. Slower than TensorRT on NVIDIA (you can enable the TRT execution provider to recover some of that). ONNX exports of complex models sometimes lose shape information.
  • Use for. Cross-vendor deploys; teams already standardised on ONNX; CPU inference paths.

Python — backend: "python"

You write a model.py with three functions:

import triton_python_backend_utils as pb_utils

class TritonPythonModel:
    def initialize(self, args):
        # load weights, set up state — called once per instance
        self.model = load_my_model()

    def execute(self, requests):
        # called per batch; return one response per request
        responses = []
        for req in requests:
            x = pb_utils.get_input_tensor_by_name(req, "input__0").as_numpy()
            y = self.model(x)
            out = pb_utils.Tensor("output__0", y)
            responses.append(pb_utils.InferenceResponse([out]))
        return responses

    def finalize(self):
        pass

The Python model runs in a separate process (a "stub") and communicates with the main Triton process over shared memory. This isolation is a feature: a segfault in your Python code kills the stub, not the server. The cost is ~0.3–1 ms of IPC overhead per call.

  • Strengths. Anything Python can do — including calling Hugging Face transformers directly, running custom preprocessing, orchestrating other models (BLS, lesson 08).
  • Constraints. Slower than the compiled backends. Each instance is a process; many instances eat RAM.
  • Use for. Tokenisation, preprocessing, postprocessing, ensembles with branching logic, prototyping new model types.

vLLM — backend: "vllm"

A Triton-distributed backend that wraps vLLM's engine. The model file is actually a JSON descriptor pointing at a Hugging Face checkpoint, and the backend manages paged KV cache and continuous batching internally. From Triton's view it's "one big LLM model"; vLLM handles all per-token scheduling underneath.

  • Strengths. vLLM's paged-attention throughput; minimal Triton config. You inherit Triton's HTTP/gRPC frontend, metrics, ensembles for RAG.
  • Constraints. Wraps the upstream engine, so it inherits vLLM's release cadence and limits.
  • Use for. LLM serving where you want vLLM's scheduler but inside Triton's process model.

TensorRT-LLM — backend: "tensorrtllm"

The NVIDIA-native LLM backend. Wraps the TRT-LLM C++ runtime: in-flight batching, paged KV cache, speculative decoding, quantisation (fp8, int8, AWQ, GPTQ), tensor-parallel multi-GPU. Model artefacts are TRT-LLM "engines" built ahead of time by the TRT-LLM build scripts.

  • Strengths. Highest LLM throughput on NVIDIA GPUs at production sizing. Tight integration with Hopper / Blackwell features.
  • Constraints. Build step is non-trivial; engine is GPU-arch-specific; faster-moving than the rest of Triton.
  • Use for. Production LLM deploys on NVIDIA. Lesson 10 is the deep dive.

In-process vs out-of-process matters more than it sounds
An in-process backend shares the Triton process's memory and CUDA context — zero-copy GPU input handoff, no IPC. The Python backend's stub adds ~0.3–1 ms of IPC, which is invisible at 100 ms LLM latencies but a big tax at 2 ms vision latencies. It also means a Python crash doesn't take Triton with it, which is why preprocessing models often live in the Python backend even when they could be C++.

The decision tree

You can write your own backend

Triton's backend API is public C++. If none of the six fits — say you need a custom CUDA runtime, or a hardware accelerator with its own SDK — you implement the five lifecycle hooks against the Triton headers, build a .so, and drop it into the backends directory. Real examples in the wild include FIL (Forest Inference Library) for tree models and DALI for GPU image preprocessing.

You probably won't need to. Python backend covers 95% of "custom logic" cases at the cost of some IPC.
