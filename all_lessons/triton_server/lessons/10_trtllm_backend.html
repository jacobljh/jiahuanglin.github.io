Triton + TensorRT-LLM

LLM serving is structurally different from "infer a tensor and return one" — variable output lengths, KV cache, prefill vs decode phases. TensorRT-LLM is the LLM-optimised runtime; Triton wraps it as a backend so you get the server stack (concurrency, multi-model, ensembles) on top of TRT-LLM's iteration scheduler.

Why LLMs need a different backend

A vision model is a static graph: fixed input shape, fixed output shape, one forward pass. Batching is easy: pad to the largest sample, run one kernel.

An LLM is none of that:

  • Output length is variable. Some completions are 10 tokens, others 2000. Batching together means short ones finish early and the slot sits idle.
  • Two phases per request. Prefill processes the whole prompt at once (compute-bound); decode emits one token at a time (memory-bound). They want different batch shapes.
  • KV cache. Each sequence carries a key/value tensor that grows with every token. Storing this in HBM is the dominant memory cost.
  • Iteration-level decisions. The scheduler needs to revisit composition every token, not every "request."

vLLM, SGLang, and TRT-LLM all solve these in their own runtimes (see vLLM and SGLang series). TensorRT-LLM is NVIDIA's, and it ships as a Triton backend.

What TRT-LLM provides

CapabilityWhat it means
Paged KV cacheKV stored in fixed-size blocks, mapped per-sequence (like vLLM's PagedAttention). Eliminates the "max-seq pre-allocation" waste.
In-flight batchingNew sequences can join the batch mid-decode (continuous batching). No "wait for the longest one to finish" idle slots.
Tensor parallelismSplits one model across multiple GPUs (TP=2/4/8). Triton's instance group declares it; TRT-LLM runs the all-reduces.
Speculative decodingA small draft model proposes tokens; the target verifies in parallel. 1.5–3× decode speedup with no quality loss.
QuantisationW8A8 (INT8), W4A16 (AWQ/GPTQ), and FP8 on Hopper. Memory + bandwidth wins.
Chunked contextLong prompts are split into chunks during prefill so decode requests aren't starved (analogous to vLLM's chunked prefill).

How Triton wraps it

The config.pbtxt — TRT-LLM-specific knobs

name: "llama-70b"
backend: "tensorrtllm"
max_batch_size: 0        # batching is managed by TRT-LLM internally
model_transaction_policy { decoupled: True }   # stream tokens (multiple responses per request)

parameters: {
  key: "gpt_model_path"        value: { string_value: "/engines/llama-70b/tp4-fp8" }
  key: "max_tokens_in_paged_kv_cache" value: { string_value: "256000" }
  key: "kv_cache_free_gpu_mem_fraction" value: { string_value: "0.9" }
  key: "batch_scheduler_policy"  value: { string_value: "max_utilization" }
  key: "enable_chunked_context"  value: { string_value: "true" }
  key: "enable_kv_cache_reuse"   value: { string_value: "true" }
}

instance_group [{ kind: KIND_GPU count: 1 gpus: [0,1,2,3] }]
# count: 1 because TRT-LLM internally uses TP across the listed GPUs

The three knobs that matter most:

  • kv_cache_free_gpu_mem_fraction (default ~0.9): how much HBM to use for KV cache after weights/activations. Higher = more concurrent sequences = more throughput, but lower margin against OOM.
  • batch_scheduler_policy: max_utilization (pack the batch aggressively) vs guaranteed_no_evict (only schedule what definitely fits). The first maximises throughput; the second protects latency.
  • enable_chunked_context: splits long prefills so decode requests don't stall. Always-on in production with long prompts.

Decoupled mode — streaming tokens

By default Triton answers one response per request. LLM chat needs to stream — emit tokens as they're generated. That's the decoupled transaction policy:

model_transaction_policy { decoupled: True }

With this set, the backend can call sendResponse() many times per request, each with a partial output. The HTTP/gRPC layer forwards them as a stream (server-sent events for HTTP, gRPC streaming server response). Clients see tokens arrive one at a time.

This is what an OpenAI-compatible /v1/chat/completions endpoint expects for stream: true. Triton's tritonfrontend module can serve such an API on top of a decoupled TRT-LLM model.

The canonical LLM deployment — an ensemble

Production LLM serving on Triton is almost never a single model. The standard shape is an ensemble:

Three model directories under one repository: preprocessor/, llama-70b/, postprocessor/, plus an ensemble/ directory whose config glues them together. Clients call POST /v2/models/ensemble/infer with a string; the ensemble does tokenisation → generation → detokenisation server-side.

TRT-LLM KV sizing

Rough math: GPU HBM minus weights (after TP split) minus activation overhead → KV budget. Per-token KV ≈ 2 · n_layers · n_heads · head_dim · 2 bytes (fp16).
