The architecture

One request from arrival to response, traced through every component of a Triton process. The five boxes and where time goes. By the end you can name what every Triton config knob in lessons 03–07 controls — they all attach to one of these boxes.

The five components

Tracing one request, step by step

  1. Client sends a request over HTTP/gRPC. It targets a model name + version: POST /v2/models/resnet50/versions/1/infer for HTTP, or the equivalent gRPC ModelInfer call.
  2. The frontend parses the payload — JSON for HTTP, Protobuf for gRPC — and checks the inputs against config.pbtxt: right names, right dtypes, right shapes. If the model declares max_batch_size = 64, requests with batch > 64 are rejected here.
  3. The scheduler decides when to dispatch. For dynamic batching (lesson 06): the request joins the queue, and either it completes the queue's batch immediately or Triton waits up to max_queue_delay_microseconds for more compatible requests. For sequence batching (lesson 07): the request is routed to the instance pinned to its sequence id.
  4. A backend instance picks up the batch. The instance is one loaded copy of the model on a specific GPU. Backends are pluggable: TensorRT, PyTorch (LibTorch), ONNX Runtime, Python, vLLM, TRT-LLM. The instance executes infer(batch) in its own CUDA stream.
  5. The GPU runs the inference. The backend's runtime does whatever it does — TensorRT engine execute, ONNX session run, Python function call — and produces output tensors. Triton doesn't see inside; it just gets the result.
  6. The scheduler unbatches and the frontend serialises. Each user's request gets back exactly the response shape that was promised by the model's outputs in config.pbtxt.

The model repository — the source of truth

Everything Triton knows about a model lives in the repository directory. Bare minimum:

model_repo/
└── resnet50/                ← model name (URL path)
    ├── config.pbtxt          ← schema: inputs, outputs, backend, batching
    └── 1/                    ← version directory
        └── model.plan        ← the actual model file (here, a TRT engine)

The server scans the repository at startup (and on demand if --model-control-mode=poll). Each subdirectory is a model. Each version subdirectory (named with a positive integer) is a version. The config.pbtxt tells Triton how to load and call it. Lesson 03 is the full anatomy of config.pbtxt.

The latency budget — where the milliseconds go

For a typical ResNet-50 request at moderate load:

decode + validate
~0.5 ms
queue wait
~2.5 ms (configurable)
dispatch
~0.2 ms
GPU inference (b=32)
~4.5 ms
serialise + respond
~0.5 ms
p50 total
~8 ms

The queue wait is what trades latency for throughput. Set max_queue_delay_microseconds = 0 and you get the lowest latency (but tiny batches and low throughput). Set it to 10 ms and you get fat batches but a 10ms p99 tax. Lesson 06 is the lesson on this knob.

Process model — one Triton, many models

One Triton process serves all the models in its repository. Each model has its own scheduler queue, its own instance group, its own GPU placement. The process shares: GPU context, CUDA streams (one per instance), the HTTP/gRPC listener.

Two consequences:

  • Models compete for GPU. If one model floods, others queue. Use instance pinning and per-model rate limiting to isolate.
  • Crashes can take the process down. A backend that segfaults takes Triton with it. The Python backend runs out-of-process by default for this reason.
