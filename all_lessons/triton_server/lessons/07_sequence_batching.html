Sequence batching

For stateful models — recurrent, streaming, KV-cached — pin a sequence's requests to one instance and keep its state slot warm. Direct vs Oldest scheduling. When dynamic batching's "every request is independent" assumption is wrong, this scheduler is what you need.

Why dynamic batching breaks for stateful models

Dynamic batching (lesson 06) implicitly assumes every request is independent — that you can route request A to any instance, then route the next request to any instance, and the outputs are correct. For an RNN over time, streaming ASR, a chat model with a KV cache, that's false. Each call needs the model to remember the previous call:

  • RNN / LSTM — the hidden state at step t is the input at step t+1.
  • Streaming ASR — overlap window across audio chunks; the encoder's state carries forward.
  • Chat with KV cache — each token decode reads the KV from prior tokens of the same conversation.

If those requests get sprayed across instances, the state is on the wrong device. Sequence batching solves this with two ideas:

  1. Sequence identity. Every request carries a client-assigned sequence_id (a.k.a. correlation id, CORRID). Triton routes all requests with the same id to the same model instance.
  2. Slot pinning. Within that instance, the sequence occupies a fixed slot of the batch dimension — slot 3 of an instance with max_batch_size: 8 means batch row 3 is always this sequence's row. The backend can keep its state at that row across calls.

The protocol — start, end, ready

A client manages a session like a stream:

The three control signals are passed as named tensors per the model's config:

ControlMeaningKind enum
START1 on the first request of a sequence, 0 otherwise. Tells the backend to reset its state slot.CONTROL_SEQUENCE_START
END1 on the last request. The backend can finalise.CONTROL_SEQUENCE_END
READY1 if this slot has a valid request this step. (Multiple slots batch together — some may not have a request.)CONTROL_SEQUENCE_READY
CORRIDThe sequence id itself, available to the backend.CONTROL_SEQUENCE_CORRID

The config block

sequence_batching {
  max_sequence_idle_microseconds: 5000000   # 5s timeout for silent sessions

  control_input [
    { name: "START"
      control { kind: CONTROL_SEQUENCE_START fp32_false_true: [0, 1] } },
    { name: "END"
      control { kind: CONTROL_SEQUENCE_END   fp32_false_true: [0, 1] } },
    { name: "READY"
      control { kind: CONTROL_SEQUENCE_READY fp32_false_true: [0, 1] } },
    { name: "CORRID"
      control { kind: CONTROL_SEQUENCE_CORRID data_type: TYPE_UINT64 } }
  ]

  oldest {
    max_candidate_sequences: 1024
    preferred_batch_size: [4, 8]
    max_queue_delay_microseconds: 5000
  }
}

Same shape as dynamic_batching but with the scheduling strategy block — here oldest — picking the algorithm.

Two scheduling strategies

Direct

Each sequence is assigned a fixed slot in a fixed instance at START time, and that slot is reserved until END. Simple. Per-sequence cost is constant. The drawback: when sequences end at very different times, slots sit idle (their owners are silent or finished) and no other sequence can use them.

sequence_batching {
  direct { max_queue_delay_microseconds: 5000 }
  ...
}

Direct is right when the population of active sequences is small and roughly known — telemetry from a fixed number of devices, for instance.

Oldest

Slots are not pinned to one sequence forever; they're recycled. When a slot frees up (sequence ended, sequence idle), the scheduler hands it to the oldest waiting sequence. max_candidate_sequences bounds how many sequences can be tracked at once (LRU beyond that).

sequence_batching {
  oldest {
    max_candidate_sequences: 1024
    preferred_batch_size: [4, 8]
    max_queue_delay_microseconds: 5000
  }
  ...
}

Oldest is right when the sequence population is unbounded or much larger than your slot count — a chat server with millions of users but only 128 GPU slots. It also lets you batch across sequences each step.

Direct vs Oldest — the picking rule
Use Direct when sequences are few, long-lived, and roughly equal-rate (one slot per stream, no recycling needed). Use Oldest when sequences are many, bursty, or short — the slot becomes a multiplexed resource and the scheduler does the multiplexing.

Implicit state management

For RNN-style models, you can let Triton carry the hidden state between calls without exposing it to the client:

sequence_batching {
  state [
    {
      input_name:  "input_state"
      output_name: "output_state"
      data_type:   TYPE_FP32
      dims: [ 128 ]
      initial_state: [ { data_type: TYPE_FP32 dims:[128] zero_data:true name:"zero" } ]
    }
  ]
  oldest { ... }
}

On each call Triton reads output_state from the model's previous output and feeds it as input_state on the next call for the same sequence. The client never sees the state tensor. On START, the initial_state (zeros here) is used. This is the cleanest way to serve a recurrent model without making the client a stateful protocol participant.

What about LLMs?

LLM serving (chat with KV cache) is structurally identical to "stateful model with implicit state". But Triton's general sequence batcher isn't what LLM backends use. TensorRT-LLM and vLLM backends ship their own paged-KV scheduler with in-flight batching, prefix sharing, and continuous batching — sophistication beyond what sequence batching offers.

The split is: sequence batching is for non-LLM RNN-shaped models you write by hand; LLMs use the specialised backends covered in lesson 10. The two paths don't compose; you pick one per model.

Idle and timeout — the housekeeping

  • max_sequence_idle_microseconds — how long a sequence can be silent before the scheduler frees its slot. The next request with that CORRID will be treated as a fresh start (and may fail if the backend expected state).
  • Resource pressure with Direct: if all slots are pinned and a new START arrives, Triton queues the new sequence until a slot frees. Tune your slot count (max_batch_size × instance count) to your peak concurrent-session count.
  • Resource pressure with Oldest: max_candidate_sequences limits tracked sequences; beyond that, the LRU sequence gets evicted (its next request restarts).
