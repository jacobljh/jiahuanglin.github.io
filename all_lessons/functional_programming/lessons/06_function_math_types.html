Function math and types for programmers

Lesson 05 taught you to treat functions as values — things you pass, return, and compose. This lesson zooms out one level and asks what a function is, mathematically: a map between two sets, sending every input to exactly one output. That single picture connects three things you already half-know — purity (lesson 04), Scala's sealed traits (lesson 02), and the design rule "make illegal states unrepresentable" — into one precise tool. The key move is small and powerful: a type is just a set of values, so you can count the states a design admits and notice when it admits ones that should never exist. No category theory yet — this is the plain math vocabulary that lesson 07 will lift into a pattern language.

Linear position
Prerequisite: lesson 05 (functions as first-class values that compose) and lesson 04 (purity / referential transparency). You should be comfortable reading a Scala def, a case class, and a sealed trait (lesson 02).
New capability: see a function as a mathematical map between sets and a type as the set of its possible values, count the cardinality of product and sum types, and use that counting to redesign a type so that illegal states cannot be written down at all.

1 · A function is a map: domain → codomain

In mathematics a function f : A → B is a rule that assigns to every element of a set A (the domain) exactly one element of a set B (the codomain). Two demands hide in that sentence and both matter for programming: every input gets an output (nothing is left undefined), and each input gets exactly one output (no input maps to two different results). That second demand is just determinism — and determinism is precisely the property a pure function has (lesson 04): same input, same output, every time. So "pure function" and "mathematical function" are the same idea wearing two hats.

f : A → B   means   for every a ∈ A there is exactly one b ∈ B with f(a) = b

A function that reads the clock or a mutable global is not a function in this sense: feed it the same a twice and it can hand back two different bs, which violates "exactly one output." That is why lesson 04 insisted purity is not a style preference — it is the condition under which your Scala def actually denotes a mathematical map, and therefore the condition under which all the reasoning in this series applies.

// A genuine map Int -> Int: every input has exactly one output.
def square(n: Int): Int = n * n      // domain Int, codomain Int

// NOT a map from () to Int: same input, different outputs.
def now(): Long = System.currentTimeMillis()   // depends on hidden state

2 · Total vs partial: partiality is a lie in the type

A function is total if it is defined on its whole domain — every value of the input type produces a real output. It is partial if some inputs have no valid output. The trouble is that Scala's type system lets you claim totality while delivering partiality: a signature says List[A] => A, promising an answer for every list, but it explodes on the empty one.

def head[A](xs: List[A]): A = xs match {
  case h :: _ => h
  case Nil    => throw new NoSuchElementException  // empty list: no output!
}
def divide(a: Int, b: Int): Int = a / b           // b = 0: no output!

The type List[A] => A says "total," but head is partial: the empty list is in the domain and has no defined result. The gap leaks out at runtime as an exception — a side effect (lesson 04) that the type never warned you about. This is the central motivation for Option (lesson 11): change the codomain to Option[A] and the function becomes honestly total again — every list, including the empty one, has a real output (None).

def headOption[A](xs: List[A]): Option[A] = xs match {   // now TOTAL
  case h :: _ => Some(h)
  case Nil    => None
}

Hold that thought: making a partial function total is just enlarging the codomain by one value (the "no answer" value). We will see in §7 that this is literally 1 + |A| values — the cardinality of Option[A].

3 · Injective, surjective, bijective — and why bijections are lossless

Three properties describe how a map relates its two sets. Define each on a tiny example.

injective (one-to-one)
No two inputs share an output: f(x) = f(y) ⇒ x = y. Example: n => n + 1 on Int — distinct inputs stay distinct. Counter: n => n * 0 is not injective (everything maps to 0).
surjective (onto)
Every element of the codomain is hit by some input. Example: n => n + 1 on Int is onto Int (any m comes from m−1). Counter: n => n * n onto Int misses every negative number.
bijective (both)
Injective and surjective: a perfect pairing. Then an inverse f⁻¹ exists that undoes it. Example: b => !b on Boolean — its own inverse.

Why a programmer cares: a bijection is a lossless, reversible encoding. If encode : A → B is a bijection, then decode = encode⁻¹ recovers the original exactly — no information is lost. Serialization that round-trips, a reversible cipher, a one-to-one ID remapping: all are bijections. The moment a map is not injective, two inputs collapse to one output and you can no longer recover which you started from — that is information loss, by definition. So "can I reliably decode this?" is the same question as "is my encode injective?"

4 · Types as sets: count the values

Here is the idea that turns the rest of the lesson into arithmetic: a type is the set of all values that inhabit it. Its cardinality is just how many values that is. Some Scala types are small enough to count exactly.

TypeIts set of valuesCardinality |T|
Nothing{ } — no values at all0
Unit{ () } — exactly one value1
Boolean{ true, false }2
Byte−128 … 127256
Intall 32-bit integers2³² (≈ 4.3 billion)

Unit being a one-value set explains why a function returning Unit carries no information in its result — there is only one possible answer, so the reason you called it must be a side effect. Nothing being the empty set explains why a value of type Nothing can never be produced: there is nothing to produce. These are not curiosities; they are the base cases of an algebra. The next two sections show that Scala's two ways of building compound types correspond to the two operations of that algebra: multiply and add.

5 · Product types: cardinality multiplies (A and B)

A case class (or a tuple) that bundles fields together is a product type — the set-theoretic cartesian product of its fields. A value exists for every combination of a first field and a second, so the cardinalities multiply.

case class Point(x: Boolean, y: Boolean)   // |Point| = |Boolean| * |Boolean| = 2 * 2 = 4
// the 4 values: (false,false) (false,true) (true,false) (true,true)

case class Pixel(r: Byte, g: Byte, b: Byte) // |Pixel| = 256 * 256 * 256 = 16,777,216

Read a product as "A and B": a Point has an x and a y; you must supply both, and every pairing is a legal value. The general rule is |A × B| = |A| × |B|. (Note |A × Unit| = |A| × 1 = |A| — pairing with a one-value type changes nothing, just as multiplying by one does.)

6 · Sum types: cardinality adds (A or B)

A sealed trait with a fixed set of cases (lesson 02) is a sum type — a disjoint union. A value is one of the cases, never several at once, so the cardinalities add.

sealed trait Light                 // |Light| = 1 + 1 + 1 = 3
case object Red    extends Light
case object Yellow extends Light
case object Green  extends Light

sealed trait Shape                 // |Shape| = |Circle| + |Square|
case class Circle(r: Double) extends Shape
case class Square(side: Double) extends Shape

Read a sum as "A or B": a Light is Red or Yellow or Green. The rule is |A + B| = |A| + |B|. The "sealed" matters: it tells the compiler the case list is closed, so the cardinality is fixed and a match can be checked for exhaustiveness — the same closedness that made pattern matching safe in lesson 02. Sum and product are the two halves of algebraic data types: every data model you build is some combination of "ands" (products) and "ors" (sums), and you can compute its exact size by multiplying and adding.

7 · The payoff: make illegal states unrepresentable

Now the tool. Choose the type whose set of values is exactly the set of legal states — no more, no fewer. If a type admits values that correspond to nonsense, those bugs become writable; if it admits only the legal ones, the nonsense is a compile error, not a runtime surprise. Counting cardinalities is how you check.

Worked example — a contact has an email or a phone, never neither
The rule: a contact must have at least one of {email, phone}; having neither is illegal (both is fine).

Bad model (two nullable strings). Collapse each field to "present or absent" — that is a Boolean each, so the shape of the state space is (Boolean, Boolean): a product, 2 × 2 = 4 combinations: (email, phone), (email, no phone), (no email, phone), (no email, no phone). That last combination is illegal, yet the type happily represents it. 1 of 4 states is a bug you must remember to guard against everywhere. 
case class Contact(email: String, phone: String)  // null/"" sneak in the illegal 4th state
Good model (a sum type). Enumerate exactly the legal cases: 
sealed trait ContactInfo
case class EmailOnly(email: String)            extends ContactInfo
case class PhoneOnly(phone: String)            extends ContactInfo
case class Both(email: String, phone: String)  extends ContactInfo
Now |ContactInfo| = |Email| + |Phone| + |Both| — three cases, and there is no case for "neither." The illegal state has been deleted from the type; you cannot construct it, so no downstream code needs a guard, a test, or a comment about it. You traded a 4-state type with a landmine for a 3-shape type with none.

Two asides that fall straight out of the algebra. Option as 1 + |A| (preview of lesson 11): Option[A] is a sum — None (a single value, cardinality 1) or Some(a) (one per A), so |Option[A]| = 1 + |A|. That is exactly the "enlarge the codomain by one" we did to make head total in §2. Functions as exponents (fun fact): the number of total functions A → B is |B||A| — for each of the |A| inputs you independently pick one of |B| outputs. So there are 2² = 4 functions Boolean => Boolean, which is why this algebra is called the algebra of sum, product, and exponential types.

Common mistakes / failure modes

Boolean blindness
Modeling state with a pile of independent booleans (isPaid, isShipped, isCancelled) — a product of 2 × 2 × 2 = 8 states when only, say, 4 are legal. The illegal combinations (shipped and cancelled?) are now writable. Use a sealed trait of the real states instead.
Strings for finite sets
Using String for a status with five legal values gives a type of effectively unbounded cardinality — "shipped", "Shipped", "shpped" are all distinct values. Use a sealed trait / enum so the set is exactly the five.
Lying with totality
Returning A from a function that can fail (head, parseInt, a map lookup) hides partiality behind an exception. Make the codomain honest: Option[A] / Either[E, A] (lesson 11).
Over-wide products
Bundling fields that are only valid together with others (a discount % field that's meaningless unless a flag is set) multiplies in illegal combinations. Push the conditional field into the sum case where it belongs.

Checkpoint exercise

Try it
Three small tasks. (1) Compute the cardinality of case class Toggle(on: Boolean, label: Unit) — show your multiplication and explain why the label field does not change the count. (2) Compute the cardinality of sealed trait Coin { case object Heads; case object Tails } plus a third case case class Edge(angle: Boolean) — show your addition. (3) Redesign: here is an over-permissive type — case class Order(status: String, cancelledReason: String), where cancelledReason is only meaningful when status == "cancelled". Count how many of its states are illegal in spirit, then rewrite it as a sealed trait so that a non-cancelled order cannot carry a cancellation reason. Confirm the new type's value-set equals the set of legal orders.

Where this points next

We now have two clean pictures: types are objects (sets of values you can count) and pure functions are maps between them that compose (lesson 05) and have identities (the do-nothing map identity). When you have objects, composable arrows between them, an identity arrow, and composition that obeys associativity, you have — by definition — a category. Lesson 07 introduces category theory not as abstraction for its own sake but as the pattern language that names the lawful shapes of composition, setting up functor, monoid, and monad (lessons 08–10) as instances of patterns you will already half-recognize from this lesson.

Takeaway
A function is a mathematical map sending every domain element to exactly one codomain element — which is just purity (lesson 04) restated; a function that lies about being total (like head on an empty list) leaks the gap as an exception, which is why honest codomains like Option exist. A type is the set of its values, with a cardinality you can compute: Unit = 1, Boolean = 2, Nothing = 0. Product types (case class, tuples) mean "A and B" and multiply cardinalities; sum types (sealed trait cases) mean "A or B" and add them. The payoff is a precise design tool — make illegal states unrepresentable — by picking the type whose value-set is exactly the legal states: a contact modeled as two nullable strings admits 4 states with 1 illegal, while a 3-case sum admits exactly the legal ones, deleting the bug at the type level. (Option[A] = 1 + |A|; functions A→B number |B||A|.)

Interview prompts

(§1 — every input maps to exactly one output; "exactly one" is determinism, which is exactly what a pure function guarantees, so a pure function denotes a real mathematical map.)
  • Define total vs partial and give a partial function hiding in a "total" signature. (§2 — total = defined on the whole domain; head: List[A] => A is partial because the empty list has no output and throws; headOption: List[A] => Option[A] restores totality by enlarging the codomain.)
  • Why does a bijection matter to a programmer? (§3 — injective + surjective ⇒ an inverse exists, so encode/decode is lossless and reversible; a non-injective map collapses inputs and loses information.)
  • What does "a type is a set" let you do, and what are the cardinalities of Unit, Boolean, Nothing? (§4 — it lets you count the states a design admits; |Unit| = 1, |Boolean| = 2, |Nothing| = 0.)
  • How do product and sum types differ in meaning and in cardinality arithmetic? (§5–6 — product = "A and B", a case class/tuple, cardinalities multiply |A|×|B|; sum = "A or B", a sealed trait's cases, cardinalities add |A|+|B|.)
  • Walk through "make illegal states unrepresentable" on a contact that must have an email or phone but never neither. (§7 — two nullable strings = a 2×2 product with 1 of 4 states illegal (neither); a 3-case sum (EmailOnly/PhoneOnly/Both) has no "neither" case, so the illegal state cannot be constructed.)
  • Why is |Option[A]| = 1 + |A|, and how many functions are there of type Boolean => Boolean? (§7 — Option is a sum of None (1) and Some(a) (one per A); functions A→B number |B|^|A|, so Boolean⇒Boolean = 2² = 4.)
