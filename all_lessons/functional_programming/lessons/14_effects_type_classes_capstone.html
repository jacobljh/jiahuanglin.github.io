Effects, type classes, and the road ahead

Lesson 13 split describing a computation from running it: a LazyList is a recipe, and laziness is only safe because the recipe is pure. This last lesson takes that one move and aims it at the thing we have been quietly avoiding since Lesson 04 — side effects. The new idea: don't perform an effect, describe it as an ordinary value (an IO), compose those descriptions purely, and run the whole thing exactly once at the program's edge. That single trick assembles the functional-core / imperative-shell picture Lesson 00 promised — and it turns out Functor, Monoid, and Monad were one mechanism, the type class, the entire time.

1 · The last tension: programs must touch the world

Lesson 04 defined a pure function: its result depends only on its arguments, and calling it does nothing else — so you can replace the call with its result (referential transparency, RT) without changing meaning. That is the property the whole series protects. But a useful program must eventually print, read a file, hit the network, or read the clock. Each of those is exactly a side effect: println("hi") returns () but also does something, so it is not RT — val x = println("hi"); (x, x) prints once, while (println("hi"), println("hi")) prints twice. The expression and its value are no longer interchangeable.

So far we have answered "where do effects go?" with the slogan from Lesson 00: push them to a thin shell. The question this lesson answers is the mechanical one — how do you keep effects out of the core when the core is the part that decides what effects should happen? The trick is a reframing: distinguish performing an effect from describing one. Describing an effect is just building a value, and building a value is pure. We sequence and combine those descriptions in the core, RT intact, and hand the finished description to the shell to perform — once.

2 · The IO monad: an effect as a value you build, then run

An IO[A] is a lazy description of a computation that, when run, may perform effects and produce an A. Building one performs nothing. The essential structure, with the library noise removed:

// A first-class description of "an effect that yields an A".
final class IO[A](private val run: () => A) {
  // Functor map (Lesson 08): transform the eventual result, still describing.
  def map[B](f: A => B): IO[B]      = new IO(() => f(run()))
  // Monad flatMap (Lesson 10): sequence another effect that depends on this one.
  def flatMap[B](f: A => IO[B]): IO[B] = new IO(() => f(run()).run())
  // The ONLY impure method. Called once, at the edge of the world.
  def unsafeRunSync(): A            = run()
}
object IO {
  def apply[A](thunk: => A): IO[A] = new IO(() => thunk) // by-name (Lesson 13) = lazy
}

val hello: IO[Unit] = IO { println("hi") }   // prints NOTHING. It is a value.

hello is a value of type IO[Unit]. The println is wrapped in a thunk (Lesson 13's by-name parameter) and so does not fire when the IO is constructed — only when something calls run(). Because an IO is both lazy and a value, composing two of them is just building a bigger value, and building values is referentially transparent:

val program: IO[Unit] =
  for {
    line <- IO { scala.io.StdIn.readLine() }   // describe: read a line
    n    <- IO { line.trim.length }             // describe: a pure step (no effect)
    _    <- IO { println(s"length = $n") }      // describe: print
  } yield ()
// Up to here: NOTHING has happened. `program` is a pure description.

@main def run(): Unit = program.unsafeRunSync()  // the one place effects fire

The for-comprehension is exactly Lesson 10's sugar over flatMap/map — it sequences IO values the same way it sequenced Options in Lesson 11. Crucially, program is referentially transparent: you may pass it around, store it, run it twice, or never. List(program, program) describes running it twice and is interchangeable with val p = program; List(p, p) — building the list prints nothing either way. RT is restored for effects.

Contrast with Future (Lesson 13)
A Scala Future is eager: val f = Future { println("hi") } starts running the moment it is constructed, so val f = Future(work); (f, f) runs work once but (Future(work), Future(work)) runs it twice — Future is not referentially transparent. IO fixes precisely this: it is a description, so substitution holds, and you decide when (and whether) it runs. Same "describe vs execute" split as a LazyList — now applied to effects instead of data.

3 · This is the functional core / imperative shell, made literal

Lesson 00 drew a large pure core wrapped in a thin effectful shell and promised this lesson would make it concrete. Here it is, with no slogan left over. The core is pure functions that, given inputs, return an IO describing what should happen — it decides, it does not act. The shell is literally one line: the single unsafeRunSync() at main. The boundary between "reasoned-about" and "touches the world" is no longer a vague intention; it is a specific method call you can point at.

┌─────────────────────────────────────────────┐ │ imperative shell = program.unsafeRunSync() │ ← ONE call, at main │ ┌───────────────────────────────────────┐ │ │ │ functional core (everything else) │ │ │ │ pure fns that BUILD an IO[A] value │ │ │ │ compose via map/flatMap/for-comp │ │ │ │ referentially transparent throughout │ │ │ └───────────────────────────────────────┘ │ └─────────────────────────────────────────────┘

Everything between the two pillars (03 immutability, 04 purity) and here was a tool to make that core both pure and expressive enough to hold the logic: typed failure (Lesson 11) so errors are values, recursion and persistent structures (Lesson 12) so the loops and updates inside the core stay pure and cheap, laziness (Lesson 13) so the core can build infinite or deferred recipes. IO is the last piece: it lets the core describe even effects as values, so the only thing that is genuinely impure is the rim.

4 · Type classes: the mechanism behind every abstraction so far

We have called map "functor," combine "monoid," flatMap "monad" — and given each a set of laws. The Scala feature that encodes all three is one pattern, the type class: a trait parameterized by a type, plus instances supplied as givens (Scala 3; implicit val in Scala 2). It is ad-hoc polymorphism — adding a capability to a type without modifying that type.

trait Monoid[A] {                  // the capability: "A can be combined"
  def empty: A                     // identity element (Lesson 09)
  def combine(x: A, y: A): A       // associative op
}

given Monoid[Int]    with { def empty = 0;  def combine(x: Int, y: Int)    = x + y }
given Monoid[String] with { def empty = ""; def combine(x: String, y: String) = x + y }

// A generic function constrained by the type class. `[A: Monoid]` is a context
// bound: "there must be a Monoid[A] in scope." Works for ANY such A.
def combineAll[A: Monoid](xs: List[A]): A =
  xs.foldLeft(summon[Monoid[A]].empty)(summon[Monoid[A]].combine)

combineAll(List(1, 2, 3))          // 6      — uses given Monoid[Int]
combineAll(List("a", "b", "c"))    // "abc"  — uses given Monoid[String]

Two things are doing the heavy lifting. First, combineAll is written once and works for every type that has a Monoid instance — the compiler finds the right instance via given resolution. Second — and this is the deep difference from OO — we gave Int a Monoid without touching Int. You cannot make Int extends Combinable; you don't own Int. With a type class you supply the instance from the outside. Functor[F[_]] (a map) and Monad[F[_]] (a flatMap + pure) are the same shape over a type constructor F[_] — which is why the very same machinery describes List, Option, and our IO at once.

Worked example — one end-to-end IO program, pure until the final run
// FUNCTIONAL CORE — all pure. `greeting` does NO I/O; it returns a String.
def greeting(name: String): String =
  if name.isBlank then "Hello, stranger." else s"Hello, ${name.trim}!"

// Still core: build a DESCRIPTION of the whole interaction. Returns IO, runs nothing.
def app: IO[Unit] =
  for {
    _    <- IO { print("name? ") }              // describe a write
    name <- IO { scala.io.StdIn.readLine() }    // describe a read
    msg   = greeting(name)                      // PURE core call, no IO needed
    _    <- IO { println(msg) }                 // describe a write
  } yield ()

// IMPERATIVE SHELL — the entire effectful surface of the program is this one line.
@main def main(): Unit = app.unsafeRunSync()
Trace the boundary: greeting is testable with one line and zero setup (greeting("") == "Hello, stranger."). app is a value — evaluating the right-hand side prints nothing; you could log it, or build List(app, app), and still nothing happens. Effects fire only when unsafeRunSync() walks the description at main. That is the whole architecture in twelve lines.

5 · Common mistakes / failure modes

Running inside the core
Calling unsafeRunSync() deep in a helper to "get the value out" performs the effect early and re-pollutes the core with impurity — defeating the entire point. unsafeRun appears once, at main. Inside, you only ever flatMap.
"IO removes effects"
It does not. The effect still happens — IO relocates and sequences it as a value so the rest of the code stays RT. The world still gets printed to; you have just made when and in what order explicit and pushed it to the edge.
Type class ≠ subtyping
A Monoid[Int] instance does not make Int a subtype of anything. Dispatch is by static type via given resolution, not by an object's class — which is exactly why it works for types you cannot modify.
Forgetting the laziness
If your IO.apply takes a by-value argument instead of by-name (Lesson 13), the effect runs at construction and you have reinvented Future. The thunk is load-bearing.

6 · The road ahead: the real ecosystem

Our hand-rolled IO and Monoid exist to show the mechanism. In production you reach for libraries that are these exact ideas, hardened: Cats provides the type classes (Functor, Monoid, Monad, and more) with their laws and lawful instances for the standard types; Cats Effect provides a real IO with safe concurrency, cancellation, and resource handling; ZIO is an alternative effect system with the same description-then-run philosophy and a richer error/environment-typed IO. One more useful piece: Validated (Cats) accumulates errors instead of short-circuiting on the first — the callback Lesson 11 left open, where you want all validation failures, not just the first. And the streaming pipelines Lesson 13 §7 sketched have their pure form here too: FS2's stream is Stream[IO, A] — a backpressured Source → Flow → Sink built on exactly this IO, so a 100 GB file streams through a referentially-transparent description run only at the edge. These are not new concepts; they are the ones you already learned, packaged for scale.

7 · The pure route, the middle route, and FP beyond Scala

You do not have to go all-in to benefit. There are two routes, and they are a spectrum, not a binary. The pure route puts every effect inside IO (or ZIO) so the whole program is one description run at main — maximal reasoning power, the discipline of this lesson taken to its end (Haskell enforces it; Scala-with-Cats-Effect chooses it). The middle route applies FP in the large without an effect type: immutable values, pure functions for the logic, effects kept at the edges by convention rather than by the type system. Most production Scala, and almost all functional refactoring of existing code, lives here — and it already captures most of the benefit (local reasoning, easy tests, safe sharing) for a fraction of the ceremony.

None of this is Scala-specific — the index promised "the ideas are the point," and they travel. Kotlin/Java have records, Optional, streams, and the Arrow library; Clojure is built on persistent data structures (the Lesson 12 idea) and immutability by default; F#/C# have discriminated unions (sum types, Lesson 06), pattern matching, and LINQ. The vocabulary — immutability, purity, functor/monoid/monad, typed failure, effects-as-values — is what transfers; the syntax is a detail. Learn the shape once and you read it everywhere.

Where this points next

This is the final lesson — the road forward leaves this series. Pick up Cats and Cats Effect (or ZIO) and notice that you already understand the shapes: Monad, Monoid, IO, Validated are the things you just built, with the laws and the concurrency story filled in. Then do the one thing that closes the loop: re-read Lesson 00. The functional-core / imperative-shell diagram that read as a slogan on day one now describes a machine you can build line by line — a large pure core that decides, wrapped in a thin shell that, exactly once, makes it happen.

Takeaway
Real programs must do I/O, which breaks purity — so instead of performing effects, describe them as values. An IO[A] is a lazy description of an effectful computation; building and composing IOs via map/flatMap/for is referentially transparent (unlike eager Future), and the effect fires only at a single unsafeRunSync() at the edge — which is Lesson 00's functional core (pure code that builds an IO) wrapped in a one-line imperative shell. The abstractions that got us here — Functor, Monoid, Monad — were all the same type class mechanism: a trait plus given instances, ad-hoc polymorphism that adds capabilities to types you don't own. The production versions live in Cats / Cats Effect / ZIO. Read across the whole spine — complexity → immutable data → purity → composition → types → categories → functor/monoid/monad → typed failure → recursion → laziness → effects-as-values — and it is one argument that ends back where it began: a pure core that decides, a thin shell that acts.

  • How does IO differ from Future? (§2 — Future is eager and runs at construction, so it is not RT; IO is a deferred description you compose purely and run when/if you choose.)

  • What is a type class, and how does it differ from an OO interface? (§4 — a trait + given instances = ad-hoc polymorphism resolved on static type; unlike subtyping, you can add a capability to a type you don't own, e.g. Monoid[Int], without modifying it.)
  • In what sense were Functor, Monoid, and Monad "the same thing" all along? (§4 — each is a type class: a parameterized trait with laws plus instances; one mechanism encoding three capabilities (map, combine, flatMap) over types or type constructors.)
  • Does IO remove side effects from a program? (§5 — no; it relocates and sequences them as values so the core stays pure, and performs them once at the edge. Effects still happen; their place and order become explicit.)
