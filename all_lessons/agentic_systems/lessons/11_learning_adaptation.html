Part IV - State, collaboration, and protocols

Learning and adaptation - improving from experience

Agent learning starts with traces, feedback, and system updates before model training.

Book source
Chapter 9 - Learning and Adaptation; PDF outline pages 111-119.
Linear position
Prerequisite: Memory and traces that preserve experience.
New capability: A feedback loop for improving prompts, tools, policies, memories, evals, and models.

What the book contributes

The book describes learning and adaptation as improvement through experience and environment interaction.

It mentions reinforcement learning, supervised and unsupervised learning, few-shot adaptation, online learning, self-improving coding agents, SICA, AlphaEvolve, and OpenEvolve.

The strongest systems combine generation with evaluation and selection.

For multi-agent learning, complete trajectories matter because each agent contributes to the outcome.

Pattern shape

01Collect traces and feedback.
02Label failures or successes.
03Propose a system change.
04Test against regression evals before deployment.

Implementation sketch

for trace in production_traces:
    label = classify_failure(trace)
    if label == "wrong_test_command":
        propose(project_memory_update(trace))
    elif label == "bad_route":
        propose(router_prompt_or_threshold_change(trace))
run_regression_suite(proposed_changes)
ship_only_if_quality_improves()

Book-specific deep dive

The book's distinctive contribution here is that learning is not limited to training model weights. SICA, the self-improving coding agent, is the clearest case study. It keeps an archive of historical versions, compares benchmark performance across success rate, time, and compute cost, selects the strongest version, modifies its own codebase, benchmarks the result, and stores the new trace. The learning loop is versioned, evaluated, and replayable.

SICA also shows that adaptation often happens in tools and workflow. Its editing evolved from simple file overwrite to smarter context edits, diff-aware edits, minimal-diff output, AST-based parsing, symbol location, and hybrid search. That matters for education: an agent can improve by changing how it navigates code, edits files, manages context, or delegates to subagents, even if the base model is unchanged.

The chapter also uses an asynchronous supervisor as a safety pattern. A second model watches call graphs, messages, and tool logs to detect loops, stagnation, or inefficient behavior. Docker isolation, event-bus visualization, and subagent traces make self-improvement inspectable rather than mysterious. AlphaEvolve and OpenEvolve generalize the idea: generate candidate programs or algorithms, automatically evaluate them, keep an evolutionary database, and select improvements through measured performance.

  • SICA loop: archive versions, select by benchmark, propose code changes, run benchmark, store result, repeat under supervision.
  • Teaching point: adaptation targets prompts, memory, tools, code, routing policy, evaluation suites, and only sometimes model weights.

Running example

If many traces show the assistant uses the wrong test command in one repo, the fix may be project memory. If many repos show the same planning failure, the fix may be a prompt or policy update.

Checkpoint exercise

Try it
Take a past failure from an AI workflow. Decide whether the learning target is memory, prompt, tool design, router, eval, or model training.

Key takeaways

  • Learn from traces, not demos.
  • Test behavior changes against regression suites before deployment.
  • Keep personal, project, and global adaptation separate.

Failure modes

  • Live self-modification with no sandbox or approval.
  • Optimizing from one impressive or embarrassing run.
  • Mixing private user memory into global policy.

Implementation checklist

  • What traces are collected?
  • How are failures labeled?
  • What layer will change?
  • Which evals must pass?
  • Can the change be rolled back?
Linear takeaway
Adaptation is a measured change loop over traces, tools, prompts, memory, code, evals, and sometimes model weights.
