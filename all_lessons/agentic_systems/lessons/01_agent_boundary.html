Agent boundary - model, policy, state, environment

Separate the model from the control system around it before adding patterns.

New capability: The four nouns that make every later pattern precise: model, policy, state, environment.

What the book contributes

The book describes intelligent systems as entities that perceive context, make decisions, and interact with environments. That requires more than a model.

Level 0 is the core reasoning engine. Higher levels connect the engine to tools, strategy, and other agents. This implies a boundary between model capability and system behavior.

The infrastructure canvas manages state, communication, tool access, and logical flow. That canvas is the real agent runtime.

Patterns become easier to understand when each one is located at the boundary it modifies: context construction, action selection, tool execution, memory, or monitoring.

Book-specific deep dive

A useful agent boundary separates five things that are often blurred: the model that predicts text, the policy that decides what the model may do, the state that records the task, the tools that change the world, and the environment that returns observations. This distinction is implicit throughout the book because every pattern either changes control flow, adds state, adds tools, or changes how the agent is evaluated.

Linearized learning starts here because a pattern cannot be placed correctly until the student can say which part of the boundary it modifies. Prompt chaining changes workflow shape. Routing changes policy. Memory changes state. MCP changes the integration contract. Guardrails constrain actions. Evaluation judges the whole trace.

  • Boundary test: if a behavior fails, identify whether the failure came from model capability, missing context, bad policy, stale state, unsafe action, or poor observation.
  • Course invariant: later patterns should clarify the boundary, not hide it behind a framework.

Running example

For the coding assistant, the model does not have shell access. The environment has a run_tests tool with cwd, timeout, output truncation, and permission rules. The model may request the tool; the orchestrator decides whether the request is valid.

Checkpoint exercise

Try it
For a tool-using agent you want to build, list what belongs to the model, what belongs to policy code, what belongs to state, and what belongs to the environment.

Key takeaways

  • Keep model, policy, state, and environment separate.
  • Put hard invariants in code, schemas, permissions, and tools.
  • Treat each step as a state transition, not a chat continuation.

Failure modes

  • Using conversation history as the only state store.
  • Relying on prompt text to enforce dangerous permissions.
  • Building a loop without a terminal condition.

Implementation checklist

  • What is stored in task state?
  • What actions can the policy choose?
  • Who validates actions?
  • What observations are written back?
  • What exact condition stops the task?
Linear takeaway
A reliable agent is easier to design once the model, policy, state, tools, and environment each have a named boundary.
