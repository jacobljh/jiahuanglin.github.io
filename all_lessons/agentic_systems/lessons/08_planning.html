Part III - Action and grounding

Planning - from goal to executable path

Planning lets the agent choose a multi-step path when the path is not known in advance.

Book source
Chapter 6 - Planning; PDF outline pages 76-85.
Linear position
Prerequisite: The agent can already choose actions and observe tool results.
New capability: A mutable plan state that decomposes a goal and adapts to observations.

What the book contributes

Planning is the ability to formulate steps from an initial state toward a goal state.

The book contrasts dynamic planning with fixed workflows: use planning when the path needs exploration, and fixed flows when the path is known.

Examples include CrewAI planning, Google Deep Research, and OpenAI Deep Research style systems.

Planning becomes powerful when paired with tool observations and replanning.

Pattern shape

01Turn the user request into a goal contract.
02Generate concrete steps with dependencies and checks.
03Execute a step, observe result, update state.
04Revise the remaining plan only when observations justify it.

Implementation sketch

plan = planner(goal, current_state)
while not done(goal, state):
    step = choose_next_step(plan, state)
    observation = execute(step.action)
    state.record(step, observation)
    if observation.invalidates(plan):
        plan = replan(goal, state, previous_plan=plan)

Book-specific deep dive

Planning appears after tool use because a plan without executable actions is just an outline. The book connects planning to CrewAI-style task decomposition, Google DeepResearch, and OpenAI Deep Research: the system creates a path, gathers evidence, updates state, and revises as it learns. The plan is therefore a living control object.

A useful plan names steps, dependencies, tools, expected observations, risk points, and stop conditions. It should be short enough to revise. Overplanning is a common failure: the model spends context on an elegant plan that becomes obsolete after the first tool observation.

  • Plan fields: goal, assumptions, steps, required tools, acceptance checks, and replanning triggers.
  • Linear role: planning combines chains, routing, reflection, and tools into a multi-step path.

Running example

A deep research agent starts with subquestions, searches sources, discovers a missing concept, revises the plan, runs another search, then synthesizes only after the evidence threshold is met.

Checkpoint exercise

Try it
Write a plan for a task where the path is uncertain. Add a replan trigger for each step.

Key takeaways

  • Use planning when the path is uncertain and multi-step.
  • Store plans as mutable state with concrete checks.
  • Replan only when observations justify it.

Failure modes

  • Treating the first plan as sacred.
  • Writing motivational plans with no executable checks.
  • Letting planning consume the whole budget before action.

Implementation checklist

  • What is the goal state?
  • What are the steps and checks?
  • What observation triggers replanning?
  • What budget limits planning?
  • Who approves risky plans?
Linear takeaway
Planning composes earlier control-flow patterns into a revisable path from goal to observed progress.
