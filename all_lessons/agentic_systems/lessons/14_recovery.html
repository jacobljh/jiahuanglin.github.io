Part V - Reliability and control

Exception handling and recovery - controlled failure

Reliable agents treat errors as expected observations with recovery paths.

Book source
Chapter 12 - Exception Handling and Recovery; PDF outline pages 141-146.
Linear position
Prerequisite: Goals and monitoring, so failure can be interpreted relative to a goal.
New capability: Failure taxonomy, retry/fallback/escalation, and clean stop behavior.

What the book contributes

The book frames exception handling as essential for robust agents in unpredictable environments.

It covers error detection, graceful handling, retries, fallback strategies, logging, notification, diagnosis, self-correction, and escalation.

Failures include tool errors, unavailable services, unexpected outputs, and changing conditions.

Recovery often composes with reflection: analyze failure and try an improved action.

Pattern shape

01Detect and classify the failure.
02Choose retry, repair, fallback, degrade, escalate, rollback, or stop.
03Record the recovery attempt.
04Resume only if the state is safe and the goal remains reachable.

Implementation sketch

failure = classify(observation)
match failure.type:
    case "timeout" if attempts < 2: retry_with_backoff()
    case "invalid_schema": repair_output_or_reprompt()
    case "permission_denied": ask_human_or_stop()
    case "test_failed": route_to_debugging()
    case _: fail_with_evidence(failure)

Book-specific deep dive

Exception handling and recovery are what separate a fragile demo from an operational agent. The book's recovery pattern starts with classification: tool failure, unavailable service, malformed output, missing permission, conflicting evidence, unsafe action, or no-progress loop. Each class needs a different response.

Recovery is linearized after goals and monitoring because the agent must first know what it was trying to do and how progress is measured. Then it can retry with backoff, switch tools, ask for clarification, use a fallback model, roll back state, or stop cleanly. Silent recovery is dangerous; the trace should show what failed and what policy was applied.

  • Recovery policy: classify, choose bounded retry/fallback/escalation, update state, and preserve the failure for evaluation.
  • Stop condition: if recovery would hide uncertainty or create side effects, ask a human or fail visibly.

Running example

If a test command fails, the assistant stores command, exit code, and failure excerpt, then routes to debugging. If permission is denied, it does not try a workaround; it asks or stops.

Checkpoint exercise

Try it
List five failure types for one tool in your agent. For each, choose retry, repair, fallback, escalate, or stop.

Key takeaways

  • Classify failures before choosing recovery.
  • Retry only when safe or idempotent.
  • Record failure and recovery in state.

Failure modes

  • Blind retry loops that amplify cost.
  • Treating permission denial as a prompt problem.
  • Hiding partial failure behind a confident answer.

Implementation checklist

  • What failure types exist?
  • Which are retryable?
  • Which need fallback or escalation?
  • What is the attempt budget?
  • What does the final failure report contain?
Linear takeaway
Recovery is controlled autonomy under stress: classify the failure, choose a bounded response, and preserve the trace.
