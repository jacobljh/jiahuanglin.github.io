Part V - Reliability and control

Goal setting and monitoring - objective, progress, stop

An agent needs a measurable target and a way to know whether it is getting closer.

Book source
Chapter 11 - Goal Setting and Monitoring; PDF outline pages 131-140.
Linear position
Prerequisite: An agent loop that can act and persist state.
New capability: Goal contracts, progress fields, and explicit stop conditions.

What the book contributes

Goal setting gives the agent direction; monitoring lets it track progress and determine success.

The chapter references SMART goals and measurable success criteria.

Monitoring observes agent behavior, environment state, and tool outputs.

Feedback from monitoring lets the agent adapt, correct plans, or escalate.

Pattern shape

01Translate user intent into a goal contract.
02Track structured progress against success criteria.
03Use observations to update progress.
04Stop on success, impossibility, budget exhaustion, or human-needed state.

Implementation sketch

goal = {
  "objective": "make parser accept trailing commas",
  "constraints": ["preserve public API"],
  "success_evidence": ["targeted test passes", "diff summarized"],
  "budget": {"tool_calls": 20, "minutes": 30}
}
progress = monitor(state, goal)
if progress.success_evidence_met:
    state.status = "done"

Book-specific deep dive

Goal setting and monitoring turn autonomy into a managed process. The book treats goals as more than user wishes: they need success criteria, progress signals, constraints, and a way to notice drift. A monitoring layer can compare current state to the goal, detect stalled progress, and decide whether to continue, replan, recover, or escalate.

This lesson belongs after MCP because once an agent can access more tools and context, it needs stronger goal telemetry. Monitoring can track completed subtasks, tool errors, budget spent, evidence gathered, open risks, and confidence in the final answer. Good monitoring makes the agent explain not only what it did, but why it believes the task is complete.

  • Goal contract: objective, constraints, done condition, progress metrics, and escalation threshold.
  • Monitor signal: progress, stagnation, risk, cost, quality, and user-impact deltas.

Running example

A coding task goal should name the behavior to change, constraints, tests or evidence, and what to do if the fix requires an out-of-scope migration.

Checkpoint exercise

Try it
Rewrite a vague request into a goal contract with objective, constraints, success evidence, budget, and stop conditions.

Key takeaways

  • Turn user intent into a goal contract before autonomous work begins.
  • Monitor progress as structured state.
  • Make stop conditions explicit and reportable.

Failure modes

  • Changing the goal silently mid-run.
  • Reporting success without evidence.
  • Continuing after success criteria are met.

Implementation checklist

  • What proves success?
  • What constraints cannot be violated?
  • What progress fields are tracked?
  • What stop conditions exist?
  • What is reported on failure?
Linear takeaway
Goals and monitors convert an open-ended run into measurable progress, drift detection, and stop decisions.
