Resource-aware optimization - cost, latency, budgets

Agents must manage compute, time, money, context, tools, and human attention as scarce resources.

Book source
Chapter 16 - Resource-Aware Optimization; PDF outline pages 175-185.
Linear position
Prerequisite: The agent can retrieve, call tools, delegate, and monitor goals.
New capability: Budget-aware choices over models, tools, context, and fallback paths.

What the book contributes

Resource-aware optimization dynamically manages compute, time, and financial resources.

The chapter discusses dynamic model routing, fallback models, OpenAI/OpenRouter style examples, context pruning, adaptive tool selection, and graceful degradation.

A routing agent may choose a fast cheap model for simple questions and a stronger model for complex ones.

A critic or monitoring component can improve resource allocation over time.

Book-specific deep dive

The book starts with dynamic model switching but does not stop there. OpenAI-style and OpenRouter-style examples show two practical mechanisms: choose a model based on task complexity, or provide ordered fallback models so the system can survive rate limits, service failures, or filtering. OpenRouter also supports automatic model selection and returns metadata about the actual model used, which is important for cost attribution and evaluation.

The chapter's broader taxonomy is the real teaching payload. Resource-aware agents can optimize adaptive tool choice, context pruning and summarization, proactive workload prediction, cost-sensitive exploration, energy-aware deployment, distributed-compute awareness, learned resource allocation, and graceful degradation. This means the resource manager is not just a model router. It is a policy layer over money, latency, tokens, tools, human attention, and quality risk.

A strong implementation records resource decisions in the trace: why this model, why this context size, why this tool, why this fallback. Evaluation must compare routes, not only final answers. A cheaper route that drops a citation or misses a safety check is not an optimization; it is quality debt disguised as savings.

  • Optimization surface: model, tool, context, cache, parallelism, retry depth, human review, and degradation mode.
  • Trace requirement: every resource decision should be explainable, measurable, and reversible.

Running example

A research agent uses a small model to classify sources, a stronger model to synthesize conflicting evidence, and deterministic citation checks before final answer.

Checkpoint exercise

Try it
For each step in your agent, choose the cheapest model or tool that should be sufficient. Mark which steps require the strongest model.

Key takeaways

  • Represent budgets explicitly in task state.
  • Route model strength by difficulty, risk, and required judgment.
  • Optimize context and tools without losing provenance.

Failure modes

  • Always using the largest model because routing is hard.
  • Cutting context until evidence disappears.
  • Optimizing cost without quality regression tests.

Implementation checklist

  • What budgets exist?
  • Which steps can use cheaper models?
  • What caches are available?
  • How is cost attributed?
  • Do evals compare quality by route?
Linear takeaway
Resource awareness is a policy over models, tools, context, retries, parallelism, human attention, and graceful degradation.
