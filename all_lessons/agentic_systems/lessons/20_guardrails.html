Part VII - Reasoning, safety, and evaluation

Guardrails and safety - layered constraints

Safety is built from input checks, tool permissions, output filters, review, and monitoring.

Book source
Chapter 18 - Guardrails and Safety Patterns; PDF outline pages 199-209.
Linear position
Prerequisite: The agent can reason and act, so unsafe behavior now has consequences.
New capability: Layered constraints around inputs, prompts, tools, outputs, human review, and monitoring.

What the book contributes

Guardrails keep agents safe, compliant, and aligned with intended behavior.

They can operate at input validation, output filtering, behavior constraints, tool restrictions, external moderation, and HITL checkpoints.

The chapter stresses combining multiple guardrail techniques rather than relying on one.

It also frames reliable agents as complex software systems needing state management, testing, fault tolerance, and monitoring.

Pattern shape

01Classify or sanitize risky inputs.
02Constrain model behavior with contracts and policy.
03Enforce tool permissions outside the model.
04Check outputs and monitor violations.

Implementation sketch

if input_policy.blocks(user_request):
    return safe_refusal(user_request)
action = model_policy(context)
if not tool_policy.allowed(action, state):
    return require_approval_or_block(action)
output = generate(action)
return output_filter.enforce(output, policy)

Book-specific deep dive

The guardrails chapter combines safety patterning with engineering discipline. CrewAI and Vertex AI examples illustrate different enforcement points: constrain prompts and roles, filter inputs and outputs, restrict tools, block unsafe actions, require approval, and monitor behavior. Guardrails are most effective when they are layered rather than stuffed into one prompt.

Reliable agents need policy checks before action, during tool use, and before final response. They also need observability so blocked actions become data for evaluation rather than confusing user experiences. Guardrails should be specific enough to enforce and flexible enough to allow safe alternatives.

  • Guardrail layers: input policy, context policy, tool permission, action approval, output policy, and trace audit.
  • Design warning: a guardrail that cannot explain its block reason will be bypassed or ignored in practice.

Running example

A web page retrieved by an agent may contain instructions to reveal secrets. The page is evidence, not policy. Tool permissions and instruction hierarchy prevent the external content from taking control.

Checkpoint exercise

Try it

Key takeaways

  • Layer guardrails across input, policy, tools, output, and monitoring.
  • Enforce hard constraints outside the model.
  • Treat external content as untrusted data.

Failure modes

  • A long safety prompt as the only control.
  • Broad credentials exposed to model-facing tools.
  • No audit trail for policy-sensitive actions.

Implementation checklist

  • What hard constraints exist?
  • Where are they enforced?
  • Which tools require approval?
  • How are unsafe inputs handled?
  • What does the trace record?
Linear takeaway
Guardrails are layered enforcement points around context, tools, actions, outputs, and trace auditability.
