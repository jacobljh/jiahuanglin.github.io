Capstone - compose a reliable coding and research agent

The full book becomes one architecture: act, remember, recover, evaluate, and improve.

What the book contributes

The conclusion synthesizes patterns into a complete agent-design toolbox and emphasizes composition.

Appendix B covers GUI and real-world environment interaction; Appendix C surveys frameworks such as LangChain, LangGraph, ADK, and others.

Appendix D covers AgentSpace-style UI construction; Appendix E covers CLI agents; Appendix G covers programming agents and team workflows.

The final message is that agent systems need architecture, not only model capability.

Book-specific deep dive

The capstone should make the appendices operational. Appendix B extends the agent boundary from text to GUI and real-world environments: the agent observes screens or environment state, acts through interfaces, and must handle latency, ambiguity, and side effects. Vibe coding belongs here as an interaction style, but the course should translate it into explicit context, edits, tests, and review.

Appendix C is the framework selection layer. LangChain is useful for chains and integrations, LangGraph for explicit stateful graphs, ADK for Google-style agent development and tool integration, and other frameworks for different orchestration preferences. The design question is not which framework is fashionable; it is whether the framework exposes state, tools, handoffs, retries, evaluation, and observability clearly enough for the target system.

Appendix D and E turn the same patterns into concrete surfaces: AgentSpace-style UI construction for managed agent experiences, and command-line agents such as Claude Code, Gemini CLI, Aider, GitHub Copilot CLI, plus Terminal-Bench-style evaluation for terminal work. Appendix F's reasoning-engine material reminds students that model internals affect capability, but the system still needs contracts around evidence, action, and evaluation.

Appendix G gives the strongest capstone workflow: a human-led team of specialized programming agents. The human remains orchestrator and architect, prepares high-quality context, chooses direct model access when needed, and treats prompts as versioned project assets. Specialized agents handle implementation, testing, documentation, optimization/refactoring, and code review. The reviewer agent should first critique and then reflect on its critique to produce prioritized, actionable feedback. Local context bundles, prompt libraries, and Git hooks make this workflow repeatable.

  • Capstone architecture: human orchestrator, context preparation area, specialist agents, tool/MCP layer, memory, eval suite, and release gate.
  • Final teaching move: the book's patterns become a team operating model, not merely a single autonomous chatbot.

Running example

Capstone task: "Add a feature to this repo and explain it." The agent contracts the goal, inspects files, plans, asks approval if needed, edits through patches, runs tests, recovers from failures, reflects on the diff, and reports evidence.

Key takeaways

  • Compose patterns by dependency, not by fashion.
  • Keep the loop and trace explicit even when a framework hides details.
  • Operate the agent with evals, monitoring, and budget controls.

Failure modes

  • A polished demo with no replayable trace.
  • A framework abstraction that hides state, permissions, or evaluation.
  • Final answers that omit evidence, uncertainty, or side effects.

Implementation checklist

  • Can an operator replay the run?
  • Can risky actions be blocked?
  • Can failures recover or stop cleanly?
  • Can a new version pass regressions?
  • Can the user see evidence for success?
Linear takeaway
The final design is a human-led operating model: specialist agents, explicit context, tools, memory, evals, and release gates.
