Part II - Core execution patterns

Prompt chaining - linear decomposition

Turn one overloaded request into a sequence of focused transformations.

Book source
Chapter 1 - Prompt Chaining; PDF outline pages 24-32.
Linear position
Prerequisite: Typed prompt/context contracts from lesson 02.
New capability: A fixed sequence of model calls where each artifact narrows the next step.

What the book contributes

The book defines prompt chaining as a pipeline pattern: each step handles a smaller subtask and passes output to the next step.

It improves reliability because each call has a narrower job and the system can inspect intermediate artifacts.

The chapter connects chaining with context engineering and shows that each step can be optimized independently.

Frameworks such as LangChain, LangGraph, ADK, and CrewAI support chains, but the design principle is independent of framework syntax.

Pattern shape

01Break task into ordered transformations.
02Give each transformation a contract.
03Persist intermediate artifacts.
04Validate before moving to the next step.

Implementation sketch

chain = [
  classify_request,
  extract_requirements,
  retrieve_context,
  draft_answer,
  verify_against_requirements,
  format_final
]
state = initial_state
for step in chain:
    artifact = step.run(state)
    validate(artifact, step.output_schema)
    state.add(artifact)

Book-specific deep dive

The book presents prompt chaining as the first real control-flow pattern because it converts one overloaded request into a sequence of smaller model calls. Each step performs one transformation, such as classify, extract, draft, critique, or format. The important teaching idea is that intermediate outputs become first-class artifacts. They can be inspected, validated, cached, retried, or replaced by a deterministic tool.

The chapter also ties prompt chaining to context engineering. A chain works only when each step receives the right slice of context and emits output in the exact shape expected by the next step. This is why chaining is the bridge from prompt design to agent architecture: the prompt is no longer a single instruction; it is a typed interface between stages.

  • When to use: use a chain when the task has stable phases and each phase can be judged locally.
  • When to avoid: avoid chaining when later steps require information that earlier steps discard.

Running example

Research chain: rewrite the question, retrieve sources, summarize each source, synthesize claims, verify citations, and produce the final answer. Each step reduces ambiguity for the next.

Checkpoint exercise

Try it
Take a complex prompt you would normally ask in one shot. Split it into four typed steps and name the artifact each step returns.

Key takeaways

  • Use chains when dependencies are known and sequential.
  • Make every intermediate artifact inspectable and typed.
  • Stop or route to recovery when a validation step fails.

Failure modes

  • A chain of vague prose prompts that hides state.
  • Running every request through every step when routing could skip work.
  • Testing only the final answer.

Implementation checklist

  • Can each step be named by its transformation?
  • Does each step reduce or reorganize information?
  • Can a failed step be rerun from stored inputs?
  • Do artifacts support debugging?
Linear takeaway
Chaining is the first control-flow pattern: one complex task becomes inspectable stages with typed handoffs.
