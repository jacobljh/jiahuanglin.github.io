Part VII - Reasoning, safety, and evaluation

Evaluation and monitoring - traces, metrics, regressions

Agent evaluation scores the trajectory, not just the final answer.

Book source
Chapter 19 - Evaluation and Monitoring; PDF outline pages 210-219.
Linear position
Prerequisite: Guardrails and traces, so behavior can be measured.
New capability: Offline evals, online monitoring, trajectory scoring, and regression gates.

What the book contributes

The chapter argues that agent evaluation must measure effectiveness, efficiency, and compliance in dynamic environments.

It includes production performance tracking, A/B testing, compliance auditing, drift detection, trajectory evaluation, and ADK evaluation tooling.

Trajectory evaluation compares actual behavior sequences with expected behavior or contracts.

The book also discusses moving from informal prompts to accountable contracts with verifiable deliverables.

Pattern shape

01Record full traces.
02Score route choices, tool calls, observations, outputs, cost, latency, and policy behavior.
03Run offline regression suites before changes.
04Monitor production metrics and turn failures into new evals.

Book-specific deep dive

The evaluation chapter emphasizes agent trajectories, not just final answers. A good final response can hide a bad route, unsafe tool call, excessive cost, or lucky guess. The book also connects evaluation to the idea of advanced contractor-like agents: as agents perform more complex work, the system must judge planning, evidence gathering, tool use, handoffs, and recovery.

Google ADK appears as one implementation surface, but the general lesson is framework-independent. Evaluation should cover task success, trace quality, source grounding, safety, latency, cost, and regression behavior across versions. Monitoring then watches the same signals in production and feeds learning/adaptation.

  • Eval unit: score the trace: goal contract, route, tools, evidence, final output, and side effects.
  • Monitoring loop: detect drift, collect failures, label root causes, and send them back to lesson 11's adaptation pipeline.

Running example

A coding-agent eval should inspect whether the agent read relevant files, made a minimal diff, ran the correct tests, handled failures, and reported evidence - not only whether the final prose sounded good.

Checkpoint exercise

Try it
Define five metrics for your agent: two quality, two operational, one safety. Decide which are offline and which are production monitors.

Key takeaways

  • Evaluate trajectories, not just final answers.
  • Keep offline regression suites and online production metrics.
  • Turn failures into new test traces.

Failure modes

  • A few demos treated as an eval suite.
  • No cost or latency metrics alongside quality.
  • No replayable traces because tool outputs were not stored.

Implementation checklist

  • What is the trace schema?
  • What metrics define success?
  • Which regressions block deployment?
  • How are human reviews sampled?
  • How are costs attributed?
Linear takeaway
Evaluation scores the whole trajectory, then monitoring turns production failures back into the learning loop.
