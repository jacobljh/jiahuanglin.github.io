Part IV - State, collaboration, and protocols

Multi-agent collaboration - roles, handoffs, synthesis

Multiple agents help when specialization, isolation, or parallel judgment beats one general loop.

Book source
Chapter 7 - Multi-Agent Collaboration; PDF outline pages 86-96.
Linear position
Prerequisite: A single-agent loop with explicit state, tools, and plan.
New capability: Role-specialized loops coordinated through typed artifacts.

What the book contributes

The book presents multi-agent collaboration as multiple specialized agents working toward a shared goal.

Collaboration can be sequential handoff, parallel processing, debate, or hierarchy.

Examples include research agents, analysis agents, writer agents, manager agents, CrewAI crews, and ADK multi-agent workflows.

The core benefit is division of expertise; the core risk is coordination overhead and unclear ownership.

Pattern shape

01Define roles by responsibility and tools.
02Give each role typed inputs and expected artifacts.
03Coordinate through an orchestrator or clear handoff protocol.
04Synthesize outputs and resolve disagreements.

Implementation sketch

roles = {
  "researcher": Agent(tools=[search, read_docs]),
  "coder": Agent(tools=[read_file, apply_patch, run_tests]),
  "reviewer": Agent(tools=[diff, run_tests])
}
artifact = roles["researcher"].run(research_task)
patch = roles["coder"].run(coding_task.with_context(artifact))
review = roles["reviewer"].run(review_task.with_patch(patch))
final = synthesize([artifact, patch, review])

Book-specific deep dive

The book frames multi-agent collaboration as role specialization plus communication structure. Agents can be organized as a manager-worker hierarchy, a peer debate, a pipeline, or a marketplace of specialists. CrewAI and ADK examples make the same lesson concrete: a multi-agent system is not better because it has more agents; it is better only when separation of responsibility reduces complexity or improves checks.

The educational move is to ask what each agent owns. One agent may own research, another implementation, another critique, another final synthesis. If ownership is vague, the system gets more expensive and harder to debug. The trace must preserve who decided what, which evidence they used, and where handoffs occurred.

  • Design rule: add an agent only when it owns a distinct skill, context, permission, or evaluation role.
  • Communication rule: pass artifacts and decisions, not long conversational noise.

Running example

A code-change system can split into planner, implementer, and reviewer roles. The orchestrator owns state and final synthesis; roles do not freely overwrite one another.

Checkpoint exercise

Try it
Name three roles for a multi-agent version of your project. For each role, define allowed tools and the artifact it must return.

Key takeaways

  • Add agents only for specialization, isolation, or useful parallelism.
  • Make handoffs typed and artifact-based.
  • Centralize task state and final synthesis.

Failure modes

  • A multi-agent chat room with no owner of final state.
  • Overlapping roles that duplicate work.
  • Debate loops with no verifier or decision rule.

Implementation checklist

  • Why is each role separate?
  • What artifact does each role produce?
  • Who owns synthesis?
  • What state is shared?
  • How are disagreements resolved?
Linear takeaway
Multi-agent design only helps when roles, ownership, handoffs, and trace accountability reduce complexity.
