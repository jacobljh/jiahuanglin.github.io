Part VI - Knowledge, communication, and optimization

RAG and agentic retrieval - open-book reasoning

Retrieval grounds the agent in external knowledge; agentic retrieval decides when and how to look.

Book source
Chapter 14 - Knowledge Retrieval / RAG; PDF outline pages 153-163.
Linear position
Prerequisite: Tools, goals, and recovery, because retrieval is an action with evidence and failure modes.
New capability: External evidence acquisition with provenance and sufficiency checks.

What the book contributes

RAG lets LLMs access external, current, proprietary, or domain-specific information.

The basic flow is retrieve relevant content, augment the prompt, and generate a grounded answer.

The chapter covers semantic search, vector databases, GraphRAG, and agentic RAG.

Agentic RAG adds active reasoning: the agent can search, inspect, verify, refine queries, and resolve conflicts.

Pattern shape

01Decide whether evidence is needed.
02Retrieve candidate sources.
03Assess relevance, authority, recency, and conflicts.
04Pack compact evidence into context and synthesize with citations.

evidence = []
for query in query_plan:
    hits = retriever.search(query)

    query_plan = rewrite_queries(gaps(evidence))

Book-specific deep dive

The RAG chapter teaches the retrieval stack from the bottom up. Documents are chunked so the system can retrieve manageable passages. Chunks are embedded into high-dimensional vectors so semantic similarity can be searched even when wording differs. Vector databases such as Pinecone, Weaviate, Chroma, Milvus, Qdrant, Redis, Elasticsearch, or Postgres with pgvector store and query those embeddings, often through approximate nearest-neighbor methods such as HNSW or libraries such as FAISS and ScaNN.

The book also contrasts vector retrieval with keyword retrieval. BM25 is strong when exact terms matter; semantic search is strong when meaning matters. Hybrid retrieval combines both. Chunking strategy, metadata quality, recency, and corpus synchronization determine whether RAG grounds the answer or injects noise. The agent must keep provenance because citations are the user's way to verify the answer.

GraphRAG extends retrieval by using explicit entities and relationships instead of only vector proximity. It helps when the answer depends on connections across documents, such as finance, science, or enterprise relationship questions. Agentic RAG then adds an active reasoning layer: inspect source authority, discard outdated documents, resolve conflicts, refine queries, and decide when evidence is sufficient. The ADK and LangChain examples in the book are implementation surfaces for that same loop.

  • Retrieval stack: chunk, embed, index, search, rerank/filter, pack context, answer with provenance.
  • Agentic layer: decide when to retrieve, judge source quality, handle conflicts, and stop when evidence is enough.

Running example

A policy agent retrieves current policy documents, checks date and authority, extracts relevant passages, handles conflicts, and answers with source-grounded caveats.

Checkpoint exercise

Try it
Design a retrieval sufficiency rule for one domain question. How many sources, what authority, and what conflict policy are required?

Key takeaways

  • Treat retrieved text as evidence with provenance.
  • Use iterative retrieval only when evidence is insufficient.
  • Pack context by decision relevance, not raw top-k rank.

Failure modes

  • Dumping top-k chunks into context with no source quality check.
  • No conflict handling when sources disagree.
  • Retrieval loops with no budget or sufficiency criterion.

Implementation checklist

  • Which corpus is authoritative?
  • How are queries rewritten?
  • What metadata is preserved?
  • What counts as enough evidence?
  • How are conflicts reported?
Linear takeaway
RAG gives the agent external evidence; agentic RAG teaches it when evidence is enough, stale, conflicting, or missing.
