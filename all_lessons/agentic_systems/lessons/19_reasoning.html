Part VII - Reasoning, safety, and evaluation

Reasoning techniques - decomposition, search, verification

Reasoning is controlled extra computation spent to improve hard decisions.

Book source
Chapter 17 - Reasoning Techniques plus Appendix F; PDF outline pages 186-198 and 287-298.
Linear position
Prerequisite: Budget awareness, because deeper reasoning spends resources.
New capability: Methods for hard tasks: decomposition, ReAct, tree search, debate, self-correction, and verification.

What the book contributes

The chapter covers multi-step reasoning, Chain of Thought, ReAct, Tree of Thoughts, self-correction, debate-style collaboration, reasoning-time scaling, and MASS-like optimization of multi-agent structures.

It emphasizes that performance depends not only on model size but also on allocated thinking time and interaction with tools.

Appendix F discusses reasoning-engine internals for major assistants at a conceptual level, reinforcing that models parse context, retrieve or activate knowledge, construct reasoning paths, and generate answers.

The engineering lesson is to externalize enough reasoning artifacts for verification and trace review.

Pattern shape

01Choose a reasoning strategy based on difficulty and risk.
02Generate hypotheses or decomposition.
03Use tools or evidence to check intermediate claims.
04Select or revise using a verifier, test, rubric, or human review.

Implementation sketch

hypotheses = model.generate_hypotheses(problem, k=3)
checks = [design_check(h) for h in hypotheses]
observations = [execute(c) for c in checks]
selected = verifier.choose(hypotheses, observations, rubric)
return selected.answer_with_evidence()

Book-specific deep dive

The reasoning chapter gathers techniques that help an agent decide under uncertainty: decomposition, stepwise analysis, self-consistency, tree-style search, verification, and explicit thinking about how the agent is thinking. It also discusses reasoning scaling: some tasks benefit from more compute, more candidate paths, or deeper verification, while others do not.

The linear lesson is that reasoning is not a private monologue to expose verbatim. It is an internal control resource. The system should externalize useful artifacts: assumptions, subproblems, evidence needs, candidate plans, checks performed, and confidence. That gives users and evaluators the parts of reasoning they can inspect without relying on hidden chain text.

  • Reasoning artifacts: decomposition, assumptions, search branches, verification results, and uncertainty.
  • Budget rule: increase reasoning depth only when task risk, ambiguity, or value justifies the cost.

Running example

A debugging agent forms two hypotheses, runs a targeted test for each, observes which fails, patches the likely cause, and stores the hypothesis-test result in the trace.

Checkpoint exercise

Try it
Choose a hard problem. Decide whether it needs direct answer, decomposition, ReAct, tree search, debate, or external verification. Justify the cost.

Key takeaways

  • Spend extra reasoning only where difficulty or risk warrants it.
  • Pair reasoning with observation and verification.
  • Store reasoning artifacts needed for replay and evaluation.

Failure modes

  • Longer output mistaken for better reasoning.
  • Tree search or debate with no verifier.
  • Hidden reasoning artifacts that cannot be evaluated.

Implementation checklist

  • What strategy fits the task?
  • What can be externally verified?
  • What artifacts should be stored?
  • When does reasoning stop?
  • How is reasoning quality measured?
Linear takeaway
Reasoning depth is a budgeted control resource; expose artifacts users can inspect and score.
