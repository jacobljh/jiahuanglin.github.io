Part IV - State, collaboration, and protocols

MCP - model context protocol as integration contract

MCP standardizes how agents discover and use external resources, prompts, and tools.

Book source
Chapter 10 - Model Context Protocol; PDF outline pages 120-130.
Linear position
Prerequisite: Tool use and memory, so external capabilities already have semantics.
New capability: A protocol boundary for connecting agent hosts to resources and tools.

What the book contributes

MCP is presented as an open standard for communication between LLM applications and external systems.

It uses a client-server model: servers expose resources, prompts, and tools; clients consume them.

The chapter contrasts MCP with simple function calling: MCP standardizes external integration, while function calling is a model action mechanism.

Examples include ADK consuming MCP servers and FastMCP for creating Python servers.

Pattern shape

01Server exposes resources, prompts, and tools.
02Agent host discovers capabilities.
03Model chooses or requests a capability through the host.
04Server executes or returns context with provenance.

Book-specific deep dive

The MCP chapter distinguishes protocol-level integration from ordinary function calling. Function calling gives one model a list of callable functions. MCP defines how external servers expose tools and context in a standardized way so agents can discover, connect to, and consume capabilities without every integration becoming a one-off adapter.

The ADK and FastMCP examples make this practical: define an MCP server, expose a tool set, connect an ADK agent or web surface, and let the agent call those tools through a stable contract. The lesson is that MCP belongs at the boundary between agent and environment. It is less about one clever tool and more about maintainable context/tool infrastructure.

  • MCP value: standardized discovery, tool metadata, server boundaries, and reusable integrations.
  • Design warning: protocol standardization does not remove the need for permissions, schemas, timeouts, and audit logs.

Running example

A repo MCP server exposes search_symbols, read_file_slice, list_tests, and run_read_only_test. The coding agent discovers these instead of receiving unrestricted shell access.

Checkpoint exercise

Try it
Choose a system you would connect to an agent. Divide its capabilities into resources, prompt templates, and tools.

Key takeaways

  • Use MCP to standardize integration, not to avoid tool design.
  • Expose resources and tools at task-shaped granularity.
  • Record server provenance in the trace.

Failure modes

  • Wrapping broad dangerous APIs as convenient tools.
  • Dozens of overlapping tools with weak descriptions.
  • No audit trail for which server supplied a fact or action.

Implementation checklist

  • What resources are exposed?
  • Which tools have side effects?
  • How are credentials scoped?
  • How does discovery work?
  • Does the trace record MCP provenance?
Linear takeaway
MCP standardizes the agent-environment boundary so tools and context can be discovered and governed consistently.
