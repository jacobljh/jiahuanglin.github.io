Part IV - State, collaboration, and protocols

Memory management - session, state, long-term knowledge

Memory is scoped storage with write and read policies, not a longer transcript.

Book source
Chapter 8 - Memory Management; PDF outline pages 97-110.
Linear position
Prerequisite: The system has stateful tasks and possibly multiple roles.
New capability: Durable knowledge across steps, sessions, users, and projects.

What the book contributes

The book distinguishes short-term context memory from long-term persistent memory.

It discusses ADK sessions, state, MemoryService, LangChain memory, LangGraph stores, and Vertex Memory Bank.

Memory supports personalization, continuity, learning, and knowledge retrieval.

The chapter stresses that memory needs management: what is stored, how it is updated, and how it is retrieved.

Pattern shape

01Use working context for the next decision.
02Use task state for the current run.
03Use episodic memory for past interactions.
04Use semantic/project/user memory for durable facts and preferences.

Implementation sketch

memory_write = {
  "scope": "project",
  "fact": "Run targeted API tests with: npm test -- api",
  "source_trace": trace_id,
  "confidence": "verified",
  "expires_when": "package test script changes"
}
if stable_and_useful(memory_write):
    memory.store(memory_write)

Book-specific deep dive

The memory chapter is unusually concrete. Google ADK distinguishes session, state, and longer-term memory services. Session tracks the current conversation. State is the temporary working notebook for a run. MemoryService stores durable knowledge that can be retrieved later. LangChain, LangGraph, and Vertex Memory Bank show parallel ways to persist and retrieve that information.

Students should learn memory as scoped state, not as a single bucket. Personal preferences, project facts, workflow traces, and global lessons have different privacy and freshness requirements. The agent should also know when memory is evidence and when it is only a hint that must be verified.

  • Memory scopes: turn, session, task state, project memory, user memory, and global policy.
  • Write rule: durable memory should be explicit, useful in the future, attributable, and safe to reuse.

Running example

After verifying a repo-specific test command, the assistant stores it as project memory. It does not store a one-off failed command as a global fact.

Checkpoint exercise

Try it
Define three memories for an agent: one task-local, one project-level, and one user preference. Add write and delete rules.

Key takeaways

  • Separate working context, task state, episodic memory, semantic memory, and preferences.
  • Write memories only when stable, useful, permissioned, and attributable.
  • Retrieve memories by scope and relevance.

Failure modes

  • Treating long context as long-term memory.
  • Storing unverified inferences as facts.
  • Retrieving private user facts into unrelated tasks.

Implementation checklist

  • What memory types exist?
  • What triggers a memory write?
  • Who can read each scope?
  • How is memory corrected or deleted?
  • Does retrieval include provenance?
Linear takeaway
Memory gives continuity, but only scoped, attributable, and reviewable memory should influence future behavior.
