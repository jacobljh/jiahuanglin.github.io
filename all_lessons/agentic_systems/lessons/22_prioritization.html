Part VIII - Autonomy and discovery

Prioritization - choosing the next best action

Priority keeps an autonomous agent focused when many useful actions are possible.

Book source
Chapter 20 - Prioritization; PDF outline pages 220-227.
Linear position
Prerequisite: Goals, monitoring, budgets, and evaluation signals.
New capability: A next-action policy based on importance, urgency, dependency, risk, and cost.

What the book contributes

Prioritization helps agents handle many possible actions, conflicting goals, limited resources, and dynamic conditions.

Criteria include urgency, importance, dependencies, resource availability, cost-benefit, and user preferences.

The chapter covers strategic and tactical prioritization and dynamic adjustment as conditions change.

Prioritization distinguishes agentic systems from scripts because the agent selects focus under uncertainty.

Pattern shape

01Generate candidate next actions.
02Score them by goal impact, urgency, dependency, uncertainty reduction, risk, and cost.
03Choose or batch the best allowed action.
04Reprioritize when observations change state.

Implementation sketch

candidates = propose_next_actions(state)
for action in candidates:
    action.score = (
        3*goal_impact(action) +
        2*dependency_unlock(action) +
        info_gain(action) -
        risk(action) -
        cost(action)
    )
next_action = max_allowed(candidates, key="score")

Book-specific deep dive

Prioritization teaches an agent how to choose the next best action when there are multiple valid tasks. The book positions it after evaluation because priority is not only about urgency; it depends on expected value, risk, deadline, dependency, information gain, cost, and confidence.

In an educational sequence, prioritization turns a plan from a static list into a dynamic queue. When new evidence arrives or a tool fails, the agent should rescore possible actions. For example, it may retrieve missing evidence before drafting, escalate a high-risk action before continuing, or skip a low-value branch when the budget is nearly exhausted.

  • Priority score: value, urgency, dependency unlock, uncertainty reduction, cost, and risk.
  • Queue rule: after each observation, update priorities instead of blindly following the original plan.

Running example

After a patch, running a targeted test may outrank reading more files. After a failed test, inspecting the failure outranks broad refactoring. Priority changes with observation.

Checkpoint exercise

Try it
Create a candidate-action list for a real task. Score each action by impact, uncertainty reduction, risk, and cost.

Key takeaways

  • Score next actions by value, dependency, risk, and cost.
  • Keep candidate actions explicit.
  • Reprioritize only when state changes.

Failure modes

  • Doing the most recent suggestion instead of the most valuable action.
  • Ignoring cheap uncertainty-reducing actions.
  • No budget-aware priority, causing endless exploration.

Implementation checklist

  • What candidate actions exist?
  • Which unlocks progress?
  • Which reduces uncertainty?
  • What is the cost and risk?
  • What observation changed priority?
Linear takeaway
Prioritization turns a static plan into a dynamic queue that reacts to value, risk, uncertainty, and budget.
