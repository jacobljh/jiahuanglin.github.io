Part VI - Knowledge, communication, and optimization

A2A - agent-to-agent communication

Remote agents need identity, capability discovery, task lifecycle, and traceable messages.

Book source
Chapter 15 - Agent-to-Agent Communication; PDF outline pages 164-174.
Linear position
Prerequisite: Local multi-agent roles and external integration contracts.
New capability: Networked task delegation across agents, frameworks, or organizations.

What the book contributes

A2A is described as an open standard for communication and collaboration among agents built with different frameworks.

The chapter introduces Agent Cards, task sending, streaming updates, multi-turn input-required states, and modular distributed deployment.

It contrasts A2A with MCP: A2A coordinates agents; MCP connects agents to tools and resources.

A2A enables interoperability but adds distributed-systems concerns.

Pattern shape

01Discover remote agent capability.
02Send a task with context, constraints, and expected artifact.
03Receive progress, requests for input, result, or failure.
04Validate output and merge trace provenance.

Implementation sketch

task = {
  "id": task_id,
  "capability": "analyze_dataset",
  "input_ref": "s3://...",
  "deadline": "10m",
  "output_schema": "analysis_report_v1"
}
remote = discover_agent("data-analysis")
result = remote.send_task(task, idempotency_key=task_id)
validate(result, task.output_schema)

Book-specific deep dive

A2A moves from tools speaking to agents to agents speaking to each other. The book contrasts A2A with MCP: MCP standardizes access to tools and context; A2A standardizes communication between autonomous agents. That difference matters because an agent-to-agent message may carry intent, capability, task state, negotiation, delegated responsibility, or completion evidence.

For linear learning, A2A is placed after RAG and HITL because distributed agents need shared evidence and clear handoff contracts. Without that, one agent's conclusion becomes another agent's unsupported premise. Good A2A design includes identity, capability advertisement, message schema, task ownership, state transfer, and failure semantics.

  • A2A contract: sender, receiver, capability, task, evidence, expected response, deadline, and failure mode.
  • Comparison: use MCP for environment integrations; use A2A for delegation and coordination among agents.

Running example

A product assistant delegates cohort analysis to a remote data agent. The data agent streams progress and returns a report with queries, charts, and caveats.

Checkpoint exercise

Try it
Write an Agent Card for one specialized agent: name, capabilities, inputs, outputs, auth, limits, and expected latency.

Key takeaways

  • Treat A2A as networked task delegation.
  • Expose capabilities with schemas, limits, and auth requirements.
  • Use task ids, deadlines, idempotency, and trace correlation.

Failure modes

  • No clear owner when a remote subtask fails.
  • Retrying non-idempotent remote work.
  • Trusting remote output without provenance or validation.

Implementation checklist

  • How does discovery work?
  • What task states exist?
  • Can the caller cancel?
  • Are retries idempotent?
  • How are traces joined across agents?
Linear takeaway
A2A makes delegation explicit: agents exchange tasks, evidence, capabilities, and completion states rather than vague chat.
