Part I - Foundations

Prompt and context contracts - the interface to the model

Prompting becomes engineering when inputs and outputs are typed, scoped, and validated.

Book source
Appendix A plus Chapter 1 context-engineering discussion; PDF outline pages 250-266 and 24-32.
Linear position
Prerequisite: The agent boundary from lesson 01.
New capability: A contract for each model call: input slots, output schema, validation, and context budget.

What the book contributes

Appendix A covers prompting principles: role, clarity, examples, delimiters, structure, output format, reasoning guidance, and task-specific prompts.

The book distinguishes prompt engineering from context engineering: not only how to phrase the instruction, but what information the model should receive.

Structured output is central for agent systems because code must consume model decisions.

Prompting techniques such as few-shot examples, Chain of Thought, Tree of Thoughts, ReAct, and prompt refinement are useful only when placed inside the right system role.

Pattern shape

01Define the call purpose: classify, extract, plan, critique, synthesize, or act.
02Assemble context slots: objective, policy, state summary, evidence, available actions.
03Require a typed output schema.
04Validate output before it can update state or trigger tools.

Implementation sketch

route_schema = {
  "route": "answer_directly | retrieve | call_tool | ask_human | stop",
  "confidence": "number between 0 and 1",
  "reason": "short evidence-based reason",
  "missing_info": "list of required missing facts",
  "risk": "low | medium | high"
}

# Invalid JSON is not a route. It is a validation failure.
parsed = parse_json(model_output)
validate(parsed, route_schema)

Book-specific deep dive

This lesson intentionally pulls Appendix A forward as a prerequisite. The appendix appears later in the book, but educationally it belongs before prompt chaining because every chain, router, planner, critic, and tool call depends on a prompt/context contract. Zero-shot, one-shot, and few-shot prompting set behavior by examples. System and role prompting set authority and stance. Delimiters reduce ambiguity. Structured output turns prose into machine-checkable state.

The appendix also connects basic prompting to agent interaction: chain-of-thought style decomposition, self-consistency, step-back prompting, tree-of-thought search, ReAct, tool use, RAG, persona patterns, and prompt optimization. In a production agent, these are not magic phrases. They become contracts: what the model sees, what it must return, how uncertainty is represented, and which downstream component consumes the result.

  • Intentional reorder: Appendix A is front-loaded so the core chapters can be taught as systems, not isolated prompt examples.
  • Contract rule: a prompt is incomplete unless its input fields, output schema, refusal mode, and evaluator are named.

Running example

A support agent should not emit a paragraph saying it might need the order database. It should emit a validated action such as {route: call_tool, tool: lookup_order, args: {order_id: ...}} or ask for the missing order id.

Checkpoint exercise

Try it
Rewrite one of your normal prompts as a contract: named inputs, allowed outputs, validation rule, and recovery behavior when the output is invalid.

Key takeaways

  • Give every model call an input contract, output schema, and validation rule.
  • Separate durable policy from task-local context.
  • Use prompt techniques as local tools inside a system role, not as the architecture itself.

Failure modes

  • One giant prompt mixing policy, examples, facts, and transient tool output.
  • Free-form model output consumed directly by code.
  • Treating a larger context window as memory.

Implementation checklist

  • What fields enter the prompt?
  • What schema must leave it?
  • What happens on invalid output?
  • Which context is durable?
  • What evidence is summarized or referenced?
Linear takeaway
Prompt engineering becomes agent engineering when inputs, outputs, examples, schemas, and failure modes are explicit contracts.
