Linear Story

An agentic application adds tools, identity, state, and sometimes other agents. The book's final chapter focuses on MCP, A2A, security, and state management because once an agent can act, the platform must answer: on whose behalf, with what authority, through which tool, recorded where, and resumable how?

user goal -> agent runtime -> MCP tools / services
              |       |
              |       +-> state/checkpoints/audit
              +-> identity and policy boundary
multi-agent: agent -> A2A task/message -> agent

Identity Is the Core Trade-Off

Token passthrough preserves user context but can overexpose credentials if the agent or tool chain is compromised. Service account delegation gives the agent a bounded service identity but may lose fine-grained user intent. OAuth2 token exchange and SPIFFE/SPIRE-style workload identity can create clearer trust boundaries, but require more platform maturity. The right choice depends on whether the agent is acting as the user, as the service, or as a delegated workflow with constrained scope.

Kubernetes Mapping

Interactive agents often look like Deployments behind APIs. Long-running research or automation agents may look like Jobs with persistent checkpoints. Ambient agents often look like event-driven workers, CronJobs, or controllers. Human-in-the-loop gates should be explicit workflow states, not hidden chat messages.

Responses API, Frameworks, And Runtimes

The book notes that the boundary between inference platform and agent runtime is moving. APIs such as OpenAI's Responses API pull tool calling, structured outputs, and conversation state closer to the model-serving interface, while frameworks such as LangGraph-style graphs, CrewAI-style role teams, and other agent runtimes keep orchestration in application code. On Kubernetes, this becomes a placement question. If tool orchestration lives in the model server, scale and observe it with inference. If orchestration lives in a separate agent service, give that service its own SLOs, state store, deployment cycle, and policy boundary.

A2A On Kubernetes

Agent-to-agent communication should look like ordinary service communication with stronger contracts: discover capabilities, send a task, track status, stream or poll results, and preserve trace context. A2A does not remove the need for Kubernetes primitives; it rides on them. Services expose agents. NetworkPolicy limits who can call whom. Workload identity tells the receiver who is asking. Traces correlate the delegated task back to the original user request. Without those pieces, a multi-agent system becomes distributed hidden state.

State Store Selection

Choose state by access pattern. A KV store fits hot session memory and resumable checkpoints where lookup is by session or task ID. A relational database fits audit, analytics, retention, and cross-session queries. Object storage fits large artifacts, intermediate files, and checkpoint blobs. Many production systems use all three: KV for fast loop state, database for durable indexed records, object storage for bulk artifacts. The mistake is pretending the chat transcript is the state store.

Safety Rule

Never let the model be the authority boundary. Tool schemas, IAM, NetworkPolicy, service identity, approval states, and audit logs are the boundary. The model can recommend an action; the platform decides whether it happens.

Mental Model

An agent is a workload with authority. Kubernetes gives you service identity, network boundaries, storage, events, jobs, and policy; the agent framework must fit inside those boundaries.

Operational Checklist

  • Separate user identity, service identity, and delegated tool authority.
  • Use MCP or explicit tool gateways for capability discovery and schema-bound calls.
  • Persist long-running agent state with checkpoints and audit trails.
  • Put human-in-the-loop gates at irreversible, costly, or policy-sensitive actions.

Failure Modes

  • Token passthrough gives an agent more authority than the user or service should have.
  • A long-running agent loses state when its pod restarts.
  • Multi-agent communication is opaque, so failures cannot be reconstructed after the fact.
