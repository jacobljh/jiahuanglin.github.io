Linear Story

Disaggregated serving starts from one asymmetry. Prefill processes the prompt in parallel and tends to be compute-heavy. Decode emits tokens one step at a time and tends to be memory-bandwidth-heavy. If one pool of GPUs runs both phases, long prompts can interfere with short decode steps, and hardware may be poorly matched to the phase. The book introduces disaggregation as the production answer to that mismatch.

request -> prefill pool -> KV state transfer -> decode pool -> token stream

The payoff is specialization. You can size prefill for compute and decode for memory bandwidth/latency. You can isolate long prompt bursts from decode jitter. You can tune queues separately. The cost is that a single request now crosses services, device pools, and a state transfer boundary. The KV cache becomes a distributed artifact.

The KV Transfer Test

Disaggregation is worth considering only when the utilization win beats the transfer and coordination cost. The KV state can be large. If prefill and decode pools are far apart, the network can eat the gain. If failure handling is weak, a request can succeed in prefill and disappear before decode. If autoscaling is independent, one queue can starve the other.

Design Rule

Do not start with disaggregation. Start with measuring prefill and decode separately. Add disaggregation when those measurements show opposite bottlenecks and the platform is mature enough to operate a two-stage request path.

Leader-Worker Shape

A common Kubernetes shape for multi-pod inference is leader-worker: one logical model endpoint is backed by a leader pod plus worker pods that form the distributed runtime. The leader owns the external serving surface, readiness, and coordination. Workers own shards or phase-specific execution. This shape fits large models, tensor-parallel groups, and some disaggregated serving layouts because the platform can treat a group of pods as one serving unit.

The platform challenge is lifecycle. All members must agree on model version, runtime config, topology, and readiness. A rollout cannot update the leader and leave workers on a mismatched artifact. A readiness check must mean the group is healthy, not only that the leader process is alive. A failure policy must decide whether losing one worker drains the whole group, restarts the group, or routes around it. That is the operational reason LeaderWorkerSet-style abstractions exist: they encode group identity and lifecycle so model-parallel serving is not hand-stitched from unrelated Deployments.

Mental Model

Prefill wants compute; decode wants memory bandwidth and low jitter. Disaggregation works when the saved GPU time beats the KV transfer and scheduling overhead.

Operational Checklist

  • Measure prefill and decode bottlenecks separately before splitting pools.
  • Place prefill and decode pools where KV transfer bandwidth and latency are acceptable.
  • Keep admission control aware of both queues, not just total request count.
  • Plan failure handling for partially completed requests and transferred KV state.

Failure Modes

  • KV transfer over the network erases the utilization gain.
  • Two autoscalers fight each other because prefill and decode queues are coupled.
  • Debuggability drops because a single request now crosses multiple services and device pools.
