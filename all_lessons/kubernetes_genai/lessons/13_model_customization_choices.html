Linear Story

Model customization begins with a ladder. Prompt engineering changes instructions. Context engineering changes what evidence the model sees. RAG changes the knowledge path without changing weights. LoRA and PEFT change a small trainable adapter. Full fine-tuning changes many or all model weights. The book's Chapter 6 is useful because it treats customization as an operational choice, not a badge of sophistication.

prompt -> context/RAG -> LoRA/PEFT -> full fine-tune -> pretraining
cheap/fast                                              expensive/deep

The trade-off is control versus machinery. Prompt and RAG changes are easy to roll back and often solve factuality or freshness problems. Adapters can change style, domain behavior, or task performance with less compute than full fine-tuning. Full fine-tuning may be justified when adapters cannot express the change, when serving constraints require merged weights, or when the behavior shift is broad and well-evaluated.

Ops Cost

Every rung adds platform work: data curation, training jobs, evaluation, registry metadata, adapter or model serving, rollout, monitoring, and rollback. A fine-tuned model without eval gates is not more mature than a prompt; it is just harder to inspect.

Rule of Thumb

Do not train facts into weights if those facts change. Do not build RAG for behavior that should be learned. Do not fine-tune until a simpler baseline has been measured. The platform should make escalation deliberate.

Mental Model

Customization is a ladder: prompt, retrieval, adapter, fine-tune, pretrain. Each rung buys more control and costs more data, evaluation, deployment, and rollback machinery.

Operational Checklist

  • Start with prompt and context changes when the needed knowledge already exists in data sources.
  • Use RAG when knowledge changes or citations/provenance matter.
  • Use LoRA or PEFT when behavior or domain style must change and examples are available.
  • Use full fine-tuning only when adapter capacity or deployment constraints justify the cost.

Failure Modes

  • Fine-tuning facts that should live in retrieval.
  • Shipping a customized model without comparing it to a stronger prompt/RAG baseline.
  • Treating LoRA as a training-only decision and forgetting multi-adapter serving implications.
