Linear Story

A Kubernetes cluster does not automatically know what a GPU is. The node needs drivers and runtime libraries. Kubernetes needs a way to discover accelerator properties. The scheduler needs a resource model. The pod needs a request that can be matched to a device. The book's Chapter 3 walks this stack because most "my GPU pod does not work" incidents are not model incidents at all.

hardware -> driver/runtime -> discovery -> device plugin or DRA -> scheduler -> pod device visibility

Node Feature Discovery and GPU Feature Discovery label what exists. Device plugins expose resources such as nvidia.com/gpu. The NVIDIA GPU Operator automates driver, toolkit, device plugin, monitoring, and related components through cluster policy. DRA moves the model toward explicit resource claims and richer device selection. The platform lesson is that "GPU: 1" is often too little information for serious AI workloads.

Sharing Is Not Free

MIG slices a GPU into hardware-isolated instances. Time slicing shares a device over time. Whole-GPU allocation gives simple isolation and predictable performance. DRA can express richer requests and allocation flows. The trade-off is utilization versus isolation. Sharing can lift utilization for small inference jobs, but it can also create noisy neighbors, ambiguous accounting, and surprising latency variance.

Debugging Order

When a GPU workload fails, debug bottom-up. Does the node see the device? Does the driver match the runtime? Did discovery label it? Did the plugin or DRA driver advertise it? Did the scheduler place the pod on a compatible node? Does the container see the device file and libraries? Only after that should you debug the model server.

Current Status Note

As of the Kubernetes docs checked for this update, the core DRA path has reached stable Kubernetes releases, with setup documentation marking DRA stable in the v1.35 line and later docs expanding the DRA surface. Treat that as "the API is no longer a research idea," not "every accelerator driver and every advanced sharing feature is production-ready in your cluster." For GPUs, the practical readiness question is still vendor driver support, scheduler integration, observability, and whether your operators know how to debug ResourceClaims when allocation fails.

Mental Model

The GPU is not just a resource count. It is a stack: driver, runtime, device plugin or DRA driver, node labels, topology, isolation mode, telemetry, and scheduling policy.

Operational Checklist

  • Install discovery so nodes advertise accelerator type, count, driver, MIG state, and topology.
  • Choose between whole GPU, MIG, time slicing, or DRA-style claims based on isolation and utilization needs.
  • Treat GPU Operator configuration as platform code, not snowflake node setup.
  • Validate DCGM metrics and runtime visibility before admitting tenant workloads.

Failure Modes

  • Pods schedule onto GPU nodes but cannot see devices because the driver stack is incomplete.
  • MIG or sharing mode improves utilization but violates tenant isolation assumptions.
  • Schedulers see only 'nvidia.com/gpu: 1' and miss the topology or memory requirements that determine performance.
