Linear Story

Start with the shape Kubernetes already understands: a container listens on a port, replicas sit behind a Service, and an autoscaler adds pods when load rises. That model is still useful, but it is no longer enough. A GenAI service has a heavy object that must be present before it can answer: the model weights. It also has a two-phase request path. First the server reads the prompt and builds the KV cache; then it emits output tokens one at a time. The first phase tends to be compute-heavy, the second tends to be memory-bandwidth-heavy, and both phases compete for expensive accelerator memory.

The LLM fundamentals in the book are operational, not academic. Tokenization decides how text becomes work units. Embeddings and transformer layers decide why prompt length matters. Prefill turns the full prompt into reusable state. Decode advances one generated token at a time, repeatedly reading the KV cache. If you remember only one mechanism here, remember this: a request is not one homogeneous HTTP transaction. It is artifact readiness plus prefill plus many decode steps plus streaming and policy.

The book's introduction makes this the first platform lesson: the model is not the whole system. Once the model is inside your boundary, you own model data movement, GPU allocation, routing, autoscaling, observability, safety, and cost. Kubernetes is attractive because it gives a common control plane for those concerns, but the control plane only works when the workload exposes the right signals. A CPU HPA cannot see TTFT. A normal readiness probe cannot tell whether a 70 GB model is loaded. A generic load balancer cannot know that one replica has the right prefix cached.

The First Trade-Off

The first decision is not "which Kubernetes object should I use?" It is "why am I running this model myself?" If the answer is privacy, compliance, data gravity, latency, custom routing, model control, or cost at steady scale, Kubernetes can be the right substrate. If the answer is only "because we can," then the platform takes on hard operational work without product leverage. Running GenAI in-cluster means accepting the burden of GPUs, drivers, model registries, scaling policy, safety reviews, and incident response. The payoff is control.

Design Exercise

For any GenAI workload, write four numbers before drawing architecture: expected prompt tokens, expected output tokens, target TTFT, and target concurrent requests. Those four numbers decide whether you are solving a storage problem, a GPU memory problem, a routing problem, or a cost problem. The rest of the track is the platform machinery that makes those numbers real.

Mental Model

A GenAI platform is a token factory. Kubernetes schedules containers, but the real product is bounded latency per generated token under scarce GPU, memory, storage, and policy constraints.

Operational Checklist

  • Define the serving SLOs: TTFT, TPOT, throughput, error rate, and quality guardrails.
  • Name the scarce resource first: HBM, GPU count, interconnect, storage bandwidth, or external data access.
  • Decide what must stay inside the cluster for privacy, compliance, or latency.
  • Keep LLM fundamentals short and operational: tokenization, prefill, decode, KV cache, and model artifact size.

Failure Modes

  • Treating an LLM server like a CPU web pod and scaling only on CPU or request count.
  • Ignoring startup time for 10 GB to 800 GB model artifacts.
  • Optimizing average latency while TTFT or p99 inter-token latency is what users feel.
