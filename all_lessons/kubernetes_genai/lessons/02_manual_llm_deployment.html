Linear Story

The book starts deployment manually for a reason. Before KServe, Ray Serve, or a platform API, an inference service is still a pod with a container image, a model path, a GPU request, a port, and a Service. Doing it once by hand teaches the contract every higher-level controller must eventually satisfy.

request -> Service -> Pod
            runtime image
            model mount
            GPU resource
            readiness: model loaded
            metrics/logs endpoint

The key linear move is to separate "process started" from "model ready." A web server can usually become ready when the process binds a port. An LLM server may bind a port quickly and then spend minutes loading weights, building CUDA kernels, warming graphs, or preparing adapter state. If readiness flips early, the platform sends users into a cold model. So the readiness probe should represent the serving contract: the target model is loaded and can produce a response within the expected envelope.

What Manual Deployment Teaches

A Deployment is good when replicas are interchangeable and model state lives outside the pod. A StatefulSet can make sense when a replica owns a warm cache or persistent local model copy, but it also makes replacement and rollout semantics heavier. A PVC is familiar and flexible, but storage throughput can become the hidden bottleneck. Baking weights into an image gives immutability and registry caching, but makes images enormous and release cycles slower. Downloading weights at startup is easy for a demo and painful in production because every scale-up becomes a network event.

ChoiceBuysCosts
Runtime image onlySmall deployable, fast runtime patchingNeeds separate model delivery path
Weights in imageImmutable artifact, node cache semanticsHuge images, slower promotion

Minimal Manifest Thinking

A production-minded first manifest should declare the model path explicitly, request the GPU explicitly, pin the runtime version, expose metrics, and include startup/readiness behavior that matches model load time. Resist the urge to hide these details too early. When a controller later fails, you will debug it by asking which of these plain-Kubernetes responsibilities it failed to perform.

Failure Lens

When a manual deployment fails, classify the failure before changing YAML: image failure, model-data failure, accelerator failure, runtime configuration failure, network exposure failure, or readiness failure. That classification is the first operational habit of the whole series.

Mental Model

Manual deployment is the assembly diagram. KServe, Ray Serve, Helm charts, and platform APIs are useful only after you know which pod, volume, port, probe, and GPU request they hide.

Operational Checklist

  • Package the runtime image separately from the model artifact when startup or promotion speed matters.
  • Request GPUs explicitly and verify the pod lands on an accelerator node.
  • Use readiness to mean the model is loaded and can answer, not merely that the HTTP process started.
  • Expose a narrow service contract: health, completion endpoint, metrics, and logs.

Failure Modes

  • Readiness flips green before weights are loaded, so traffic hits a cold runtime.
  • The pod requests a GPU but the image lacks the right CUDA/runtime stack.
  • A PVC or model path mismatch silently makes the server download from the internet at startup.
