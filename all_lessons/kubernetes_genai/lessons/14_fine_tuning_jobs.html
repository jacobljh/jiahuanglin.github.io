Linear Story

A fine-tuning workload is a batch job, not a web service. It needs a dataset, a base model, code, hyperparameters, GPUs, checkpoints, logs, and a terminal state. If distributed, all workers must rendezvous and progress together. The book introduces Kubeflow Trainer and other frameworks because Kubernetes needs training-specific job semantics beyond a plain Deployment.

dataset + base model + config
        |
        v
queued training job -> workers rendezvous -> checkpoints -> model/adaptor artifact -> eval

The operational shape is different from serving. Serving wants fast replacement and continuous availability. Training wants gang scheduling, long-running resource allocation, fault recovery, high-throughput storage, and reproducibility. A single slow worker can slow the whole job. A lost checkpoint can waste days of GPU time.

Reproducibility Contract

Every run should capture code revision, container image, base model digest, dataset version, tokenizer, training config, seed, hardware shape, and output artifact. Without that, the registry cannot tell whether a better model came from better data, different hyperparameters, or accidental environment drift.

Framework Thinking

Kubeflow Trainer gives Kubernetes-native structure for distributed training jobs. Ray can be a good fit when the organization already uses Ray for distributed Python workflows. Lightweight libraries such as Unsloth can lower the barrier for small customization jobs. The platform decision is about repeatability and ownership, not only training speed.

Security Guidance For Training Runtimes

Training frameworks expand the trust boundary. Ray clusters expose dashboards, job submission paths, object stores, and worker-to-worker communication. PyTorch distributed jobs open rendezvous and communication endpoints. A production platform should scope network access, avoid unauthenticated dashboards, isolate namespaces, use short-lived credentials for data access, and prevent arbitrary user code from inheriting broad service-account permissions. The right mental model is simple: a training job is untrusted code with expensive devices and access to sensitive data unless the platform proves otherwise.

Mental Model

A training job is not a long-running service. It needs all workers together, fast shared data, resumable checkpoints, correlated logs, and a queueing policy.

Operational Checklist

  • Record code, data, base model, adapter config, seed, hyperparameters, and environment as one run artifact.
  • Checkpoint often enough to survive preemption or node loss without losing hours of GPU time.
  • Separate data access credentials by job and tenant.
  • Collect rank-aware logs and metrics so one slow worker is visible.

Failure Modes

  • One worker cannot schedule, so the rest hold GPUs idle.
  • Checkpoints are written to slow or unreliable storage and dominate wall time.
  • Training secrets leak into images, logs, or shared notebooks.
