Linear Story

RAG turns an LLM service into a system of two coupled pipelines. The ingestion pipeline reads documents, chunks them, embeds them, stores vectors, and records provenance. The query pipeline embeds the user query, retrieves candidates, filters by policy, optionally reranks, builds context, calls the model, and returns an answer with evidence. The book treats RAG as an application architecture because the model is only one piece.

ingest: source -> chunk -> embed -> index -> provenance
query:  user -> embed -> retrieve -> rerank/filter -> prompt -> answer

Application Architecture Before RAG

Chapter 8 starts wider than retrieval. A chat application is a synchronous path: user request, conversation state, policy, model call, stream response. A backend AI service is often asynchronous: an event arrives, a worker enriches it with context, the model produces a result, and another service records or acts on it. Batch jobs, CronJobs, and control loops sit beside those two patterns. The Kubernetes choice follows the latency shape: Deployments and Services for hot paths, Jobs or CronJobs for ingestion and batch, StatefulSets for durable indexes and stores, and event-driven workers for asynchronous automation.

RAG is the most common full-stack pattern in that chapter because it forces all the pieces to show up: ingestion jobs, embedding services, vector storage, query services, policy filters, rerankers, model serving, and citation/audit behavior.

Kubernetes Mapping

Ingestion often fits Jobs, CronJobs, or workflow engines. Embedding workers need model-serving capacity but can run off the hot path. Vector databases are stateful systems: they need StatefulSets or managed equivalents, backups, restore tests, storage sizing, compaction, and tenant isolation. Query services are latency-sensitive and must avoid starving inference with ingestion work.

The Consistency Trap

RAG quality depends on paired decisions. If the ingestion embedding model changes but the query embedding model does not, retrieval degrades. If chunking changes without reindexing, citations drift. If access-control filters are not applied at retrieval time, the model may receive documents the user cannot see. If provenance is missing, the answer cannot be audited.

Rule of Thumb

Version the ingestion recipe like model code. A RAG index is a derived-data product, not a pile of embeddings. It needs lineage, tests, and rollback.

Mental Model

RAG quality depends on consistency between write-time and read-time choices: chunking, embedding model, metadata, index, reranker, filters, and citation policy.

Operational Checklist

  • Version embedding model, chunker, index schema, and source-document provenance together.
  • Run ingestion as Jobs or CronJobs so it does not starve the online serving path.
  • Operate vector databases like stateful systems: backup, restore, compaction, scaling, and tenant isolation.
  • Measure retrieval quality and generation quality separately.

Failure Modes

  • The embedding model changes for ingestion but not query, so similarity scores degrade silently.
  • Vector DB restore is untested and production index loss becomes a full re-ingestion incident.
  • RAG citations point to chunks without document provenance or access-control checks.
