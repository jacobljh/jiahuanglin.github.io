Linear Story

Once one GPU works, the next question is whether to use more. More GPUs can mean more replicas, a single model sharded across devices, or a pipeline of stages. These are not interchangeable. Replication improves throughput and availability when each replica can hold the model. Tensor parallelism helps when one model or one desired concurrency target needs multiple GPUs. Pipeline parallelism splits layers and introduces bubbles and coordination.

ReplicateSame model on many GPUs; route independent requests.
Tensor parallelShard matrix work inside layers; needs fast collectives.
Pipeline parallelShard layers; useful for very large models, sensitive to bubbles.
MultinodeExtends capacity, but network becomes part of every token.

The Topology Lesson

Kubernetes scheduling decides where pods land, but inference performance depends on the physical graph underneath: PCIe, NVLink, NVSwitch, InfiniBand, rack placement, and sometimes secondary high-performance interfaces. If tensor-parallel ranks land across a slow boundary, every decode step pays. If replicas are packed without regard to cache and traffic, a cluster can have enough GPUs on paper and still miss latency targets.

StrategyBuysCosts
Data parallel replicasSimple scaling and failure isolationDuplicates weights and warm caches
Tensor parallelFits larger models or raises bandwidthCollective communication per layer/token
Pipeline parallelLayer-level model fitPipeline bubbles and scheduling complexity
Multinode inferenceCapacity beyond one hostNetwork latency and failure surface

Decision Rule

Prefer replication until model size or per-replica throughput forces sharding. Keep tightly coupled ranks inside the fastest topology domain. Use scheduling labels, node pools, topology-aware policies, and runtime placement settings together; no single layer can fix topology alone.

Mental Model

Replication buys throughput and fault isolation. Tensor parallel buys model fit. Multinode sharding buys capacity at the price of interconnect latency and operational risk.

Operational Checklist

  • Replicate small enough models before reaching for tensor parallelism.
  • Keep tensor parallel inside fast interconnect domains when possible.
  • Use topology labels, node pools, and scheduling policies to keep shards close.
  • Budget separately for weight memory, KV cache memory, activation scratch, and communication.

Failure Modes

  • Tensor parallel spans slow network links and turns every token into a communication problem.
  • The platform packs unrelated GPU pods onto the same host and breaks expected NVLink locality.
  • Capacity math ignores KV cache, so the model fits but concurrency does not.
