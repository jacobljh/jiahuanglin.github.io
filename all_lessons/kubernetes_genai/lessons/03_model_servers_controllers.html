Linear Story

The next question is ownership. A model server owns token execution. A controller owns Kubernetes lifecycle. A programmable serving framework owns application composition. The book walks through vLLM, TGI, and related servers before KServe and Ray Serve because these are different layers, not competitors in one flat list.

ServerOptimizes prefill, decode, KV cache, batching, adapters, sampling.
ControllerCreates Deployments, Services, revisions, rollout, scale, status.
FrameworkLets Python code compose models, preprocessing, routing, and workers.

vLLM is a good mental anchor because it makes LLM-specific runtime concerns visible: PagedAttention, continuous batching, prefix caching, chunked prefill, quantized KV, speculative decoding, and OpenAI-compatible endpoints. TGI occupies a similar model-server category with its own operational trade-offs. SGLang pushes the unit of work toward programs and prefix reuse. None of those by itself gives you a platform API for many teams, canary rollout, or standardized ownership. That is where controllers enter.

KServe Versus Ray Serve

KServe is Kubernetes-native in feel: declare an inference service and let the platform reconcile networking, revisions, autoscaling, and status. It is a strong fit when the platform team wants a standardized serving API and users mainly provide model/runtime configuration. Ray Serve is Python-native in feel: define deployments in code, compose them, and let Ray manage distributed execution. It is a strong fit when application logic, custom composition, or data-science ownership matters more than a narrow inference CRD.

Decision Rule

If the hard part is per-token efficiency, choose the runtime first. If the hard part is many teams safely deploying model endpoints, choose the controller first. If the hard part is composing a multi-step application in Python, evaluate Ray Serve. Good platforms keep the boundaries explicit: the controller should not obscure runtime metrics, and the runtime should not become the policy system for the whole organization.

Mental Model

A model server owns token execution. A controller owns lifecycle. A programmable serving framework owns Python-defined composition. Confusing those layers creates platforms that are both rigid and hard to debug.

Operational Checklist

  • Pick the runtime based on workload shape: chat, long context, structured output, LoRA, multimodal, or batch.
  • Pick the controller based on rollout, autoscaling, multitenancy, and team API needs.
  • Use KServe when standard inference lifecycle matters; use Ray Serve when Python-native composition and distributed workers matter.
  • Keep escape hatches for raw Deployments when debugging the controller layer.

Failure Modes

  • Adopting a controller to compensate for not understanding the runtime bottleneck.
  • Letting data scientists own Kubernetes lifecycle details that the platform should standardize.
  • Treating Ray, KServe, and vLLM as interchangeable because all expose HTTP.
